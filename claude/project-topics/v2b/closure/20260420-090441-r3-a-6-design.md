## 1. Reuse-First Sufficiency Test (leading section)

### 1.1 Default position (RATIFIED)

Entering approval wait-state → `WorkflowExecutionSuspendedEvent`.
Approval granted → `WorkflowExecutionResumedEvent`.
Approval rejected → `WorkflowExecutionCancelledEvent`.

**Zero new approval-specific lifecycle event types.** Approval context is carried exclusively via (a) canonical signal/reason conventions on the existing events, (b) command envelope metadata, (c) audit evidence, and (d) observability.

### 1.2 Enumeration of required data → existing-carrier mapping

The wait-state requires the following data across the three transitions. Each row names the carrier in existing infrastructure — no row requires a new event type.

| Dimension | Datum | Transition | Carrier | Confidence |
|---|---|---|---|---|
| **Machine signal** | "this suspend is awaiting approval (vs timer / external dep)" | Enter wait | `WorkflowExecutionSuspendedEvent.Reason`, canonical prefix `human_approval:{signal}` | HIGH — Reason is designed as audit/signal per event XML doc |
| **Machine signal** | "this resume is approval-granted (vs failure-retry)" | Granted | `WorkflowExecutionResumedEvent.PreviousFailureReason`, canonical prefix `human_approval_granted:{signal}:{actor}` | MEDIUM — field name is failure-coupled; see §1.5 trade-off and D4 |
| **Machine signal** | "this cancel is approval-rejected (vs caller-cancellation)" | Rejected | `WorkflowExecutionCancelledEvent.Reason`, canonical prefix `human_approval_rejected:{signal}:{actor}:{reason?}` | HIGH — event XML doc already contemplates tag+suffix convention |
| **Context** | Step that yielded | Enter wait | `WorkflowExecutionSuspendedEvent.StepName` | HIGH — purpose-built |
| **Context** | Step that resumed from | Granted | `WorkflowExecutionResumedEvent.ResumedFromStepName` | HIGH |
| **Context** | Step the rejection applies to | Rejected | `WorkflowExecutionCancelledEvent.StepName` | HIGH |
| **Context** | Approval key (deterministic match on Approve/Reject) | All three | Implicit: `(workflowExecutionId, StepName)` from preceding Suspended event. Optional explicit key encoded in `{signal}` for multi-approval workflows (deferred). | HIGH for single-approval; MEDIUM for multi-approval (deferred per D2) |
| **Context** | Short rationale | Rejected | `{reason?}` suffix on the Cancelled Reason; full rationale lives in command payload | MEDIUM — free-text observability |
| **Actor / audit identity** | Authoritative approver identity | Granted / Rejected | **`CommandContext.Actor`** (runtime identity middleware output) — AUTHORITATIVE | HIGH — middleware guarantee |
| **Actor / audit identity** | Approver identity (non-authoritative mirror) | Granted / Rejected | `{actor}` segment inside the carrier prefix — denormalized observability copy; non-authoritative | HIGH as observability; NEVER trusted as authoritative |
| **Actor / audit identity** | Approver role / permissions | Granted / Rejected | `PolicyEvaluatedEvent` per `R-PLAT-12` | HIGH — mandatory on every command |
| **Actor / audit identity** | Policy decision hash | Granted / Rejected | `DecisionHash` on command envelope | HIGH |
| **Actor / audit identity** | Full rationale / attachment payload | Granted / Rejected | Command payload registered in `IPayloadTypeRegistry`; preserved in command audit envelope, NOT duplicated onto lifecycle event | HIGH |
| **Correlation** | CorrelationId / CausationId | All three | Command envelope per `R-PLAT-12` | HIGH |
| **State preconditions** | "Suspended before Approve/Reject" | Command dispatch | Aggregate status + command-handler precondition (see §2.4) | HIGH — existing factory precondition on Suspended; extend to command layer |

### 1.3 Three-dimensional separation (per execution constraint)

The user required that approval context remain cleanly separated across machine signal, reason/context, and actor/audit identity. The mapping above honours that:

- **Machine signal** — canonical Reason prefixes (`human_approval`, `human_approval_granted`, `human_approval_rejected`). Projection and consumers switch on prefix without free-text parsing.
- **Reason / context** — canonical `{signal}` segment (step-scoped short string) plus StepName. Non-load-bearing; for audit/observability.
- **Actor / audit identity — AUTHORITATIVE** — command envelope (`CommandContext.Actor`, policy decision, decision hash, correlation) and `IPayloadTypeRegistry` payload (full rationale, attachments). Any `{actor}` segment in the Reason text is a non-authoritative observability mirror; never a trust boundary.

### 1.4 Sufficiency verdict (RATIFIED)

**SUFFICIENT.** Every datum enumerated in §1.2 maps to an existing carrier. No new lifecycle event type is proposed. The surface area added by R3.A.6 is:

- `WorkflowStepResult.AwaitingApproval(...)` — step result variant (new factory, not a new event)
- `ApproveWorkflowCommand(Guid WorkflowId, ApprovalDecisionPayload Decision)` — new command contract
- `RejectWorkflowCommand(Guid WorkflowId, ApprovalDecisionPayload Decision)` — new command contract
- `ApprovalDecisionPayload` — new CLR type registered in `IPayloadTypeRegistry`
- `IWorkflowExecutionReplayService.CancelSuspendedAsync(...)` — new method on existing interface (symmetric with `ResumeAsync`)
- `WorkflowLifecycleEventFactory.Resumed(aggregate, approvalSignalOverride)` — new factory overload
- Extensions to `WorkflowExecutionProjectionHandler` — handle `WorkflowExecutionSuspendedEventSchema` and `WorkflowExecutionCancelledEventSchema` (coverage gap; schemas exist, projection does not subscribe yet); add derived `ApprovalState` / `ApprovalSignal` / `ApprovalDecision` fields on the read model without replacing canonical `Status`
- New guard rules `R-WF-APPROVAL-01..07` in `runtime.guard.md` Workflow section

### 1.5 Known trade-offs (not insufficiencies)

| # | Concern | Trade-off | Blocker? |
|---|---|---|---|
| 1 | `WorkflowExecutionResumedEvent.PreviousFailureReason` field name is failure-coupled (D4 — TEMPORARY CARRIER) | The field is plain `string`. Canonical prefix `human_approval_granted:` distinguishes approval-resume from failure-resume. All design-note text, XML-doc comments on the factory overload, and guard-rule text MUST state clearly that this is **carrier context**, NOT failure semantics — approval-granted is a successful approval decision, never a failure-derived resume. | 3 | `IWorkflowExecutionReplayService.ResumeAsync` XML doc says "asserts Failed state" but factory accepts Suspended too (post-R3.A.3) | Stale doc — fix inline in R3.A.6 step 2/3 per D8. Trivial. | No. |

## 2. Design Details

### 2.1 `WorkflowStepResult.AwaitingApproval`

Current shape uses a property-based success/failure discrimination on `sealed record WorkflowStepResult`. Minimal extension:

```csharp
public sealed record WorkflowStepResult
{
    public bool IsSuccess { get; init; }
    public bool IsAwaitingApproval { get; init; }   // NEW
    public string? Error { get; init; }
    public string? ApprovalSignal { get; init; }    // NEW — canonical "human_approval:{signal}" carrier
    public string? ApprovalStepName { get; init; }  // NEW — optional, defaults to current step
    public object? Output { get; init; }
    public IReadOnlyList<object> Events { get; init; } = [];

    public static WorkflowStepResult Success(object? output = null, IReadOnlyList<object>? events = null) =>
        new() { IsSuccess = true, Output = output, Events = events ?? [] };

    public static WorkflowStepResult Failure(string error) =>
        new() { IsSuccess = false, Error = error };

    public static WorkflowStepResult AwaitingApproval(string approvalSignal, string? approvalStepName = null) =>
        new() { IsAwaitingApproval = true, ApprovalSignal = approvalSignal, ApprovalStepName = approvalStepName };
}
```

`approvalSignal` carries the canonical prefix — callers pass e.g. `"human_approval:risk-sign-off"` or `"human_approval:legal-review:urgent"`. The engine forwards it verbatim to `WorkflowLifecycleEventFactory.Suspended(...)` as the `reason` argument. `approvalStepName` is optional and defaults to the step name inferred by the engine from step context.

**Reuse-first guarantees:**
- `IsSuccess` remains the happy-path discriminator; `AwaitingApproval` is a suspend directive, not a success/failure flavour.
- The engine halts on `IsAwaitingApproval == true` and emits `WorkflowExecutionSuspendedEvent` via the existing factory — no new engine API.

### 2.2 Approve / Reject command contracts

```csharp
// src/shared/contracts/runtime/ApproveWorkflowCommand.cs
public sealed record ApproveWorkflowCommand(
    Guid WorkflowId,
    ApprovalDecisionPayload Decision) : IHasAggregateId
{
    public Guid AggregateId => WorkflowId;
}

// src/shared/contracts/runtime/RejectWorkflowCommand.cs
public sealed record RejectWorkflowCommand(
    Guid WorkflowId,
    ApprovalDecisionPayload Decision) : IHasAggregateId
{
    public Guid AggregateId => WorkflowId;
}

// src/shared/contracts/runtime/ApprovalDecisionPayload.cs
public sealed record ApprovalDecisionPayload(
    string Rationale,
    string? ApprovalKey);        // null = step-keyed; non-null = multi-approval key match (deferred per D2)
```

`ApprovalDecisionPayload` is registered in `IPayloadTypeRegistry` via the orchestration-system bootstrap module per `R-WF-PAYLOAD-REGISTRY-01`.

**Actor sourcing (authoritative constraint):** `ApprovalDecisionPayload` does NOT carry an approver field. The authoritative approver identity is sourced from `CommandContext.Actor` (populated by runtime identity middleware). Both commands flow through the canonical 11-stage pipeline (identity → idempotency → validation → policy → admission → dispatch → persist → chain → outbox → …). Actor, correlation, causation, policy decision, decision hash all arrive on the command context per `R-PLAT-12`.

### 2.3 `IWorkflowExecutionReplayService.CancelSuspendedAsync`

Symmetric with the existing `ResumeAsync`:

```csharp
public interface IWorkflowExecutionReplayService
{
    Task<WorkflowExecutionReplayState?> ReplayAsync(Guid workflowExecutionId);
    Task<object> ResumeAsync(Guid workflowExecutionId);                                        // existing
    Task<object> CancelSuspendedAsync(Guid workflowExecutionId, string reason);                // NEW
}
```

Implementation:
1. Load aggregate via event-store replay.
2. Assert `Status == Suspended`; else throw `WorkflowExecutionErrors.CannotRejectUnlessSuspended` (new error code).
3. Construct `WorkflowExecutionCancelledEvent` via `WorkflowLifecycleEventFactory.Cancelled(id, stepName, reason)` (existing factory method). `reason` is the composed `human_approval_rejected:{signal}:{actor}:{reason?}` string supplied by the caller.
4. Return as `object` so the runtime contract stays domain-agnostic (mirrors `ResumeAsync`).

**Stale-doc fix on `ResumeAsync` (per D8):** replace "asserts the aggregate is in the Failed state" with "asserts the aggregate is in Failed or Suspended state" in the XML doc comment.

### 2.4 Resume-vs-cancel branching

**ApproveWorkflowCommand handler:**
1. Load aggregate.
2. Precondition: `Status == Suspended` AND the most recent `WorkflowExecutionSuspendedEvent.Reason` starts with `human_approval` (and the signal segment matches `Decision.ApprovalKey` when an explicit key is supplied — deferred per D2).
3. **Source authoritative approver identity from `CommandContext.Actor`.** Compose the carrier string: `"human_approval_granted:" + signal + ":" + actor`, where `signal` is extracted from the preceding Suspended event's Reason and `actor` is the authoritative identity rendered in a canonical short form (e.g. `CommandContext.Actor.Id` or display token). The actor segment in the carrier is a **non-authoritative observability mirror**; audit, authorization, and trust decisions MUST resolve the authoritative actor from the command envelope and `PolicyEvaluatedEvent` — never from the carrier text.
4. Call `WorkflowLifecycleEventFactory.Resumed(aggregate, approvalSignalOverride: carrier)` (new overload — see D7) to obtain a `WorkflowExecutionResumedEvent` whose `PreviousFailureReason` is the composed carrier. The existing precondition (`Failed` OR `Suspended`) is preserved.
5. Prepend the event to the workflow's accumulated event list per existing persist pipeline. Persist → chain → outbox handle propagation.
6. Re-enter `T1MWorkflowEngine.ExecuteAsync` via the dispatcher's existing resume path (per `R-WF-RESUME-02..05`). Cursor from `WorkflowExecutionReplayState.NextStepIndex`.

**RejectWorkflowCommand handler:**
1. Load aggregate.
2. Precondition: identical to Approve (`Status == Suspended` AND Reason prefix `human_approval`).
3. Source authoritative approver identity from `CommandContext.Actor`. Compose the carrier string: `"human_approval_rejected:" + signal + ":" + actor + (":" + Decision.Rationale if short)`. Same non-authoritative caveat as Approve — the actor segment is observability, not trust.
4. Call `CancelSuspendedAsync(workflowId, carrier)` → receive pre-baked `WorkflowExecutionCancelledEvent`.
5. Append event; persist → chain → outbox.
6. Workflow is terminal (Cancelled status); no engine re-entry.

### 2.5 Projection extension

Extend `WorkflowExecutionProjectionHandler` to subscribe to and reduce `WorkflowExecutionSuspendedEventSchema` and `WorkflowExecutionCancelledEventSchema`. **Canonical lifecycle `Status` is preserved** — `"Suspended"` and `"Cancelled"` remain the authoritative status values, aligned with `WorkflowExecutionStatus` on the aggregate.

Add three derived approval-context fields to `WorkflowExecutionReadModel` for UX/query clarity per D6:

| Field | Type | Source | Values |
|---|---|---|---|
| `ApprovalState` | `string?` | parsed from the latest Suspended/Resumed/Cancelled Reason's canonical prefix | `"AwaitingApproval"` (Suspended with `human_approval` prefix) / `"Granted"` (Resumed with `human_approval_granted` prefix) / `"Rejected"` (Cancelled with `human_approval_rejected` prefix) / `null` (non-approval Suspend/Cancel) |
| `ApprovalSignal` | `string?` | parsed `{signal}` segment from the carrier | step-scoped signal string or null |
| `ApprovalDecision` | `string?` | parsed `{reason?}` suffix from the Cancelled carrier (when rejected); null on Suspended/Granted | short rationale or null |

Projection reducer contract for each new event:

- `WorkflowExecutionSuspendedEventSchema` → upsert `Status = "Suspended"`; set `ApprovalState = "AwaitingApproval"` and `ApprovalSignal = <parsed>` when Reason starts with `human_approval`; otherwise leave `ApprovalState` null (e.g. future timer/external-dep suspends).
- `WorkflowExecutionCancelledEventSchema` → upsert `Status = "Cancelled"`; set `ApprovalState = "Rejected"` and `ApprovalDecision = <parsed>` when Reason starts with `human_approval_rejected`; otherwise leave `ApprovalState` null.
- Existing `WorkflowExecutionResumedEventSchema` reducer — extend to set `ApprovalState = "Granted"` and clear `ApprovalDecision` when `PreviousFailureReason` starts with `human_approval_granted`; `Status = "Running"` and `FailedStepName`/`FailureReason` clearing behaviour is unchanged.

**Rationale (per D6):** authoritative lifecycle `Status` tracks domain truth (Running/Suspended/Failed/Cancelled/Completed); derived `ApprovalState` carries approval-specific semantics without polluting the canonical enum. Cancelled stays terminal; "Rejected" is approval context on top of that terminal lifecycle state.

Idempotency / replay-safety: preserved via existing `LastEventId` guard; log-and-skip on missing prior state per current convention.

### 2.6 `IPayloadTypeRegistry` interaction

Register `ApprovalDecisionPayload` in the orchestration-system bootstrap module (`WorkflowExecutionBootstrap.RegisterPayloadTypes`). Required so the command payload round-trips through Postgres-backed persistence with a stable type discriminator per `R-WF-PAYLOAD-REGISTRY-01`.

## 3. Proposed Guard Rules (to promote into `runtime.guard.md` Workflow section)

- **`R-WF-APPROVAL-01`** — A step returning `WorkflowStepResult.AwaitingApproval(...)` MUST cause `T1MWorkflowEngine` to emit `WorkflowExecutionSuspendedEvent` via `WorkflowLifecycleEventFactory.Suspended(...)` with `Reason` starting with the canonical `human_approval` prefix. Direct event-store writes bypassing the factory are FORBIDDEN. **Severity:** S1.
- **`R-WF-APPROVAL-02`** — `ApproveWorkflowCommand` / `RejectWorkflowCommand` handlers MUST reject with a canonical `CannotApproveUnlessAwaitingApproval` / `CannotRejectUnlessAwaitingApproval` reason when the aggregate `Status != Suspended` OR the latest `WorkflowExecutionSuspendedEvent.Reason` does not start with `human_approval`. **Severity:** S1.
- **`R-WF-APPROVAL-03`** — `ApproveWorkflowCommand` resume path MUST re-enter the existing `IWorkflowExecutionReplayService.ResumeAsync` seam (no parallel `ApproveAsync`). The emitted `WorkflowExecutionResumedEvent.PreviousFailureReason` MUST start with the canonical `human_approval_granted` prefix when the preceding state was Suspended-for-approval. **Note (per D4):** this field-overloading is a temporary carrier convention; `PreviousFailureReason` does NOT denote failure semantics when the prefix is `human_approval_granted`. **Severity:** S1.
- **`R-WF-APPROVAL-04`** — `RejectWorkflowCommand` cancel path MUST call `IWorkflowExecutionReplayService.CancelSuspendedAsync(...)`. The emitted `WorkflowExecutionCancelledEvent.Reason` MUST start with the canonical `human_approval_rejected` prefix. The service method MUST assert `Status == Suspended` before constructing the event. **Severity:** S1.
- **`R-WF-APPROVAL-05`** — New approval-specific lifecycle event types (e.g. `WorkflowExecutionAwaitingApprovalEvent`, `WorkflowExecutionApprovedEvent`, `WorkflowExecutionRejectedEvent`) are DISALLOWED in `src/domain/**` and `src/shared/contracts/events/**`. Approval state transitions ride on the canonical Suspended/Resumed/Cancelled triple. **Severity:** S1.
- **`R-WF-APPROVAL-06`** — Approval context (approver identity, approval decision, full rationale, attachments) MUST be carried via (a) Reason prefix/suffix conventions on the reused events, (b) command envelope per `R-PLAT-12` (`PolicyEvaluatedEvent`, `DecisionHash`, `CorrelationId`, `CausationId`, actor propagation), and (c) `ApprovalDecisionPayload` registered in `IPayloadTypeRegistry`. Approval context MUST NOT be duplicated onto lifecycle event fields beyond the canonical signal prefix + short suffix. **Severity:** S2.
- **`R-WF-APPROVAL-07`** (additional authoritative-actor constraint) — Authoritative approver identity for `ApproveWorkflowCommand` / `RejectWorkflowCommand` MUST be sourced from `CommandContext.Actor` (runtime identity middleware output). Any actor segment written into the `human_approval_granted` / `human_approval_rejected` Reason carrier is a denormalized observability mirror and is NON-AUTHORITATIVE. Authorization decisions, audit resolution, projection trust boundaries, and any downstream consumer MUST resolve the authoritative actor from the command envelope and `PolicyEvaluatedEvent` — never by parsing the carrier text. Implementations MAY optionally validate that the carrier actor matches `CommandContext.Actor` at emit time (defence-in-depth) but MUST NOT trust the carrier actor on re-read. **Severity:** S1.

Projection-specific rule augmenting the above (per D6):

- **`R-WF-APPROVAL-PROJ-01`** — `WorkflowExecutionProjectionHandler` MUST preserve canonical lifecycle `Status` values (`Suspended`, `Cancelled`, `Running`, `Failed`, `Completed`) aligned with `WorkflowExecutionStatus`. Approval-specific semantics are carried via derived fields (`ApprovalState`, `ApprovalSignal`, `ApprovalDecision`), never by overloading `Status` with values like `"AwaitingApproval"` or `"Rejected"`. **Severity:** S2.

Matching audit entries added to `claude/audits/runtime.audit.md` Workflow section with the same IDs.
