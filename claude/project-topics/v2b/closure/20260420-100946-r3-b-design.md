# R3.B Design Note — External Side-Effect Control

**Status:** RATIFIED 2026-04-20. D-R3B-1/3/4/10 locked with additional constraints (§17). R3.B.1 Step 1 design note proceeding next.
**Scope:** §15 closure — enterprise-grade outbound side-effect control. 6 S1 rows per the closure matrix.
**Builds on:** R3.A.6 (human-approval wait-state pattern — reuse-first lifecycle-event discipline will carry over to outbound-effect aggregates).
**Timestamp:** 2026-04-20 10:09:46Z.

---

## Survey baseline (grounding)

- **HTTP/external-client surface today:** one sanctioned adapter family — `OpaPolicyEvaluator` + `OpaWarmupService` + `OpaEnforcementEventEvaluator` (governed by `OpaOptions`, circuit breaker, meter, linked-CTS timeout). Zero drift elsewhere.
- **Third-party SDKs:** none. Settlement `SettlementProvider` is an opaque value object — no protocol logic in domain.
- **Outbox:** Kafka-event-specific today (`IOutbox`, `PostgresOutboxAdapter`, `KafkaOutboxPublisher`). No generic intent→execution seam.
- **Webhook / notification / integration domain:** aggregates exist as contract-definition layer only (`WebhookAggregate`, `DeliveryAggregate`, `ChannelAggregate`, `TemplateAggregate`, `RegistryAggregate`, `CommandBridgeAggregate`, `EventBridgeAggregate`). Lifecycles are defined; execution layer is missing.
- **Compensation:** rich internal compensation for payouts (`PayoutCompensationWorkflow`, T1M/T2E + Kafka routing). Reconciliation aggregates exist (`DiscrepancyAggregate`, `ProcessAggregate`). External-effect compensation path is absent.
- **Finality / ack surface:** placeholder value object `SettlementFinality` exists in shared contracts; no populated acknowledgement logic.

**Implication:** R3.B is a greenfield governance opportunity. We design the canonical seam now — before many external adapters land — and codify guards that prevent any future drift.

## 1. External Side-Effect Taxonomy

### 1.1 Effect categories

| Category | Definition | Examples | Finality shape |
|---|---|---|---|
| **Notification** | Fire-and-forget outbound message with no business-critical finality tied to provider response | Email / SMS / push notification | Transport ack only |
| **Integration (command)** | Request to a partner system instructing it to do something | Send command to partner ERP; call credit-reporting API | Provider ack + optional async business finality |
| **Integration (event-bridge)** | Publish our event into a partner topic / webhook target | Publish `InvoicePaidEvent` to partner webhook | Transport ack (partner receipt) + optional async business finality |
| **Transactional** | Outbound call whose business outcome mirrors local state and MUST reconcile | Settle payment with Stripe / Wise; provision resource on cloud provider | Provider ack + async business finality (callback or poll) |
| **Webhook delivery (outbound)** | We are the webhook source; partner is the receiver; includes delivery guarantees, signing, retry | Webhook push to partner on event | Transport ack + optional signed receipt |
| **Inbound callback ingress** | Partner-initiated webhook / ack coming back to us (the dual of outbound) | Stripe webhook confirming `payment_intent.succeeded` | Not an outbound effect — consumed as inbound command through existing dispatch pipeline |

R3.B scope covers rows 1-5. Inbound callback ingress (row 6) is in scope only insofar as it reconciles against outbound-effect state — it does NOT introduce a new outbound seam.

### 1.2 Idempotency shape

| Shape | Description | Dedup strategy |
|---|---|---|
| **Provider-idempotent** | Provider accepts an idempotency key (Stripe, Square, modern APIs) | Deterministic key derived from effect ID sent to provider |
| **Natural-key-idempotent** | Provider treats repeated call with same business key as no-op (e.g., upsert by external reference) | Natural key from domain aggregate |
| **At-most-once-required** | Provider has no idempotency surface; duplicate = duplicate (email send, SMS, legacy bank transfer) | Local single-claim on durable intent row; never retry after confirmed dispatch |
| **Compensatable-only** | Duplicate is possible and costly but reversible (bank reversal, refund) | Single-claim + compensation on duplicate detection |

Every outbound-effect adapter MUST declare its idempotency shape.

### 1.3 Classification consequences

A provider's `(category, idempotency shape)` pair determines:
- Whether retry is safe under transient failure (yes for provider-idempotent and natural-key-idempotent; NO for at-most-once unless first call definitively failed)
- Whether reconciliation can be automatic (yes if finality can be polled) or must be operator-driven (if provider has no finality API)
- Whether the aggregate owning the effect needs a compensation workflow (mandatory for transactional; optional for notification)

## 2. Canonical Side-Effect Lifecycle / State Model

R3.B introduces a canonical aggregate: **`OutboundEffectAggregate`** in a new domain context `src/domain/integration-system/outbound-effect/` (classification: `integration-system`; context: `outbound-effect`; no domain-group level — single-domain context).

### 2.1 State machine

```
         Scheduled ──▶ Dispatched ──▶ Acknowledged ──▶ Finalized
            │              │               │               │
            │              │               │               ├──▶ Succeeded (terminal)
            │              │               │               ├──▶ BusinessFailed (terminal)
            │              │               │               └──▶ PartiallyCompleted (terminal, operator-reviewed)
            │              │               │
            │              │               └──▶ ReconciliationRequired ──▶ Reconciled (Succeeded|Failed|PartiallyCompleted)
            │              │                                           └──▶ ManualIntervention (terminal, operator-owned)
            │              │
            │              └──▶ TransientFailed ──▶ Scheduled (retry)
            │                                  └──▶ RetryExhausted (terminal refusal — triggers compensation path if defined)
            │
            └──▶ Cancelled (pre-dispatch cancel; terminal)
```

### 2.2 State semantics

| State | Meaning | Authoritative event |
|---|---|---|
| **Scheduled** | Durable intent persisted; not yet dispatched. Can be cancelled before dispatch. | `OutboundEffectScheduledEvent` |
| **Dispatched** | Adapter invoked the provider. Transport-level call completed (HTTP 2xx received OR connection-level success). | `OutboundEffectDispatchedEvent` |
| **Acknowledged** | Provider returned an operation identifier / confirmation code indicating "accepted for processing." Not yet business-final. | `OutboundEffectAcknowledgedEvent` |
| **Finalized** | Business outcome is known: provider confirmed terminal state via async callback or poll. Resolves into one of the three terminal sub-states. | `OutboundEffectFinalizedEvent(outcome)` |
| **ReconciliationRequired** | Local view disagrees with provider view OR provider ack arrived without subsequent finality inside the finality window. Operator / automated reconciler MUST resolve. | `OutboundEffectReconciliationRequiredEvent(cause)` |
| **TransientFailed** | Dispatch failed with a retryable classification (timeout, breaker-open, 5xx, network). | `OutboundEffectDispatchFailedEvent(classification: Transient)` |
| **RetryExhausted** | Max attempts reached. Terminal refusal — ignites compensation path if aggregate has one. | `OutboundEffectRetryExhaustedEvent` |
| **Cancelled** | Pre-dispatch cancellation (caller abort, policy disallow, deadline expired before dispatch). | `OutboundEffectCancelledEvent` |

### 2.3 Finality sub-outcomes (on `Finalized`)

- **Succeeded** — provider reports success.
- **BusinessFailed** — provider reports terminal failure (insufficient funds, card declined, recipient rejected, etc.). Triggers compensation per §9.
- **PartiallyCompleted** — provider reports partial success (e.g., 4 of 5 recipients delivered). Requires operator / policy resolution.

### 2.4 Reconciliation triggers (on `ReconciliationRequired`)

- `AckTimeoutExpired` — dispatched but no ack within `Provider.AckTimeoutMs`.
- `FinalityTimeoutExpired` — acknowledged but no finality within `Provider.FinalityWindowMs`.
- `ProviderDisagreement` — poll or callback disagrees with our expected outcome.
- `DispatchAmbiguous` — HTTP-level ambiguity (timeout mid-response, read-side ack failure).

### 2.5 Replay discipline

- State is reconstructed from the event stream on the `OutboundEffectAggregate`.
- Reasons/causes on events are audit-only — not replay discriminators.
- Carrier conventions (see §10) encode provider, idempotency key, attempt number, finality outcome deterministically.

## 3. Sanctioned Outbound Execution Seam

### 3.1 Core contracts (new, under `src/shared/contracts/runtime/outbound-effects/`)

```csharp
public interface IOutboundEffectDispatcher
{
    Task<OutboundEffectId> ScheduleAsync(
        OutboundEffectIntent intent, CommandContext context, CancellationToken ct);
}

public sealed record OutboundEffectIntent(
    string ProviderId,           // "stripe", "twilio/sms", "partner-xyz/webhook"
    string EffectType,           // "payment.settle", "notification.send", "webhook.deliver"
    string IdempotencyKey,       // deterministic, caller-supplied — single-claim on this
    object Payload,              // CLR type registered in IPayloadTypeRegistry
    TimeSpan? AckTimeout = null, // override Provider default
    TimeSpan? FinalityWindow = null,
    int? MaxAttempts = null);

public interface IOutboundEffectAdapter
{
    string ProviderId { get; }
    OutboundIdempotencyShape IdempotencyShape { get; }
    Task<OutboundAdapterResult> DispatchAsync(OutboundEffectDispatchContext ctx, CancellationToken ct);
}

public enum OutboundIdempotencyShape
{
    ProviderIdempotent,
    NaturalKeyIdempotent,
    AtMostOnceRequired,
    CompensatableOnly,
}
```

### 3.2 Layer placement

- **Contracts:** `src/shared/contracts/runtime/outbound-effects/` — reachable from runtime, engines (for intent composition), and domain (for the payload type registry).
- **Dispatcher implementation:** `src/runtime/outbound-effects/OutboundEffectDispatcher.cs` — runtime-owned (R7 — only runtime persists / publishes).
- **Adapter implementations:** `src/platform/host/adapters/outbound-effects/{Provider}Adapter.cs` — infrastructure. Each adapter is a thin wrapper over the provider SDK / HTTP client.
- **Domain:** `src/domain/integration-system/outbound-effect/` — `OutboundEffectAggregate` + lifecycle events. Pure. No SDK references.
- **Projection:** `src/projections/integration-system/outbound-effect/` — read-model for operator surface (R4 feed).

### 3.3 Execution flow

1. Domain / engine calls `IOutboundEffectDispatcher.ScheduleAsync(intent, context, ct)`.
2. Dispatcher creates `OutboundEffectAggregate` and emits `OutboundEffectScheduledEvent`. The aggregate is persisted via the canonical runtime persist → chain → outbox pipeline (the **event** goes to Kafka like any domain event). Durable intent is now in the event store.
3. A separate **OutboundEffectRelay** background service (equivalent to `KafkaOutboxPublisher` but for outbound effects) polls an `outbound_effect_dispatch_queue` table or a dedicated partition / topic, claiming rows in `Scheduled` state and invoking the registered adapter for `intent.ProviderId`.
4. Adapter returns one of: `Acknowledged(providerOpId)`, `TransientFailed(classification, retryAfter?)`, `BusinessFailed(reason)`, `DispatchAmbiguous(cause)`.
5. Relay emits the corresponding lifecycle event onto the aggregate — `OutboundEffectDispatchedEvent` + `OutboundEffectAcknowledgedEvent` / `OutboundEffectDispatchFailedEvent`. Event goes through the canonical pipeline.
6. Finality:
   - **Push mode** (webhook): an inbound callback handler validates the signature, correlates to `ProviderOperationId`, and dispatches an internal command (e.g. `FinalizeOutboundEffectCommand`) that emits `OutboundEffectFinalizedEvent`.
   - **Poll mode** (API check): a scheduled `OutboundEffectFinalityPoller` queries the provider at `FinalityWindowMs/N` intervals, same event on success.
   - **Reconciliation mode**: if neither happens within `FinalityWindowMs`, the poller emits `OutboundEffectReconciliationRequiredEvent`.

## 4. Uniform Outbox-Based Dispatch Rules

### 4.1 Two concerns, reuse existing pattern

- **Durable intent → execution** for outbound effects is a NEW surface (§3).
- **Domain event → Kafka topic** remains the existing `KafkaOutboxPublisher` path (unchanged).

Both are durable-relay patterns. R3.B adds a second relay (`OutboundEffectRelay`) that is semantically equivalent but with different storage shape.

### 4.2 Storage

- **Table:** `outbound_effect_dispatch_queue` (new) — columns: `effect_id UUID PK`, `aggregate_id UUID`, `provider_id TEXT`, `effect_type TEXT`, `idempotency_key TEXT`, `status TEXT`, `attempt_count INT`, `next_attempt_at TIMESTAMPTZ`, `last_error TEXT`, `claimed_by TEXT NULL`, `claimed_at TIMESTAMPTZ NULL`, `created_at TIMESTAMPTZ`.
- The `outbound_effect_dispatch_queue` is a **projection of outbound-effect lifecycle events** — not the source of truth. Source of truth is the event stream on `OutboundEffectAggregate`. The queue exists for relay-side performance (avoid replaying the full event store to find pending effects).

### 4.3 Rules

- **R-OUT-EFF-OUTBOX-01:** Every outbound effect MUST originate as an `OutboundEffectScheduledEvent` on an `OutboundEffectAggregate`. No adapter execution without a persisted `Scheduled` event.
- **R-OUT-EFF-OUTBOX-02:** Only `OutboundEffectRelay` may invoke an `IOutboundEffectAdapter.DispatchAsync`. No direct adapter invocations from engines / systems / domain.
- **R-OUT-EFF-OUTBOX-03:** The relay claims a row via single-claim semantics (identical to `KafkaOutboxPublisher` — single-row update with `claimed_by = host_id WHERE claimed_by IS NULL`). Concurrent claims yield no duplicate dispatch.
- **R-OUT-EFF-OUTBOX-04:** Adapter result is recorded as a lifecycle event on the aggregate BEFORE the queue row is updated (event-first, projection-second — matches the existing CQRS rule GE-05).
- **R-OUT-EFF-OUTBOX-05:** Relay operates on a single queue per host; horizontal scale is achieved via `claimed_by` host-partitioning (identical to outbox claim model).

## 5. Duplicate Suppression / Outbound Idempotency Model

### 5.1 Three layers of dedup

| Layer | Where | Guarantee |
|---|---|---|
| **Local single-claim** | `outbound_effect_dispatch_queue.claimed_by` | Prevents two relay workers from dispatching the same effect concurrently |
| **Replay-safe retry** | Adapter sends provider's idempotency key (derived from `effect_id` — stable across retries) | Provider-side collapses duplicate calls |
| **Effect-intent dedup** | `OutboundEffectAggregate.IdempotencyKey` is unique per scheduled effect; scheduling an effect with a duplicate key is a no-op (returns the existing `OutboundEffectId`) | Prevents the domain from scheduling the same logical effect twice (e.g., replay of a command) |

### 5.2 Rules

- **R-OUT-EFF-IDEM-01:** `OutboundEffectIntent.IdempotencyKey` is REQUIRED. Derivation convention: `{aggregateRootType}:{aggregateRootId}:{effectType}:{businessKey}` (e.g., `PayoutAggregate:payout-123:payment.settle:attempt-1`). Caller's responsibility; architecture test asserts non-null.
- **R-OUT-EFF-IDEM-02:** The dispatcher DOES NOT create a new aggregate when called with a duplicate `IdempotencyKey` — it returns the existing `OutboundEffectId` and logs a dedup hit (metric `outbound.effect.schedule.dedup_hit`).
- **R-OUT-EFF-IDEM-03:** Every adapter that can supply a provider idempotency key MUST supply it (`Stripe-Idempotency-Key: {effect_id}` or equivalent). Adapter declares `IdempotencyShape = ProviderIdempotent` when it does. Grep asserts any HTTP call in `DispatchAsync` carries the key header for `ProviderIdempotent` adapters.
- **R-OUT-EFF-IDEM-04:** Retries reuse the same `effect_id` as idempotency key — never regenerate on retry.
- **R-OUT-EFF-IDEM-05:** `AtMostOnceRequired` adapters MUST NOT retry after `Dispatched`. A single local attempt → either `Acknowledged` or `TransientFailed (pre-dispatch)` or `BusinessFailed` — never `TransientFailed (post-dispatch)` → retry. Architecture-level adapter category flag.

## 6. Third-Party Timeout / Cancellation Discipline

### 6.1 Two-tier deadline (mirrors `T1MWorkflowEngine.WorkflowOptions`)

- **`Provider.DispatchTimeoutMs`** — per-attempt deadline. Linked CTS from caller token + provider config. On expiry → `TransientFailed(Timeout)`.
- **`Provider.TotalBudgetMs`** — cumulative deadline across all retry attempts. On expiry → `RetryExhausted`.

Both are declared per provider in `OutboundEffectOptions.{ProviderId}.*` — typed options, NOT magic values in adapter code.

### 6.2 Circuit breaker per provider

- Reuse the existing `ICircuitBreaker` pattern (R2.A.D — OPA breaker is the template).
- Each adapter registers its breaker by provider ID (`breaker-name = outbound.{providerId}`).
- Breaker-open during dispatch → `TransientFailed(BreakerOpen, retryAfterSeconds)` — the dispatch attempt doesn't count against the retry budget, because the provider was never actually called.
- Breaker state rolls into `RuntimeStateAggregator.CanonicalBreakerReasons` so operator dashboards see per-provider breaker posture.

### 6.3 Cancellation propagation

- Every adapter takes `CancellationToken`. Caller cancellation propagates as `OperationCanceledException` → cancel the dispatch; emit `OutboundEffectCancelledEvent` if pre-dispatch.
- Host shutdown: in-flight dispatches are allowed to complete; scheduled (not yet claimed) effects carry over to next host boot via durable queue.

### 6.4 Rules

- **R-OUT-EFF-TIMEOUT-01:** Every adapter MUST use a linked CTS derived from the caller token + `DispatchTimeoutMs`. Direct `HttpClient.SendAsync(..., ct)` without linked CTS is FORBIDDEN. Architecture test enforces.
- **R-OUT-EFF-TIMEOUT-02:** Per-provider options (`DispatchTimeoutMs`, `TotalBudgetMs`, `AckTimeoutMs`, `FinalityWindowMs`) are REQUIRED — no hardcoded defaults. Composition root throws on missing config.
- **R-OUT-EFF-TIMEOUT-03:** Timeouts are classified as `Transient` for retry classification (unless the adapter's `TotalBudgetMs` has expired, in which case → `RetryExhausted`).

## 7. Retry / Exhaustion / DLQ Rules for Outbound Work

### 7.1 Retry policy (mirrors `T1MWorkflowEngine` retry loop + jitter)

- **Classification:** adapter result drives retry decision.
  - `TransientFailed` → retry (subject to `MaxAttempts` and `TotalBudgetMs`).
  - `BusinessFailed` → no retry; move to `Finalized(BusinessFailed)` directly; ignites compensation path (§9).
  - `DispatchAmbiguous` → retry is DANGEROUS (could duplicate). Policy-gated: for `ProviderIdempotent` or `NaturalKeyIdempotent` → retry; for `AtMostOnceRequired` → `ReconciliationRequired`.
  - `AdapterConfigError` / `PolicyDenied` → no retry; terminal refusal; alert operator.
- **Backoff:** exponential with capped max, deterministic per attempt (`base * 2^(attempt-1)`, capped at `MaxBackoffMs`). Seeded jitter via `IRandomProvider` (introduced in R1) for horizontal-scale smoothing.
- **Next attempt scheduling:** `next_attempt_at = IClock.UtcNow + backoff` — persisted on the queue row.

### 7.2 DLQ discipline

- `RetryExhausted` is a first-class terminal state. The aggregate stays in `RetryExhausted` until (a) compensation workflow consumes it and transitions to `Compensated`, or (b) operator explicitly moves to `ManualIntervention`.
- No physical DLQ topic needed — the `RetryExhausted` state IS the DLQ. Read-model projects the set for operator surface (R4).
- Dedicated metric `outbound.effect.retry_exhausted{provider, effect_type}` tracks exhaustion rate.

### 7.3 Rules

- **R-OUT-EFF-RETRY-01:** Retry loop lives in `OutboundEffectRelay` — adapters are stateless and ONLY report classification. No in-adapter retry loops.
- **R-OUT-EFF-RETRY-02:** Retry increments `attempt_count` on the queue row AND emits `OutboundEffectRetryAttemptedEvent` (or batched attempt-summary on exhaustion — trade-off to ratify per D-R3B-2).
- **R-OUT-EFF-RETRY-03:** `RetryExhausted` is terminal on the lifecycle unless a compensation workflow resolves it. Compensation is domain-level — NOT automatic.

## 8. External Finality / Acknowledgement / Reconciliation Model

### 8.1 Three distinct stages

| Stage | Signal | State after | Typical latency |
|---|---|---|---|
| **Transport ack** | HTTP 2xx received OR connection success | `Dispatched` | ms–s |
| **Provider ack** | Provider returned operation id / status="accepted" / idempotency-receipt | `Acknowledged` | ms–s (or ack-timeout) |
| **Business finality** | Provider reports terminal outcome via callback / poll | `Finalized(outcome)` | s–days |

Each stage is distinct — **conflating them is explicitly forbidden**. An adapter that returns only `Dispatched` without `Acknowledged` is acceptable only if the provider has no ack surface (rare — most modern providers return an operation id in the response body).

### 8.2 Finality resolution strategies

- **Push (webhook)** — PREFERRED. Partner calls our callback endpoint; we correlate by `ProviderOperationId`; emit finality event.
- **Poll** — scheduled poll at `FinalityWindowMs/N` intervals. Circuit-broken + backoff.
- **Hybrid** — push first, poll as fallback after `AckTimeoutMs` silence.

Finality-resolution strategy is declared per adapter and selected at registration time.

### 8.3 Reconciliation-required is a first-class outcome

Per the user's non-negotiable: **reconciliation is not an exception path**. It has its own lifecycle state (§2), its own event (`OutboundEffectReconciliationRequiredEvent(cause)`), its own projection row state, its own operator surface (R4), and its own canonical resolution ritual:

1. Reconciler (human or automated) investigates — typically by polling the provider with the known `ProviderOperationId`.
2. Reconciler emits `OutboundEffectReconciledCommand(outcome, evidence)` with `outcome ∈ {Succeeded, BusinessFailed, PartiallyCompleted, ManualIntervention}`.
3. Aggregate transitions to `Reconciled(outcome)` — replay-equivalent to the Finalized path.

### 8.4 Rules

- **R-OUT-EFF-FINALITY-01:** Adapter MUST return one of the canonical `OutboundAdapterResult` variants. Returning "success" without explicit `(Acknowledged | Finalized)` discrimination is FORBIDDEN.
- **R-OUT-EFF-FINALITY-02:** Every non-trivial effect type MUST declare its finality-resolution strategy at adapter registration. Unregistered effect types cannot be scheduled.
- **R-OUT-EFF-FINALITY-03:** `OutboundEffectReconciliationRequiredEvent` is emitted deterministically on `AckTimeout` / `FinalityWindow` expiry — NOT on ad-hoc errors or unexpected states. The expiry clock uses `IClock` per determinism.
- **R-OUT-EFF-FINALITY-04:** Callback/webhook inbound handlers MUST correlate to an existing `ProviderOperationId` on an `OutboundEffectAggregate`. Orphan callbacks (no matching aggregate) emit `OrphanProviderCallbackReceivedEvent` to the reconciliation surface — NEVER silently discarded.

## 9. Compensation Boundaries for External Failures

### 9.1 Three compensation triggers

1. **Finalized(BusinessFailed)** — provider reported terminal failure; local state diverges from intent → compensation workflow.
2. **RetryExhausted** — we never got past `TransientFailed`; no provider commitment → local state rollback (usually not compensation per se, but state-revert).
3. **Reconciled(BusinessFailed | PartiallyCompleted)** — reconciliation confirmed divergence → compensation workflow or operator-driven partial unwind.

### 9.2 Compensation is domain-level, not runtime-level

- R3.B introduces NO generic auto-compensation. Compensation is a domain workflow (analogous to existing `PayoutCompensationWorkflow`).
- The aggregate owning the effect — the "caller" — observes the lifecycle event (via projection or direct integration handler) and dispatches its own compensation workflow if one exists.
- Runtime emits a canonical `OutboundEffectCompensationRequestedEvent` on the effect aggregate as a signal; subscribers decide what to do.

### 9.3 Rules

- **R-OUT-EFF-COMP-01:** Effect aggregates with `IdempotencyShape ∈ {AtMostOnceRequired, CompensatableOnly}` MUST have a registered compensation workflow at the domain-integration layer. Unregistered = S1 fail.
- **R-OUT-EFF-COMP-02:** Compensation workflows consume `OutboundEffectCompensationRequestedEvent` — they do NOT directly manipulate the effect aggregate's state. The effect aggregate is event-sourced by the compensation workflow emitting its own events (analogous to T1M compensation pattern).
- **R-OUT-EFF-COMP-03:** Terminal failures without a registered compensation path raise an operator alert via metrics + the reconciliation operator surface (R4).

## 10. Audit / Evidence / Observability Model

### 10.1 Lifecycle events (all on `OutboundEffectAggregate`)

| Event | Purpose | Carrier fields (beyond AggregateId) |
|---|---|---|
| `OutboundEffectScheduledEvent` | Intent persisted | ProviderId, EffectType, IdempotencyKey, PayloadType, CorrelationId, CausationId |

| `OutboundEffectAcknowledgedEvent` | Provider accepted | ProviderOperationId, ProviderAckMetadata (small JSON — size-capped) |

| `OutboundEffectDispatchFailedEvent` | Adapter failure | Classification (Transient/Terminal/Ambiguous), Reason, AttemptNumber |
| `OutboundEffectRetryAttemptedEvent` | Retry scheduled | AttemptNumber, NextAttemptAt, BackoffMs, PrecedingClassification |
| `OutboundEffectRetryExhaustedEvent` | All attempts failed | FinalClassification, TotalAttempts |

Every outbound-effect lifecycle event is chain-anchored via the canonical runtime pipeline (persist → chain → outbox). Tamper-evident audit trail.

### 10.4 Rules

- **R-OUT-EFF-AUDIT-02:** Metric vocabulary is locked. New metrics require guard-level promotion.
- **R-OUT-EFF-AUDIT-03:** Every outbound-effect lifecycle event carries `PolicyEvaluatedEvent` hash (R-PLAT-12) — the scheduler's policy evaluation propagates through the aggregate's audit envelope.

## 11. Architecture Guards Prohibiting Unsanctioned Direct Outbound Effects

### 11.1 Prohibition rules (new guard family `R-OUT-EFF-*`)

- **R-OUT-EFF-PROHIBITION-01 (S0):** Direct `HttpClient` / `HttpClientHandler` / `HttpMessageInvoker` / `WebClient` usage is FORBIDDEN outside sanctioned paths. Sanctioned paths whitelist:
  - `src/platform/host/adapters/OpaPolicyEvaluator.cs`, `OpaWarmupService.cs`, `OpaEnforcementEventEvaluator.cs` (R2.A.D grandfathered — policy evaluation is an internal dependency, not a business outbound effect).
  - `src/platform/host/adapters/outbound-effects/**/*Adapter.cs` — the ONLY place new outbound-effect adapters may live.
  - `src/runtime/outbound-effects/**` — relay and dispatcher plumbing (no direct HTTP calls — delegates to adapters).
  - CI grep: any new `HttpClient` usage outside the whitelist → fail.

Architecture test: every file under `src/platform/host/adapters/outbound-effects/**` implements `IOutboundEffectAdapter`.
- **R-OUT-EFF-PROHIBITION-04 (S1):** No outbound-effect scheduling from `src/domain/**` directly — domain may OBSERVE `OutboundEffectAggregate` lifecycle via projection, but scheduling happens through engines / runtime. Prevents domain from becoming impure.

### 11.2 Supporting rules

- **R-OUT-EFF-SEAM-01:** `IOutboundEffectDispatcher.ScheduleAsync` is the ONLY way to introduce a new outbound intent. Constructors of `OutboundEffectAggregate` are non-public outside the runtime layer (enforced via structural guard).
- **R-OUT-EFF-SEAM-02:** Adapters are stateless singletons registered in DI. No mutable per-call state on the adapter instance. Architecture-test pin.

### 11.3 Replay-determinism guards

- **R-OUT-EFF-DET-01:** `OutboundEffectRelay` and all adapters MUST use `IIdGenerator` + `IClock` + `IRandomProvider` only. `Guid.NewGuid` / `DateTime.UtcNow` / `Random` FORBIDDEN.
- **R-OUT-EFF-DET-02:** Adapter outputs MUST be deterministic given identical input + provider response — in particular, classification mapping is a pure function.

### 17.2 Additional constraints locked before R3.B.1

The following constraints MUST be enforced at R3.B.1 landing — not deferred:

1. **Acknowledged never implies Finalized.** The adapter result model MUST preserve six explicit outcomes: dispatch-failed-pre-acceptance, dispatched-without-provider-op-id, acknowledged-with-provider-op-id, finalized-success, finalized-failure, reconciliation-required. Conflation is a guard violation.
2. **Provider operation identity is first-class from R3.B.1.** Every `Acknowledged` event carries `{local_outbound_effect_id, provider_operation_id, correlation_id, causation_id, idempotency_key_used_externally}` as named fields — NOT as free-text observability.
3. **Outbound idempotency contract is mandatory at R3.B.1.** Even before the first production adapter, R3.B.1 defines: local idempotency key (required, non-null), provider idempotency key propagation rule (where supported), duplicate-suppression rule (dedup-on-schedule), replay-safe redispatch rule (retries reuse the same key).
4. **Webhook / poll / manual-reconciliation duality designed at R3.B.1 even if only one is implemented later.** The aggregate's lifecycle MUST compatibly host push finality (webhook ingress), pull finality (scheduled poller), and manual reconciliation resolution. No reshaping of the aggregate to accommodate a second modality after the first ships.
5. **Unsanctioned direct outbound effects become a hard architecture violation at R3.B.1.** Prohibition guards (`R-OUT-EFF-PROHIBITION-01..05`) are enforced from the day R3.B.1 lands, not deferred until adapters exist. The legal surface is empty-but-guarded on day one; subsequent checkpoints fill it in.
