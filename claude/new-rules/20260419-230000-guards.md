# R2.E.4 ‚Äî Topic Provisioning Alignment (2026-04-19)

Captured per CLAUDE.md $1c. Fourth R2.E deliverable. Closes the last ¬ß11 ABSENT row from the 2026-04-19 post-R2.A.D gap-matrix re-survey.

**Insight:** topic creation is already handled by `infrastructure/event-fabric/kafka/create-topics.sh` (operator-run during infra setup). The gap is NOT provisioning ‚Äî it is ALIGNMENT between what the runtime declares as required topics and what the broker actually has. Without verification, a missing topic surfaces as `UNKNOWN_TOPIC_OR_PARTITION` at consume time (slow degradation), not at startup (fail fast).

## R-TOPIC-ALIGNMENT-01 ‚Äî Startup-Time Topic Verification

- **CLASSIFICATION:** runtime / event-fabric / consumer / startup
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `TopicProvisioningHostedService` (an `IHostedService` registered in `KafkaInfrastructureModule.AddMessaging`) runs `KafkaTopicVerifier.FindMissingTopicsAsync(adminClient, KafkaCanonicalTopics.All, timeout)` during `StartAsync`. The service queries broker metadata via `IAdminClient.GetMetadata(timeout)` ‚Äî a single round-trip that returns all known topics ‚Äî and compares against the canonical declared set.
  - Missing topics log WARN per topic + one aggregate WARN summary. When `Kafka:FailIfTopicsMissing = true` (default `false`), the service throws `InvalidOperationException` during `StartAsync`, preventing the host from reaching "ready" until infrastructure is reconciled. Default `false` so existing deployments don't break ‚Äî opt-in via config.
  - Startup verification is controlled by `Kafka:VerifyTopicsOnStartup` (default `true`). Integration tests that spin up a bare broker without topics opt-out via this flag.
  - The hosted service runs BEFORE the consumer workers by virtue of registration order ‚Äî `KafkaInfrastructureModule.AddMessaging` registers this service, and projection/saga workers register later via their own composition modules.
- **CHECK:**
  - `KafkaInfrastructureModule.AddMessaging` registers `TopicProvisioningHostedService` as `IHostedService`.
  - `KafkaCanonicalTopics.All` is the single source of truth for expected topics in runtime. A comment in both `create-topics.sh` and `KafkaCanonicalTopics.cs` cross-references the other so the mirror is visible.

## R-TOPIC-CANONICAL-DECLARATION-01 ‚Äî Central Canonical Topic List

- **CLASSIFICATION:** runtime / event-fabric / consumer / api
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `src/platform/host/adapters/KafkaCanonicalTopics.cs` ‚Äî static class with `IReadOnlyList<string> All` property listing every topic the runtime uses. Mirrors `infrastructure/event-fabric/kafka/create-topics.sh`. Both files carry a cross-reference comment noting that the other must be kept in sync.
  - The list is flat strings (not `TopicSpec` records with partition counts) because at verify-time we only care about existence. Partition-count alignment is operator concern ‚Äî the bash script is the source of truth for sizing.
  - Future R2.E.5 may introduce `TopicSpec` for partition-count verification; for R2.E.4 existence-only is sufficient.
- **CHECK:**
  - `KafkaCanonicalTopics.All` returns a non-empty IReadOnlyList<string>.
  - Header comment in `KafkaCanonicalTopics.cs` references `infrastructure/event-fabric/kafka/create-topics.sh` and vice versa.

## R-TOPIC-VERIFIER-HELPER-01 ‚Äî Canonical Verifier Type

- **CLASSIFICATION:** runtime / event-fabric / consumer / api
- **SEVERITY:** S3
- **DESCRIPTION:**
  - `src/platform/host/adapters/KafkaTopicVerifier.cs` ‚Äî static class exposing `FindMissingTopicsAsync(IAdminClient, IEnumerable<string> expected, TimeSpan metadataTimeout, ILogger?, CancellationToken)` returning `IReadOnlyList<string>` of missing topic names.
  - Uses `IAdminClient.GetMetadata(timeout)` which is synchronous ‚Äî the async wrap uses `Task.Run` so cancellation propagates through the continuation. `KafkaException` from a broker timeout surfaces as a re-thrown exception (the caller decides verify vs fail-fast). `OperationCanceledException` propagates unchanged.
  - Stateless ‚Äî no internal state, no side effects. Safe to call from any thread.
  - Unit-testable via a minimal `IAdminClient` fake (only `GetMetadata` is consumed).
- **CHECK:**
  - `KafkaTopicVerifier.cs` exists.
  - Unit tests cover the three paths: (a) all topics present ‚Üí empty result, (b) some missing ‚Üí returns missing subset, (c) admin client throws ‚Üí exception propagates.

## R-TOPIC-PROVISIONING-HOSTED-SERVICE-01 ‚Äî Hosted Service Shape

- **CLASSIFICATION:** runtime / event-fabric / consumer / startup
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `TopicProvisioningHostedService.StartAsync(CancellationToken)`:
    1. If `Kafka:VerifyTopicsOnStartup = false` ‚Üí return immediately (test-friendly opt-out).
    2. Build `IAdminClient` from `AdminClientConfig { BootstrapServers }`.
    3. `var missing = await KafkaTopicVerifier.FindMissingTopicsAsync(admin, KafkaCanonicalTopics.All, 10s, logger, ct);`
    4. If `missing.Count > 0`: log WARN with each missing topic name + aggregate count.
    5. If `missing.Count > 0 && Kafka:FailIfTopicsMissing = true`: throw `InvalidOperationException`.
    6. Dispose admin client.
  - `StopAsync` is a no-op ‚Äî the service has no long-running state.
  - On `KafkaException` during metadata (broker unreachable), log ERROR and continue (do NOT fail host startup on transient broker issues ‚Äî operator visibility is enough, and the consumer workers will report their own health separately).
- **CHECK:**
  - `TopicProvisioningHostedService.cs` exists and implements `IHostedService`.
  - Three log paths: success (INFO), missing topics (WARN), metadata failure (ERROR).
  - `InvalidOperationException` thrown ONLY when both `FailIfTopicsMissing` is true AND missing count > 0.

## Out of scope for R2.E.4

- **Auto-create missing topics** ‚Äî R2.E.4 is verification-only. The bash script is the canonical provisioning tool. Auto-create would require operator buy-in on partition-count + replication-factor defaults for test/CI environments and is out of scope.
- **Partition-count verification** ‚Äî checking that `topic.partitions.count == expected` is R2.E.5. Existence is the primary alignment concern.
- **Retention / config verification** ‚Äî same ‚Äî operator concern, out of scope.
- **Cross-cluster / multi-region topic layout** ‚Äî not applicable to current single-broker deployment.
