## Captured finding

During P2.6.CS.0 pre-flight, enumerated mirror layers for `content` classification
across the canonical layer set:

| Layer                                              | Status |
|----------------------------------------------------|--------|
| `src/engines/T2E/content/`                         | ABSENT |
| `src/projections/content/`                         | ABSENT |
| `src/systems/downstream/content/`                  | UNVERIFIED |
| `infrastructure/policy/domain/content/`            | ABSENT |
| `infrastructure/event-fabric/kafka/topics/content/`| ABSENT |

Per `domain.guard.md` DS-R4, any domain at
`src/domain/{classification}-system/{context}/[{domain-group}/]{domain}/`
MUST have mirrored paths in all of the above. The canonical content-system
tree is LIVE with 485 files of domain artifacts, but no mirror layers exist.

This drift is **pre-existing** and is not introduced by the Phase 2.6
realignment. It is captured here for backlog because the realignment will
MOVE domain folders around — when mirror layers are eventually scaffolded,
they must follow the post-realignment layout (4-level grouped), not the
current legacy layout.

## Recommended follow-up

1. Freeze the content-system domain tree at end of Phase 2.6 (after CS.13).
2. Scaffold mirror layers against the post-realignment layout only.
3. Do NOT attempt to create mirror layers now — they would be built against
   a tree that is about to change.
