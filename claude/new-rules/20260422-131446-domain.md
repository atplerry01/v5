# Finding 1 — D-ERR-TYPING-01: Systemic raw framework exceptions in domain value objects

## CLASSIFICATION
domain

## SOURCE
Full audit of `src/domain/` against `domain.guard.md` D-ERR-TYPING-01.

## DESCRIPTION
80 domain files (~109 occurrences) throw raw framework exceptions (`ArgumentException`,
`InvalidOperationException`, `ArgumentNullException`) directly in value-object constructors and
aggregate guards, instead of using the shared-kernel `Guard.Against(condition, message)` helper
(which throws `DomainInvariantViolationException`) or BC-specific `{Bc}Errors` domain exceptions.

`Guard.Against(...)` exists at `src/domain/shared-kernel/primitives/kernel/guard/Guard.cs` and is
the canonical replacement.

Affected systems (non-exhaustive sample):
- `economic-system`: settlement, transaction (wallet, transactionId, transactionReference,
  settlementCurrency, settlementId, settlementProvider, settlementReferenceId), vault (account,
  metrics), exchange (fx/CurrencyPair, rate), enforcement (lock, restriction, sanction, rule),
  revenue (distribution)
- `constitutional-system`: policy (rule value objects, enforcement value objects)
- `trust-system`: credential (CredentialHashValue, CredentialDescriptor, CredentialId), identity
- `structural-system`: relationship-rules (SpvScopeConstraint)
- `operational-system`: sandbox/kanban value objects

## PROPOSED_RULE
DOM-ERR-FRAMEWORK-01 (supplements D-ERR-TYPING-01):
All domain value-object constructors and aggregate guard blocks MUST NOT throw
`ArgumentException`, `InvalidOperationException`, `ArgumentNullException`, or any BCL-derived
exception type directly. The canonical replacements are:
  (a) `Guard.Against(condition, message)` from `Whycespace.Domain.SharedKernel.Primitives.Kernel`
      (throws `DomainInvariantViolationException`)
  (b) `throw {Bc}Errors.{ErrorName}()` where a BC-specific error type is appropriate

Static check: `grep -rn "throw new (ArgumentException|InvalidOperationException|ArgumentNullException)" src/domain/` must return zero hits.

## SEVERITY
S1 — architectural (domain must not surface BCL exception types; caller cannot distinguish domain
invariant violations from programming errors; breaks typed error-handling at the engine boundary).
Block merge on new violations. Remediate existing violations in a dedicated sweep pass.

# Finding 2 — Guard canonical-12 table is stale: content-system and integration-system not listed

## CLASSIFICATION
guards

## SOURCE
Full audit of `src/domain/` — discovered 14 top-level system folders while guard states "exactly 12".

## DESCRIPTION
`domain.guard.md` §Domain Layer Purity table lists exactly 12 root systems. The actual
`src/domain/` contains 14 classification folders (excluding `shared-kernel`):
  - `content-system` — referenced in DS-R3a (4-level form registry) but absent from the table
  - `integration-system` — documented at ~line 1610 of the guard but absent from the table

Both systems are intentionally present; the guard body documents them. The canonical summary table
at the top of `domain.guard.md` was not updated when these systems were added.

This is a guard documentation drift per GUARD-VERS-002 (no silent guard mutation).

## PROPOSED_RULE
Update the canonical 12-systems table in `domain.guard.md` to include:
  - `content-system`: Asset and media lifecycle (documents, media, streaming)
  - `integration-system`: Outbound-effect and cross-system integration bridges
Making the authoritative count 14 root systems (plus `shared-kernel`).

## SEVERITY
S2 — structural documentation drift; no runtime impact; must be corrected to prevent future
"exactly 12" checks from incorrectly flagging the two new systems as violations.

# Finding 3 — Structural non-standard artifact folder: relationship-rules

## CLASSIFICATION
domain

## SOURCE
`src/domain/structural-system/structure/relationship-rules/` discovered during full domain scan.

## DESCRIPTION
Three files sit under `src/domain/structural-system/structure/relationship-rules/` in a folder
named `relationship-rules` — which is not one of the 7 mandatory artifact folders (`aggregate/`,
`entity/`, `error/`, `event/`, `service/`, `specification/`, `value-object/`).

Files: `SpvScopeConstraint.cs`, `AuthoritySubclusterConstraint.cs`, `AuthorityProviderMatrix.cs`.

These appear to be domain specification / constraint objects that should live in the `specification/`
artifact subfolder of the `structure` domain.

Also: these files throw `ArgumentNullException` directly (see Finding 1).

## PROPOSED_RULE
Files under `src/domain/**/` that serve as domain specifications or constraints MUST reside in the
`specification/` artifact subfolder, not in custom-named folders outside the mandatory-7. The
`relationship-rules/` folder under `structural-system/structure/` must be dissolved:
  - Move files to `src/domain/structural-system/structure/relationship-rules/specification/` or
    into the nearest containing domain's `specification/` folder.
  - Remove the non-standard `relationship-rules/` wrapper once migrated.

## SEVERITY
S2 — structural naming drift; does not break runtime but violates mandatory-artifact-folder rule
(domain.guard.md rule 2). Remediate in the next structural-system cleanup pass.

# Finding 4 — D-VO-TYPING-01: Raw Guid on policy decision events

## CLASSIFICATION
domain

## SOURCE
`src/domain/constitutional-system/policy/decision/event/PolicyEvaluatedEvent.cs` and
`PolicyDeniedEvent.cs`

## DESCRIPTION
Both constitutional-system policy decision events use raw `Guid` for three fields that logically
represent identity/correlation values:
  - `EventId { get; init; }` — Guid
  - `CorrelationId { get; init; }` — Guid
  - `CausationId { get; init; }` — Guid

Domain guard D-VO-TYPING-01 requires aggregates and events to use VO types for IDs, not raw
`Guid`/`string`. Both files are explicitly documented as sourcing these fields from upstream
runtime context (they do NOT generate new IDs), which is the correct design, but the field types
should still be wrapped in VO types.

The intentional use pattern (upstream-sourced IDs, no generation) is documented in the file's
XML comments and does not violate D-DET-01 (no generation occurs). However, the raw `Guid` type
means the domain cannot enforce invariants (e.g., non-empty) at the type boundary.

## PROPOSED_RULE
Track as a known exception to D-VO-TYPING-01 for policy-decision events if:
  (a) The doc comment explicitly states the IDs are upstream-sourced and not generated, AND
  (b) There is no alternative VO type in the shared-kernel that fits (e.g., `CorrelationId`,
      `CausationId` wrappers)

If appropriate VO types exist in `shared-kernel/`, the policy events should migrate to them.
If they don't exist, add `CorrelationId` and `CausationId` value objects to shared-kernel and
migrate the policy events in the next constitutional-system hardening pass.

## SEVERITY
S2 — structural typing discipline; does not affect runtime. Evaluate for remediation alongside
shared-kernel VO extensions.
