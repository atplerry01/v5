## FINDING 1 — Runtime contains Todo-typed projection bridge

**CLASSIFICATION:** runtime / projection / domain-leakage
**SEVERITY:** S1
**SOURCE:** `src/runtime/projection/bridges/TodoProjectionBridge.cs`
**DESCRIPTION:** `TodoProjectionBridge` implements a typed switch on `TodoCreatedEventSchema` / `TodoUpdatedEventSchema` / `TodoCompletedEventSchema`. Domain event types are referenced directly inside the runtime layer.
**PROPOSED RULE:** Runtime projection bridges MUST be event-type agnostic. They may dispatch by string event-type key against a registry, but MUST NOT reference concrete domain event/schema types via `using` or constructor parameters.

## FINDING 2 — Host adapter holds domain type dictionary

**CLASSIFICATION:** platform / host / domain-leakage
**SEVERITY:** S1
**SOURCE:** `src/platform/host/adapters/EventTypeResolver.cs`
**DESCRIPTION:** Static `KnownTypes` dictionary maps event-type strings to concrete `Whycespace.Domain.OperationalSystem.Sandbox.Todo.*` types. Host adapter has direct domain dependency.
**PROPOSED RULE:** Host adapters MUST source event-type→Type mappings from a runtime-side registry (e.g., `EventSchemaRegistry`). No host adapter may hold static dictionaries keyed by domain types.

## FINDING 3 — Kafka projection worker is single-domain coupled

**CLASSIFICATION:** platform / host / kafka / domain-leakage
**SEVERITY:** S1
**SOURCE:** `src/platform/host/adapters/KafkaProjectionConsumerWorker.cs`
**DESCRIPTION:** The worker (a) takes `TodoProjectionHandler` as a constructor dependency, (b) hardcodes the topic `whyce.operational.sandbox.todo.events`, (c) hardcodes the consumer group, (d) defines private `RawTodoCreatedEvent`/`RawTodoUpdatedEvent`/`RawTodoCompletedEvent` records, (e) hardcodes the Postgres projection table `projection_operational_sandbox_todo.todo_read_model`.
**PROPOSED RULE:** Kafka projection consumers MUST be generic over (topic, handler-resolver, schema-registry, projection-table-resolver). Per-domain workers are forbidden in `src/platform/host/`.

## FINDING 4 — Runtime hosts domain policy identifier constants

**CLASSIFICATION:** runtime / policy / domain-leakage
**SEVERITY:** S2
**SOURCE:** `src/runtime/policies/operational-system/sandbox/todo/TodoPolicyDefinition.cs`
**DESCRIPTION:** A `runtime/policies/operational-system/sandbox/todo/` directory tree exists holding 4 string constants (`whyce.operational.sandbox.todo.create`, etc.). These are policy *identifiers*, not policy logic, but their location places domain-shaped paths inside runtime.
**PROPOSED RULE:** Policy identifier constants belong in `src/shared/contracts/`. Runtime MUST NOT contain folders nested by classification/context/domain.

## FINDING 5 — Composition root carries pervasive domain wiring

**CLASSIFICATION:** platform / host / composition / domain-leakage
**SEVERITY:** S1
**SOURCE:** `src/platform/host/Program.cs`
**DESCRIPTION:** `Program.cs` directly references Todo types in ~10 sites: engine-registry bindings (lines 60-63), step registrations (line 77), engine registration (line 81), projection handler/consumer (lines 129-130), Kafka worker (lines 134-137), projection bridge construction (lines 144-147), event schema mappers (lines 163-182), workflow registry (lines 226-232), intent handler (line 250).
**PROPOSED RULE:** `Program.cs` MUST NOT reference any concrete domain type. Per-domain wiring MUST be encapsulated in a bootstrap module under `src/systems/` or a host composition folder, invoked from `Program.cs` by a single non-typed call.

## FINDING 6 — Domain layer cannot host DI bootstrap modules

**CLASSIFICATION:** domain / layer-purity
**SEVERITY:** S1
**SOURCE:** Original patch proposal (rejected variant placed `TodoDomainModule.cs` under `src/domain/`)
**DESCRIPTION:** A bootstrap module accepting `IServiceCollection` cannot live in `src/domain/` because it would force a dependency on `Microsoft.Extensions.DependencyInjection.Abstractions`, violating CLAUDE.md $7 ("Domain = zero dependencies").
**PROPOSED RULE:** Files under `src/domain/` MUST NOT reference `Microsoft.Extensions.DependencyInjection.*` or any DI container abstraction. Domain assembly references MUST be empty beyond the standard library and shared kernel.

---

## FINDING 7 — Guard rule duplication risk for R-DOM-01

**CLASSIFICATION:** governance / guards
**SEVERITY:** S3
**SOURCE:** `claude/guards/runtime.guard.md` rule 11
**DESCRIPTION:** Existing rule 11 ("NO DOMAIN LOGIC IN RUNTIME") already covers business-rule-shaped domain logic. Adding R-DOM-01 as a parallel rule would create overlapping enforcement and drift between formulations.
**PROPOSED RULE:** R-DOM-01 (lexical/path-scoped Todo-style leakage) MUST be added as a sub-clause under rule 11 with explicit allowlist of dispatcher infrastructure. It MUST NOT be a standalone parallel rule.
