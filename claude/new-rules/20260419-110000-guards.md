## Reference migration verdict — SanctionExpirySchedulerWorker

- **Current N-instance behaviour:** N replicas each query `IExpirableSanctionQuery.QueryExpirableAsync` every interval and dispatch `ExpireSanctionCommand` for each expired row. The idempotency middleware dedups at the command layer (deterministic CommandId per `(SanctionId, ExpiresAt)` tuple), but N-1 attempts still traverse validation / policy / authorization / execution-guard / persistence. Concretely wasted DB + Kafka + CPU per tick scales with replica count.
- **Leader-death gap SLA:** sanction expiry windows are measured in hours / days by policy. A 30-120s Postgres TCP-keepalive gap during leader replacement is operationally insignificant — an expired sanction may remain past its expiry for up to the gap duration, well within SLA tolerance.
- **No existing forbid block.** File comments are about determinism + failure isolation + liveness, not multi-instance. The determinism section explicitly notes "two replicas ticking at the same second produce identical keys and the IIdempotencyStore picks exactly one winner per (SanctionId, ExpiresAt) pair" — confirming the correctness of the legacy N-way-with-idempotency path while leaving room for the leader optimization.
- **Verdict: MIGRATE.**

## Forward rule

Future BackgroundServices added to `src/platform/host/adapters/` MUST receive an audit entry in `claude/new-rules/` BEFORE being considered for leader migration. The default choice is the existing safe pattern for the worker's type (Kafka consumer → group; DB-scan → leader-election; admission-sampler → per-instance).
