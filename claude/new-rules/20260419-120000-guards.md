Captured per CLAUDE.md $1c and R2 entry criteria. D4 LOCKED → one circuit breaker per external dependency (Kafka producer, Postgres pool, OPA policy engine, WhyceChain anchor service, Redis). `OpaPolicyEvaluator` already has an inline breaker (phase1.5-S5.2.1 / PC-2) that proves the pattern; R2.A.D.1 extracts it into a reusable contract + implementation, R2.A.D.2 refactors the OPA evaluator to consume it, R2.A.D.3 adds breakers to the remaining dependencies.

## R-CIRCUIT-BREAKER-01 — ICircuitBreaker contract and state machine

- **CLASSIFICATION:** runtime / resilience / circuit-breaker
- **SEVERITY:** S1
- **DESCRIPTION:**
  - `ICircuitBreaker` exposes: `Name` (stable identifier), `State` (side-effect-free getter for health posture queries), `ExecuteAsync<T>(Func<CancellationToken, Task<T>> operation, CancellationToken ct)` (runs operation under breaker protection).
  - State machine (classic three-state):
    - **Closed** — operations execute normally. Each failure increments a consecutive-failure counter.
    - **Closed → Open** — when consecutive failures reach `FailureThreshold`. Opened at `_openedAt = IClock.UtcNow`.
    - **Open** — operations throw `CircuitBreakerOpenException` immediately; operation is NOT executed.
    - **Open → HalfOpen** — when `IClock.UtcNow - _openedAt >= WindowSeconds`. The very next `ExecuteAsync` call becomes the trial call.
    - **HalfOpen → Closed** — trial call returns successfully. Failure counter resets.
    - **HalfOpen → Open** — trial call throws. `_openedAt` reset to `IClock.UtcNow` (window starts over).
  - `State` getter is side-effect-free: it reports Open if `_openedAt` is within window, HalfOpen-pending if window elapsed but no trial call yet, Closed if never-opened / trial-succeeded. The HalfOpen → Closed / Open transition happens inside `ExecuteAsync` on the trial call, NOT in `State` reads — so health-posture queries never accidentally admit a trial.
  - `OperationCanceledException` with the caller's cancellation token is NOT recorded as a failure. It is a propagation signal, not evidence of dependency unavailability.
  - Every other exception type thrown by `operation` IS a failure for breaker accounting. Callers that want typed exception filtering wrap the operation themselves.
- **CHECK:**
  - `ICircuitBreaker` lives at `src/shared/contracts/runtime/` adjacent to `IRetryExecutor`.
  - `CircuitBreakerOpenException` at the same location carries `BreakerName` and `RetryAfterSeconds` fields (parallel to `PolicyEvaluationUnavailableException`).
  - `DeterministicCircuitBreaker` at `src/runtime/resilience/` uses `IClock.UtcNow` for all timing. No `DateTime.UtcNow`, no `Stopwatch`-derived values on the hot path.
  - Unit tests cover every state transition, contention (N concurrent Open-state calls all throw without running operation), HalfOpen trial-success / trial-failure, cancellation propagation, `State` getter side-effect-freeness.

## R-CIRCUIT-BREAKER-DET-01 — Breaker decisions are replay-deterministic

- **CLASSIFICATION:** runtime / resilience / determinism
- **SEVERITY:** S0 (constitutional extension — loss of determinism breaks replay)
- **DESCRIPTION:**
  - `DeterministicCircuitBreaker` state transitions are a pure function of (inputs observed so far, `IClock.UtcNow` reads). No wall-clock reads, no `Random` (breaker decisions don't need randomness — no half-open probe selection in this batch; that is a future enrichment).
  - `_openedAt` timestamp MUST come from `IClock.UtcNow`. No `DateTime.UtcNow`, no `Stopwatch.GetTimestamp`.
  - The window-elapsed check MUST use `_clock.UtcNow - _openedAt` — no local-computed TimeSpan or cached "opened until T" value.
  - An architecture guard scans `src/runtime/resilience/DeterministicCircuitBreaker.cs` for forbidden tokens.
- **CHECK:**
  - Architecture test `CircuitBreaker_Source_Contains_No_Wall_Clock_Reads` passes.
  - Replay determinism test: two executor instances with the same `FakeClock` + same operation outcomes produce identical state transitions.

## R-CIRCUIT-BREAKER-PER-DEPENDENCY-01 — One breaker per external dependency

- **CLASSIFICATION:** runtime / resilience / composition
- **SEVERITY:** S2 (compositional discipline)
- **DESCRIPTION:**
  - Per D4 LOCKED (2026-04-19): one `ICircuitBreaker` instance per external dependency. Known dependencies: OPA policy engine, Postgres event-store pool, Kafka producer, WhyceChain anchor service, Redis. Each gets its own breaker with its own `FailureThreshold` / `WindowSeconds` sized for the dependency's failure profile.
  - Per-operation breakers (one per outbound command type) are explicitly REJECTED (D4 option b). Rationale: fewer breakers to tune; matches §18 health posture model; a Kafka outage is one event, not N events by command type.
  - Breaker INSTANCES are registered in host composition. The breaker NAME is a stable identifier used in metrics tags and health reports. Callers receive their breaker via DI (`ICircuitBreaker` keyed by name, or typed aliases for the well-known dependencies).
- **CHECK:**
  - R2.A.D.3 composition registrations create named breakers for each dependency.
  - R4 health-posture integration surfaces `State` via `IRuntimeStateAggregator`.
