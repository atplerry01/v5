# R2.A.C.1 — Distributed Lease Primitive (2026-04-19)

Captured per CLAUDE.md $1c and R2 entry criteria. D3 LOCKED → Postgres advisory locks for distributed leases (not Redis SETNX, not K8s leases). R2.A.C.1 ships the contract + Postgres implementation. R2.A.C.2 wires it into outbox/projection leadership. R3 workflow sequencing (per-workflow lease) consumes the same primitive.

## R-LEASE-CONTRACT-01 — IDistributedLeaseProvider semantics

- **CLASSIFICATION:** runtime / concurrency / lease
- **SEVERITY:** S1
- **DESCRIPTION:**
  - `IDistributedLeaseProvider` exposes a single acquire operation: `TryAcquireAsync(leaseKey, holder, cancellationToken)` → `ILease?`. Returns the handle on success or `null` if another holder currently holds the key. MUST NOT throw on contention — null is the canonical "another holder already has it" signal.
  - `ILease` is the returned handle: `Key`, `Holder`, `AcquiredAt` (from `IClock`), plus `IAsyncDisposable`. Disposing the handle releases the lease. A handle MUST NOT be shared across threads without external synchronization.
  - Contract is DIFFERENT from `IExecutionLockProvider` (Redis-backed, short-lived per-command execution lock). `IDistributedLeaseProvider` is for longer-lived coordination (leadership, per-workflow sequencing) where holders may run for minutes / hours.
  - `TryAcquireAsync` is NOT retryable on failure — callers that want retry wrap the call in `IRetryExecutor` themselves. Keeping the primitive non-retrying preserves determinism at the lease boundary.
- **CHECK:**
  - `IDistributedLeaseProvider` lives at `src/shared/contracts/infrastructure/persistence/` adjacent to `IIdempotencyStore`.
  - `ILease` is `IAsyncDisposable`.
  - Contract unit tests (in-memory fake) pin acquire / release / contention / reacquire-after-release semantics.

## R-LEASE-POSTGRES-01 — Postgres advisory-lock implementation

- **CLASSIFICATION:** runtime / concurrency / lease / postgres
- **SEVERITY:** S1
- **DESCRIPTION:**
  - `PostgresAdvisoryLeaseProvider` implements `IDistributedLeaseProvider` via Postgres session-level advisory locks (`pg_try_advisory_lock(bigint)`).
  - String lease keys are hashed to `bigint` via SHA256-prefix — deterministic across processes and versions. Different keys MAY collide to the same `bigint` (birthday paradox on 64 bits ≈ 1 in 2^32 for two random keys), which means two DIFFERENT lease keys could contend as if they were the same. This is documented and acceptable at current scale. A future batch with `(int, int)` pair advisory locks can namespace keys if needed.
  - Session lifetime = lease lifetime. The provider opens a dedicated `NpgsqlConnection` per acquired lease; the connection is held until `ILease.DisposeAsync` runs. This is the canonical advisory-lock pattern — no application-level TTL is needed.
  - Operational implication: each active lease = 1 open DB connection. The provider uses the declared event-store pool; callers MUST NOT acquire more leases than the pool's max connection count minus headroom for command traffic. This constraint is R2.A.C.2 configuration work.
  - `TryAcquireAsync` on a busy key releases the connection (returns it to the pool) before returning null. Only a successful acquire retains the connection.
- **CHECK:**
  - `PostgresAdvisoryLeaseProvider` uses `EventStoreDataSource` pool.
  - SQL: `SELECT pg_try_advisory_lock(@key_hash)` on acquire; `SELECT pg_advisory_unlock(@key_hash)` on dispose.
  - Release path is idempotent — double-dispose is a no-op.

## R-LEASE-CRASH-SAFE-01 — Crash-safe recovery via session termination

- **CLASSIFICATION:** runtime / concurrency / lease / safety
- **SEVERITY:** S0 (constitutional — loss of this invariant produces split-brain)
- **DESCRIPTION:**
  - Session-level Postgres advisory locks are released automatically when the connection closes. This means: process crash, network partition, host termination — any event that closes the Postgres session → Postgres releases the lock within the TCP keepalive window (default 30-120 seconds, tunable via `tcp_keepalives_idle`).
  - **No client clock involved.** Unlike Redis SETNX + TTL, there is no application-level "lease expires at T+N seconds" calculation. The lock is held exactly as long as the session is alive, period. This rules out clock skew between lease holder and lease checker — a class of bugs Redis-based leases are prone to.
  - **Hung holder caveat.** A process that is NOT crashed but has a stuck worker thread (say, deadlock inside a projection handler) will continue to hold its Postgres session and therefore its lease. This is NOT detected by R-LEASE-CRASH-SAFE-01. Mitigation: the holder must itself implement liveness (e.g. heartbeat writes to a monitoring table) and release the lease if liveness is violated. A future batch may add heartbeat-validated leases as an enrichment — OUT OF SCOPE for R2.A.C.1.
- **CHECK:**
  - `PostgresAdvisoryLeaseProvider.DisposeAsync` on the provider closes all leased connections deterministically on host shutdown.
  - Integration test (R5 certification): kill process mid-lease → another instance acquires within keepalive window.
  - NO application-level TTL / expiry calculation in the adapter. Source code search `TimeSpan.*TTL|expires_at|lease_expiry` inside `PostgresAdvisoryLeaseProvider.cs` returns zero hits.

## Out of scope for R2.A.C.1

- **Heartbeat-validated leases** — detects hung (non-crashed) holders. Future batch.
- **Lease metrics** — active-count gauge, acquire-failure counter. Landed in R2.A.C.2 alongside leader workers.
- **Auto-renewal** — session-based leases don't need renewal. If TTL-based leases are added later, they do.
- **Multi-tenant key namespacing** — today lease keys are global across the Postgres instance. Callers prefix keys (e.g. `{tenant}:{name}`) to avoid collisions.
