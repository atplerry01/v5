# R2.A.D.3b — Kafka Producer Breaker (2026-04-19)

Captured per CLAUDE.md $1c. R2.A.D.3a closed the chain adapter refactor. R2.A.D.3b adds a brand-new breaker around Kafka `ProduceAsync` calls — there was no prior inline breaker, so this is greenfield integration rather than refactor. Shared across three call sites, which require DIFFERENT open-state behaviors because the downstream failure semantics differ.

## R-KAFKA-BREAKER-01 — Shared "kafka-producer" breaker protects all Kafka publish paths

- **CLASSIFICATION:** runtime / resilience / messaging
- **SEVERITY:** S1
- **DESCRIPTION:**
  - One `ICircuitBreaker` instance named `"kafka-producer"` is registered in host composition (`KafkaInfrastructureModule.AddMessaging`) as a plain `ICircuitBreaker` so the R2.A.D.4 registry picks it up automatically.
  - Three call sites wrap `IProducer<string,string>.ProduceAsync` through this shared breaker:
    1. `KafkaOutboxPublisher.PublishBatchAsync` (outbox → primary topic)
    2. `KafkaOutboxPublisher.TryPublishToDeadletterAsync` (outbox exhausted → `.deadletter` topic)
    3. `GenericKafkaProjectionConsumerWorker.PublishToDeadletterAsync` (consumer poison → `.deadletter` topic)
  - **Shared, not per-site.** A Kafka broker outage impacts all three call sites simultaneously; keeping one breaker means consecutive-failure accounting converges faster than distributing it across three separate breakers.
  - `ProduceException<string,string>` thrown inside the wrapped lambda is counted as a breaker failure. Every other exception is also counted per R-CIRCUIT-BREAKER-01. `OperationCanceledException` with the caller's token is NOT counted (propagation signal per R-CIRCUIT-BREAKER-01).
- **CHECK:**
  - `sp.GetRequiredService<ICircuitBreakerRegistry>().Get("kafka-producer")` returns the registered breaker.
  - Architecture test / grep: `_producer.ProduceAsync` and `producer.ProduceAsync` call sites inside `KafkaOutboxPublisher` and `GenericKafkaProjectionConsumerWorker` MUST appear inside `_kafkaBreaker.ExecuteAsync(...)` lambdas (not as bare calls).

## R-KAFKA-BREAKER-OPEN-BEHAVIOR-01 — Open-state behavior is per call site

- **CLASSIFICATION:** runtime / resilience / messaging / boundary
- **SEVERITY:** S1
- **DESCRIPTION:**
  - When the shared Kafka breaker is Open, `ExecuteAsync` throws `CircuitBreakerOpenException`. The three call sites handle this differently because their downstream semantics differ:
    1. **Outbox publish (primary):** catch + log + `continue` the batch loop. Do NOT call `RecordFailureAsync` — a broker outage is not a per-row failure, so retry_count MUST NOT advance. The row remains `status='pending'` / `status='failed'` (prior state) and will retry on the next poll cycle when the breaker transitions back toward Closed. This preserves the per-row retry budget across outage windows.
    2. **Outbox DLQ publish:** catch + log + skip the DLQ attempt. The outbox DB row still moved to `status='deadletter'` via `RecordFailureAsync` on the primary-publish failure; the Kafka DLQ visibility is degraded but not data-losing (R2.A.3b DLQ store still records the entry). Do not re-throw.
    3. **Consumer DLQ publish:** RE-THROW the exception. Downstream, the consume loop's existing K-DLQ-001 contract ("DLQ publish failure must propagate so the caller does NOT commit the source offset") means the Kafka consumer will re-deliver the poison message on the next poll — consistent with the pre-R2.A.D.3b behavior for any DLQ publish failure.
- **WHY DIFFERENT?**
  - Outbox rows are ours to retry at our leisure (Postgres-backed queue); a missed broker-blip doesn't lose data because the row stays in the outbox.
  - Consumer offsets are Kafka-managed; not committing the offset keeps the message redeliverable. Committing after a skipped DLQ would lose the message.
- **CHECK:**
  - Outbox `continue` path under `CircuitBreakerOpenException` MUST NOT call `RecordFailureAsync` and MUST NOT call `TryPublishToDeadletterAsync`.
  - Consumer DLQ path under `CircuitBreakerOpenException` MUST re-throw (caller's existing outer catch already refuses to commit the offset).

## R-CIRCUIT-BREAKER-VOID-EXT-01 — ICircuitBreaker void-returning helper

- **CLASSIFICATION:** runtime / resilience / ergonomics
- **SEVERITY:** S3
- **DESCRIPTION:**
  - Adds a small extension method `CircuitBreakerExtensions.ExecuteAsync(Func<CancellationToken, Task>, CancellationToken)` on `ICircuitBreaker` so side-effect-only operations (like `ProduceAsync`) don't need a dummy return value. Forwards to the existing `Task<T>` core via `return 0`.
  - Extension-only: contract `ICircuitBreaker` is unchanged; no new member on the interface. Implementations are not affected.
- **CHECK:**
  - Extension lives at `src/shared/contracts/runtime/CircuitBreakerExtensions.cs` adjacent to the contract.
  - Consumers: `KafkaOutboxPublisher` + `GenericKafkaProjectionConsumerWorker` use the void overload when wrapping `ProduceAsync`.
