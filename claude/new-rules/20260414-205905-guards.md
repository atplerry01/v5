# Guard Consolidation Canonicalization Rule

**CLASSIFICATION:** guards
**SOURCE:** Governance-critical refactor executed 2026-04-14 under explicit authorization (overrides CLAUDE.md §5 Anti-Drift for the guard tree only).
**SEVERITY:** S1 (architectural)

## DESCRIPTION

Guard files under `claude/guards/` historically proliferated into ~32 fragmented files, many containing duplicated rules (WBSM GE-01..GE-05 appeared in 8+ files; determinism rules appeared across runtime, clean-code, e2e, phase1.5; enforcement tables repeated across platform/systems/kafka). The fragmentation diluted the authoritative surface and made post-execution guard loading per $1a ambiguous.

This refactor consolidated all 32 source guards into exactly four canonical files aligned to Whycespace layer doctrine:

- `claude/guards/constitutional.guard.md` — WHYCEPOLICY authority, determinism, deterministic IDs, hash determinism, replay determinism.
- `claude/guards/runtime.guard.md` — execution pipeline/ordering, engine, projection, prompt-container, dependency graph, contracts boundary, code quality enforcement (clean-code/no-dead-code/stub-detection), tests & E2E validation.
- `claude/guards/domain.guard.md` — domain purity, structure, classification-suffix, DTO naming, behavioral, structural, plus inlined domain-aligned (economic/governance/identity/observability/workflow).
- `claude/guards/infrastructure.guard.md` — platform, systems, kafka, config safety, composition loader, program composition.

## PROPOSED_RULE

**GUARD-CANONICAL-STRUCTURE-01**

1. All guard files MUST live directly under `claude/guards/` as one of exactly four files: `constitutional.guard.md`, `runtime.guard.md`, `domain.guard.md`, `infrastructure.guard.md`.
2. No guard may exist outside these four categories. Subdirectories under `claude/guards/` are FORBIDDEN (domain-aligned subtree is eliminated; domain-aligned content lives as subsections inside `domain.guard.md`).
3. Placement doctrine is non-negotiable:
   - **Constitutional** = WHYCEPOLICY authority, determinism primitives, identifier/hash/replay primitives.
   - **Runtime** = enforcement, execution rules, layer boundaries, dependency rules, code quality enforcement, test validation.
   - **Domain** = business truth only (aggregates, invariants, structure, naming, behavior). Never enforcement.
   - **Infrastructure** = platform edge, host, systems integration, eventing, configuration, composition.
4. STRICT CORRECTIONS (precedent-setting):
   - `dependency-graph` belongs to Runtime (NOT Domain).
   - `contracts-boundary` belongs to Runtime (NOT Domain).
   - `policy` and `policy-binding` belong to Constitutional (NOT Runtime).
   - Code quality (`clean-code`, `no-dead-code`, `stub-detection`) and test validation (`tests`, `e2e-validation`) belong to Runtime (not a separate quality/testing tier).
5. Rule duplication across canonical files is FORBIDDEN. WBSM GE-01..GE-05 and enforcement-severity tables exist exactly once per file as shared appendices (not replicated across sections).
6. When a new guard is introduced, it MUST be added as a section inside the correct canonical file. Creating a fifth top-level guard file is S1 architectural drift and MUST be captured under `/claude/new-rules/` per §1c.
7. Pre-execution guard loading per $1a loads only these four files. Any stale fragmented guard file discovered in the tree is evidence of drift and MUST be deleted.

## ENFORCEMENT

- Post-execution audit sweep MUST fail if `claude/guards/` contains anything other than the four canonical files plus (optionally) a README.
- Any PR that creates a subdirectory under `claude/guards/` is S1 and auto-blocked.
- Any PR that restates WBSM GE-01..GE-05 in more than one location within a single canonical file is S2 and MUST be deduplicated inline before merge.
