# R2.A.3a — Tiered Topic Naming + DLQ Store Contract (2026-04-19)

Captured per CLAUDE.md $1c and per R2 entry criteria in [`runtime-upgrade-plan.md`](../project-topics/v2b/runtime-upgrade-plan.md) §3.

D2 LOCKED → tiered naming `{topic}.events` → `{topic}.retry` → `{topic}.deadletter`. Codebase already uses `.deadletter` per phase1.6-S1.6 / DLQ-RESOLVER-01. R2.A.3a extends `TopicNameResolver` with a canonical retry-tier resolver and adds the persistence contract for DLQ entries.

---

## R-TOPIC-TIER-01 — Canonical three-tier topic naming

- **CLASSIFICATION:** runtime / event-fabric / topic-naming
- **SEVERITY:** S1
- **DESCRIPTION:**
  - The runtime fabric emits events on exactly one of four topic tiers per `(classification, context, domain)` coordinate: `.commands` (inbound) / `.events` (primary outbound) / `.retry` (first-failure retry tier) / `.deadletter` (terminal).
  - Tier transitions follow the locked order: a message failing on `.events` routes to `.retry`; a message failing on `.retry` after `MaxAttempts` routes to `.deadletter`; `.deadletter` is terminal and never feeds back into `.events` or `.retry` except via an explicit operator re-drive (R4 admin surface).
  - Poison messages (deserialization failure, schema validation failure) route directly to `.deadletter` — no retry tier, because the failure mode is deterministic and retry will not help.
  - `TopicNameResolver` is the single source of truth for these names. Any caller (outbox publisher, consumer worker, projection consumer) that constructs these strings inline is S1 drift.
- **CHECK:**
  - Architecture test scans `src/platform/host/adapters/**` and `src/runtime/event-fabric/**` for regex `"\\.(retry|deadletter)"` outside `TopicNameResolver.cs`. Zero hits expected.
  - `TopicNameResolverTests.Resolve_Retry_From_Events_Topic` + `Resolve_DeadLetter_From_Retry_Topic` pin the canonical transitions.

## R-DLQ-STORE-01 — Dead-letter persistence contract

- **CLASSIFICATION:** runtime / event-fabric / dead-letter
- **SEVERITY:** S1
- **DESCRIPTION:**
  - Every message that lands on a `.deadletter` topic MUST be mirrored to durable storage via `IDeadLetterStore.RecordAsync(DeadLetterEntry, ct)`. A Kafka-only deadletter path is insufficient because:
    1. Operator inspection needs queryable storage (R4.B "DLQ inspection controls" ABSENT today).
    2. Re-drive controls (R4.B) operate against a persisted entry, not a Kafka offset.
    3. Retention-sweep / aging (R2.A.4) requires durable lineage.
  - `DeadLetterEntry` captures the minimum audit shape: `EventId` (idempotency key), `SourceTopic`, `EventType`, `CorrelationId`, `EnqueuedAt` (from `IClock`), `FailureCategory`, `LastError`, `AttemptCount`, `Payload` (raw bytes for replay).
  - `RecordAsync` MUST be idempotent on `EventId` — multi-instance consumers may both try to record the same poison message. Duplicate records on the same EventId collapse.
  - `IDeadLetterStore` lives in `src/shared/contracts/infrastructure/messaging/` (NOT `persistence/`) because the contract is a messaging-fabric concept, not a generic persistence seam. Implementation (`PostgresDeadLetterStore`) lands in R2.A.3b.
- **CHECK:**
  - `IDeadLetterStore` exists as a shared contract and compiles.
  - `DeadLetterEntry` carries all required fields with `IClock`-sourced timestamp.
  - R2.A.3b will add: idempotency test (two RecordAsync calls on same EventId → single row); schema migration; R4 inspection API.
- **R2.A.3a DELIVERABLE:** contract only — no implementation.
