# R1 Wiring Discipline — 2026-04-19

Captured per CLAUDE.md $1c during execution of [runtime-upgrade-plan.md](../project-topics/v2b/runtime-upgrade-plan.md) Phase R1 Batch 3.5 (activation).

Batch 3.5 wired 19 previously-uncategorized `CommandResult.Failure(...)` call sites across `src/runtime/middleware/**` and `src/runtime/control-plane/**` into the canonical `RuntimeFailureCategory` / `ValidationFailureCategory` / `PersistenceFailureCategory` taxonomy landed in Batch 1. These rules lock the wiring against drift.

---

## R-FAIL-CAT-01 — Every runtime failure must carry a canonical category

- **CLASSIFICATION:** runtime / failure-classification
- **SEVERITY:** S1
- **DESCRIPTION:**
  - Every `CommandResult.Failure(...)` call inside `src/runtime/**` MUST use a categorized overload: `Failure(error, RuntimeFailureCategory)`, `ValidationFailure(error, ValidationFailureCategory)`, or `PersistenceFailure(error, PersistenceFailureCategory)`.
  - The 1-arg `Failure(error)` overload is retained ONLY for backwards compatibility with tests and boundary code outside runtime. Any call to it inside `src/runtime/**` is S1 drift.
  - Reason: R2 retry logic, operator UI, and audit classification rely on the category field. Uncategorized failures default to `FailureCategory = null`, which retry logic treats as "unknown → do not retry" — masking recoverable failures.
- **CHECK:** `grep -Pn "CommandResult\.Failure\([^,)]+\)" src/runtime/` MUST return zero hits.
- **CURRENT STATUS:** All 19 known sites closed in Batch 3.5. This rule pins the invariant forward.

---

## R-FAIL-CAT-02 — Category must match semantic intent

- **CLASSIFICATION:** runtime / failure-classification
- **SEVERITY:** S2
- **DESCRIPTION:**
  - Middleware failures must pick the category that matches WHERE in the pipeline they rejected:
    - `ValidationMiddleware` → `ValidationFailureCategory.{InputSchema,Semantic,CommandPrecondition,...}` via `CommandResult.ValidationFailure`.
    - `AuthorizationGuardMiddleware` → `RuntimeFailureCategory.AuthorizationDenied`.
    - `PolicyMiddleware` deny paths (OPA + WhycePolicy) → `RuntimeFailureCategory.PolicyDenied`.
    - `PolicyMiddleware` identity-resolution failure → `RuntimeFailureCategory.AuthorizationDenied`.
    - `ExecutionGuardMiddleware` lock/violation/restriction/escalation → `RuntimeFailureCategory.RuntimeGuardRejection`.
    - `ExecutionGuardMiddleware` lock-state-unavailable (fail-closed) → `RuntimeFailureCategory.DependencyUnavailable`.
    - `IdempotencyMiddleware` duplicate → `RuntimeFailureCategory.ConcurrencyConflict` + `IsDuplicate=true`.
    - `RuntimeControlPlane` cancellation → `RuntimeFailureCategory.Cancellation`.
    - `RuntimeControlPlane` lock-unavailable → `RuntimeFailureCategory.DependencyUnavailable`.
    - `RuntimeControlPlane` WHYCEPOLICY hard-stop branches → `RuntimeFailureCategory.PolicyDenied`.
  - Mis-classification produces wrong retry behavior in R2 (e.g. retrying an `AuthorizationDenied` wastes cycles; not retrying a `DependencyUnavailable` blocks recovery).

## R-EXC-MAP-01 — Unhandled exceptions must route through `RuntimeExceptionMapper`

- **CLASSIFICATION:** runtime / exception-mapping
- **SEVERITY:** S2 (target S1 post-R1)
- **DESCRIPTION:**
  - Any top-level `try/catch` in `src/runtime/control-plane/**` that converts an exception to a `CommandResult` MUST route through `RuntimeExceptionMapper.ToCommandResult(ex, correlationId)`. Ad-hoc `catch { return CommandResult.Failure(ex.Message); }` is forbidden — leaks internal type names and stack traces, and produces uncategorized failures.
  - Current state (R1 end): `RuntimeExceptionMapper` exists and maps `DbException` by PostgreSQL SQLSTATE. `RuntimeControlPlane.ExecuteAsync` does not yet wrap the dispatch call in a try/catch; that wiring is a follow-up in R2 where retry logic needs the categorized exceptions.
  - Severity S2 today (advisory) → S1 once R2 lands the try/catch.
- **CHECK:** `grep -Pn "catch.*\{\s*return CommandResult\.Failure" src/runtime/` SHOULD return zero hits.

---

## R-IDEM-EVIDENCE-01 — Idempotency middleware must stamp context evidence

- **CLASSIFICATION:** runtime / idempotency
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `IdempotencyMiddleware` MUST stamp `CommandContext.IdempotencyKey` and `CommandContext.IdempotencyOutcome` on every invocation (Hit on duplicate, Miss on fresh claim).
  - Duplicate path MUST return `CommandResult.Failure(msg, ConcurrencyConflict) with { IsDuplicate = true, IdempotencyKey = key }` so downstream consumers (audit, metrics, operator tooling) can distinguish replay hits from fresh failures without parsing error strings.
  - Reason: without the evidence, idempotency hit-rate is unmeasurable and replay audits cannot confirm duplicate detection actually ran.

---

## R-POLICY-OVERLAY-01 — Policy middleware must call 6-arg `PolicyInputBuilder.Enrich`

- **CLASSIFICATION:** runtime / policy-input
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `PolicyMiddleware` MUST invoke `PolicyInputBuilder.Enrich(baseContext, command, resourceState, now, environment, jurisdiction)`. The 4-arg overload is retained for backwards compatibility but is deprecated for new callers.
  - Environment / jurisdiction may be passed as `null` until overlay sources are wired — the builder emits omitted OPA input fields in that case, preserving pre-R1 rego behavior.
  - Reason: locking the 6-arg path forward ensures that when overlay sources land (host config for environment, tenant→jurisdiction resolver), the swap is one-line, not a breaking API migration.

---

## R-CTX-SESSION-01 — Session / token / idempotency fields are write-once with replay reset

- **CLASSIFICATION:** runtime / context-discipline
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `CommandContext.SessionId`, `TokenFingerprint`, `StepUpAuthenticatedAt`, `IdempotencyKey`, `IdempotencyOutcome` are write-once. Second-write throws, matching the existing R-CTX-01 pattern for `PolicyDecisionHash` and peer fields.
  - `CommandContext.ResetForReplay()` MUST clear all five fields alongside the pre-existing write-once set. Missing a field in the reset breaks replay determinism — the S1.4 seam only works if EVERY write-once field is in the reset path.
- **CHECK:** Visual inspection of `CommandContext.ResetForReplay()` shows 13 resets (8 pre-R1 + 5 R1). Architecture test `CommandContext_reset_for_replay_clears_all_write_once_fields` (R1 Batch 5) locks this forward.

---

## R-STATE-BOUNDARY-01 — Runtime middleware must not hold static mutable state

- **CLASSIFICATION:** runtime / state-discipline
- **SEVERITY:** S1
- **DESCRIPTION:**
  - Types under `src/runtime/middleware/**` and `src/runtime/control-plane/**` MUST NOT declare static mutable fields. `static readonly` singletons (e.g. `Meter`, `Counter<T>`) are permitted; `static const` is permitted; any other `static` field is S1 drift.
  - Reason: middleware instances are shared across concurrent requests. Static mutable state is a classic source of cross-request contamination and breaks both replay determinism and multi-instance deployment (no shared state between replicas either).
- **CHECK:** Architecture test `RuntimeMiddleware_holds_no_static_mutable_state` (R1 Batch 4) scans `src/runtime/middleware/**` and `src/runtime/control-plane/**` for `(public|private|internal|protected)?\s+static\s+(?!readonly|const)` on non-method declarations and asserts zero matches.
