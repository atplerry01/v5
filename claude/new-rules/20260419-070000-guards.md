# R2.A.3c — Consumer DLQ Mirror + Poison Classification (2026-04-19)

Captured per CLAUDE.md $1c and R2 entry criteria in [`runtime-upgrade-plan.md`](../project-topics/v2b/runtime-upgrade-plan.md) §3.

R2.A.3a introduced the `IDeadLetterStore` contract. R2.A.3b wired the outbox publisher to mirror deadletter publishes into the store. This rule (R2.A.3c) closes the equivalent gap on the consumer side: `GenericKafkaProjectionConsumerWorker` currently routes poisoned messages to `.deadletter` via Kafka only — operator inspection / re-drive / retention queries cannot see those entries.

## R-DLQ-STORE-CONSUMER-MIRROR-01 — Consumer DLQ publishes mirror to store

- **CLASSIFICATION:** runtime / event-fabric / dead-letter
- **SEVERITY:** S1
- **DESCRIPTION:**
  - Every `GenericKafkaProjectionConsumerWorker.PublishToDeadletterAsync` call MUST mirror the message to `IDeadLetterStore.RecordAsync` after the Kafka producer acknowledges the `.deadletter` publish. Best-effort: store-write failure is caught and logged; the Kafka message remains the authoritative record.
  - The mirror call MUST be idempotent on `EventId` (guaranteed by `PostgresDeadLetterStore.ON CONFLICT`). Multi-consumer replicas routing the same poison message collapse to a single row.
  - `IDeadLetterStore` is an optional constructor dependency — composition sites without it registered fall back to Kafka-only behaviour (pre-R2.A.3c) with no regression.
- **CHECK:**
  - `GenericKafkaProjectionConsumerWorker` constructor exposes `IDeadLetterStore? deadLetterStore` trailing parameter.
  - `PublishToDeadletterAsync` records to the store on the success branch (after Kafka ack, before re-throwing on Kafka failure).
  - Integration test (R5 certification): poison message → single DLQ row + single Kafka deadletter message.

## R-DLQ-CATEGORY-POISON-01 — Poison-message rejections carry canonical category

- **CLASSIFICATION:** runtime / event-fabric / dead-letter
- **SEVERITY:** S1
- **DESCRIPTION:**
  - Every consumer-side DLQ routing site MUST pass a `RuntimeFailureCategory` to `PublishToDeadletterAsync`. Header-malformation and deserialization-failure sites use `RuntimeFailureCategory.PoisonMessage` (canonical poison classification per `RetryEligibility.IsRetryable(PoisonMessage) == false`).
  - The category is persisted on `DeadLetterEntry.FailureCategory` so operator inspection / re-drive logic can filter by root cause without parsing reason strings.
  - Future non-poison DLQ paths (e.g. handler-reported policy-denied) use their specific category. Free-text reason strings remain permitted as human-readable context — they do NOT replace the typed category.
- **CHECK:**
  - All 6 in-worker callsites to `PublishToDeadletterAsync` pass `RuntimeFailureCategory.PoisonMessage`.
  - `DeadLetterEntry.FailureCategory` populated on every stored row originating from the consumer worker (null only if the category lookup itself threw — never on success).
