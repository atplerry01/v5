# R2.A.C.2 — Single-Leader Helper (2026-04-19)

Captured per CLAUDE.md $1c and R2 entry criteria. R2.A.C.1 shipped the distributed lease primitive; R2.A.C.2 ships a composable leader helper and migrates ONE reference worker (`SystemLockExpirySchedulerWorker`).

## R-LEADER-ELECTION-01 — `LeaderElection.RunAsLeaderAsync` composable helper

- **CLASSIFICATION:** runtime / concurrency / leader
- **SEVERITY:** S1
- **DESCRIPTION:**
  - `Whycespace.Runtime.Resilience.LeaderElection.RunAsLeaderAsync(leaseProvider, leaseKey, holder, leaderWork, retryInterval, logger, cancellationToken)` is the canonical helper for BackgroundService-style workers that must run at most one instance at a time. It:
    1. Tries to acquire the lease via `IDistributedLeaseProvider.TryAcquireAsync`.
    2. On contention (null handle), waits `retryInterval` and retries. No exception.
    3. On acquisition, invokes `leaderWork(cancellationToken)` — the worker's existing loop body — while holding the lease.
    4. On `leaderWork` return / exception / cancellation: lease released via `await using` dispose.
    5. On exception (not cancellation), logs + waits `retryInterval` + retries.
    6. On cancellation: returns cleanly.
  - The helper MUST NOT introduce any TTL / expiry arithmetic — lease lifetime is owned by the session-level advisory lock (R-LEASE-CRASH-SAFE-01). The helper's only timing concern is the `retryInterval` between acquire attempts when another instance holds the lease.
  - The lease key identifies a WORKER ROLE, not an instance. Multiple instances of the same worker pass the same `leaseKey` and different `holder` strings. Holder is observability-only (surfaced on `ILease.Holder`).
  - Leader-death-to-replacement gap is bounded by:
    - Postgres TCP keepalive (OS + DB config, typically 30-120s) → releases the advisory lock,
    - PLUS `retryInterval` → next instance probes.
    Workers that cannot tolerate this gap MUST NOT use leader election — use row-level / idempotency-based multi-instance safety instead (see R-MULTI-INSTANCE-AUDIT-01).
- **CHECK:**
  - Helper lives at `src/runtime/resilience/LeaderElection.cs` adjacent to `DeterministicRetryExecutor`.
  - Unit tests against in-memory `IDistributedLeaseProvider` fake pin: solo-leader runs `leaderWork` immediately / contention skips to retry / cancellation exits cleanly / exception in `leaderWork` releases lease and retries.

## R-MULTI-INSTANCE-AUDIT-01 — Per-worker safety audit before leader migration

- **CLASSIFICATION:** runtime / concurrency / safety
- **SEVERITY:** S1
- **DESCRIPTION:**
  - Before migrating ANY BackgroundService worker from row-lock / idempotency-based multi-instance safety to `LeaderElection.RunAsLeaderAsync`, an audit MUST confirm:
    1. The worker's current N-instance behaviour produces duplicate work (N SELECTs, N command dispatches), NOT merely per-instance state that happens to be equivalent.
    2. The leader-death-to-replacement gap (Postgres keepalive + retry interval) is operationally acceptable for the worker's SLA.
    3. No existing comment block in the worker explicitly forbids a leader layer (the `KafkaOutboxPublisher` MI-2 block is the canonical example of such a forbid — do not add leader election to workers whose current design explicitly documents row-lock sufficiency).
  - This rule is anti-drift: adding leader election to the outbox publisher or projection consumers would violate existing MI-2 / Kafka-consumer-group invariants.
  - R2.A.C.2 migrates ONE reference worker — `SystemLockExpirySchedulerWorker`. Other scheduler workers (SanctionExpirySchedulerWorker, enforcement workers, reconciliation workers) migrate in follow-up batches, one at a time, each with its own audit entry.
- **REFERENCE MIGRATION AUDIT — `SystemLockExpirySchedulerWorker`:**
  - Current N-instance behaviour: N instances each query `IExpirableLockQuery.QueryExpirableAsync` every interval and dispatch `ExpireSystemLockCommand` for each expired row. The runtime idempotency middleware dedups at the command layer, but N-1 attempts still traverse the full pipeline (validation / policy / authorization / execution guard). Concretely wasted DB + Kafka + CPU per tick scales with instance count.
  - Leader-death gap SLA: expiry windows are minutes (system locks use TTLs measured in hours by policy). A 30-120s gap during leader replacement is operationally acceptable — an expired lock may remain past its expiry for up to the gap duration, which is well within lock-TTL tolerance.
  - No existing forbid block. Worker comment is about suspend-safety (WHERE clause filters `status='Locked'`), not multi-instance.
  - **Verdict: migrate.**
