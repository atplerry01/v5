## §CD-07 four-criteria evaluation

### C1 — No SegmentAggregate invariants exist beyond storage metadata
Verdict: **PASS (thin invariants).**
Invariants are limited to state-machine transitions (cannot-publish-twice,
cannot-retire-after-archive, etc.) and a single `SourceRef != empty`. No
durability, no cryptographic assertion, no cross-aggregate rule. State-machine-
only invariants are equally expressible as infrastructure-level read-model
constraints if the analytics use-case ever arrives.

### C2 — No projection/read model depends on SegmentAggregate events
Evidence: `grep Segment` in `src/projections/` returns zero matches.
Historical SQL projections exist under `infrastructure/data/postgres/projections/content/...`
but are DELETED in the current working tree (see `git status` at CS.0), and
none targeted segment events.
Verdict: **PASS.**

### C3 — No downstream subscribes to Segment* topics
Evidence:
- `grep Streaming.DeliveryArtifact.Segment` in *.cs returns matches only
  INSIDE the segment/ folder itself.
- `src/engines/` has no content-system routing.
- `src/systems/downstream/` has no content-system routing.
- `infrastructure/event-fabric/kafka/topics/content/...` is ABSENT (confirmed
  CS.0 §5 mirror-layer finding).
Verdict: **PASS.**

### C4 — Analytics requires infrastructure store present OR explicitly deferred
Evidence: no existing segment analytics. Current need: none.
Future need: if segment-level CDN telemetry becomes relevant, it is a classic
infrastructure/read-store concern (high-volume, short-TTL, append-only
time-series data). Does not belong in the domain layer even if it becomes
important.
Deferred with ticket: this capture IS the deferral marker; when the need
arises, an infrastructure phase will introduce
`infrastructure/observability/segment-telemetry/` or equivalent.
Verdict: **PASS.**

## Overall verdict

All four criteria pass. **Segment aggregate retires at CS.5.**
