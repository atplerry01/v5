## OPS-VAL-005 — No engine registered for `CreateRevenueContractCommand`

- **CLASSIFICATION:** runtime / engine-registration
- **SOURCE:** Scenario A re-run, context: revenue — `error.message = "No engine registered for CreateRevenueContractCommand."`
- **SEVERITY:** S1 (functional path broken; only the revenue.contract write API is affected)
- **DESCRIPTION:**
  - After Fix 4 (OPA permit for `ops-validator`), the revenue endpoint previously blocked at OPA now reaches the runtime dispatcher and fails with `No engine registered for <CommandName>`.
  - The existence of the command DTO in `Whycespace.Shared.Contracts.Economic.Revenue.Contract` and the controller POST in `ContractController.CreateContract` both indicate the write path is *intended* to be live, but the T2E handler registration is absent or conditionally excluded.
- **PROPOSED_RULE:**
  - Inventory all exposed POST endpoints under `src/platform/api/controllers/economic/**` against the engine registry. Any endpoint whose command has no registered handler SHOULD fail-fast at host startup, not at request time. Candidate sub-clause of `G-COMPLOAD-01` (registry membership).
  - Concrete fix (out of scope for the current remediation): register `CreateRevenueContractHandler` in the revenue bootstrap module.
- **RELATED GUARD RULES:** G-COMPLOAD-01 (registry membership), G-PROGCOMP-03 (classification-aligned domain wiring)

## OPS-VAL-006 — Domain enum values duplicated as controller string whitelists

- **CLASSIFICATION:** platform / api / validation
- **SOURCE:** Fix 2 implementation choice — whitelists in `AuditController`, `DiscrepancyController`, `SubjectController` (×3 enums), `ChargeController`
- **SEVERITY:** S2
- **DESCRIPTION:**
  - To avoid a domain-project reference from `src/platform/api/**`, Fix 2 mirrored each domain enum as a `HashSet<string>` whitelist on the controller. This keeps the API layer domain-independent but introduces a silent drift risk: adding a new enum member in `src/domain/economic-system/**/value-object/*.cs` does NOT automatically update the controller whitelist, so the controller will 400-reject valid values until the whitelist is edited.
- **PROPOSED_RULE:**
  - Either: (a) permit `src/platform/api/**` to reference enum types from `Whycespace.Domain.*.ValueObject` (small scope — enum imports only) and use `Enum.TryParse<TEnum>` directly; or (b) ship a shared `EnumContract<T>` record in `Whycespace.Shared.Contracts.Common` that publishes the domain enum members as a string array, so the API whitelists read from one authoritative list.
  - Candidate addition to `infrastructure.guard.md` Platform Boundaries section.
- **RELATED GUARD RULES:** R-PLAT-05 (API must not reference runtime/engines/infrastructure — note: Domain is NOT in that block-list today; this rule would clarify)

## OPS-VAL-007 — EventSchemaRegistry missing `InboundEventType` for `ExposureCreatedEvent`

- **CLASSIFICATION:** projection / event-schema / dead-letter
- **SOURCE:** host-1 log during Scenario A re-run — `GenericKafkaProjectionConsumerWorker` routed every message from `whyce.economic.risk.exposure.events` to its deadletter topic with:
  > `System.InvalidOperationException: EventSchemaRegistry has no InboundEventType registered for 'ExposureCreatedEvent'. Register via the (string, EventVersion, Type stored, Type inbound) overload.`
- **SEVERITY:** S1 (silent projection failure — API reports 200 and event is persisted, but the read model is never updated; reads will always 404)
- **DESCRIPTION:**
  - The risk context now passes Scenario A at the command/API level, but the projection consumer cannot deserialize `ExposureCreatedEvent` because the schema registry lacks the inbound type mapping. Every such message is poisoned and routed to deadletter.
  - Risk projection schemas exist in `whyce_projections` (per the earlier readiness summary), but rows never land because the consumer deadletters before calling the reducer.
- **PROPOSED_RULE:**
  - Startup-time guard: enumerate registered event topics vs the `EventSchemaRegistry`; fail fast on any topic whose event type has no inbound mapping. Candidate sub-clause of R-K-24 (Kafka header contract) or a new R-PROJ-SCHEMA-COMPLETENESS-01.
  - Concrete fix (out of scope here): register `ExposureCreatedEvent` inbound type in the risk bootstrap module alongside its outbound type.
- **RELATED GUARD RULES:** R-K-14 (event schema versioning), R-K-24 (header contract), GE-04 (event-first architecture)

## OPS-VAL-008 — Domain invariant violations surface as 500, not 400

- **CLASSIFICATION:** platform / api / error-handling
- **SOURCE:** Scenario A re-run in-flight observations: exchange `CurrencyPair invariant violated: Base ('USD') must precede Quote ('EUR-...')` and subject `Invalid EconomicRefType 'VaultAccount' for SubjectType 'Participant'` both bubbled to `ExceptionHandlerMiddleware` as generic 500 ProblemDetails
- **SEVERITY:** S2
- **DESCRIPTION:**
  - Fix 2 catches enum *parse* failures at the controller boundary. But domain invariants raised from inside aggregate constructors / rule classes (`CurrencyPair..ctor`, `EconomicRefRules.Validate`) still throw `ArgumentException` / `InvalidOperationException`, propagate through the engine handler, up through the dispatcher, and into the global exception handler. The caller sees a generic 500 with only a `traceId`, no domain error code, no `ApiResponse.Fail(code, ...)` envelope.
  - Not a blocker for the original four defects (my payloads were technically invalid), but the user experience is poor and violates the spirit of Fix 2.
- **PROPOSED_RULE:**
  - Either: (a) a dedicated `DomainExceptionMiddleware` placed before `ExceptionHandlerMiddleware` that converts `DomainException` (and by policy-override, `ArgumentException` / `InvalidOperationException` thrown from `Whycespace.Domain.*` assemblies) into `400 BadRequest(ApiResponse.Fail(...))` with a stable code derived from the command type; or (b) adopt a Result-type pattern in handlers so invariant violations flow back as `CommandResult.Fail` and never throw.
  - Candidate sub-clause of R-PLAT-12 (CommandResult envelope) or a new R-PLAT-13.
- **RELATED GUARD RULES:** R-PLAT-12 (CommandResult envelope), R-PLAT-01 (API layer purity)
