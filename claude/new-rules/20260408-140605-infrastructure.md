# NEW RULE: bootstrap must apply outbox + chain + HSID migrations

```
CLASSIFICATION: infrastructure
SOURCE:         post-execution audit sweep of docs/infrastructure/bootstrap-and-operations-playbook.md
DATE:           2026-04-08
SEVERITY:       S2 (structural)
RELATED CHECKS: CHECK-INFRA-TOPICS-01, CHECK-INFRA-HSID-INITDB-01 (claude/audits/infrastructure.audit.md)
RELATED ARCHIVE: claude/new-rules/_archives/20260408-000000-phase1-gate-blockers.md
```

## DESCRIPTION

The canonical bootstrap path applies only `infrastructure/data/postgres/event-store/migrations/` to the `whyce-postgres` event store. Three additional migration directories exist on disk and are required for a healthy first-run host but are **never executed**:

- `infrastructure/data/postgres/outbox/migrations/` (5 files: 001_outbox.sql … 005_outbox_next_retry_at.sql)
- `infrastructure/data/postgres/chain/migrations/` (1 file: 001_whyce_chain.sql)
- `infrastructure/data/postgres/hsid/migrations/` (1 file: 001_hsid_sequences.sql)

Evidence:

1. `infrastructure/deployment/docker-compose.yml` mounts only the event-store migrations into `/docker-entrypoint-initdb.d`:
   ```yaml
   - ../data/postgres/event-store/migrations:/docker-entrypoint-initdb.d
   ```
2. `infrastructure/deployment/scripts/migrate.sh` iterates only `event-store/migrations`:
   ```bash
   MIGRATIONS_DIR="$INFRA_DIR/data/postgres/event-store/migrations"
   for SQL_FILE in "$MIGRATIONS_DIR"/*.sql; do ... done
   ```
3. `infrastructure/deployment/scripts/bootstrap.sh` invokes `migrate-projections.sh` (projections DB only) and never invokes `migrate.sh` for the event store at all.

Net effect: a clean clone + `bootstrap.sh` produces an event-store database missing the outbox, chain anchor, and HSID sequence tables. The runtime crashes with `HSID FATAL` on first command.

## PROPOSED RULE

`bootstrap.sh` MUST apply, in order, against `whyce_eventstore`:

1. `event-store/migrations/*.sql`
2. `hsid/migrations/*.sql`
3. `outbox/migrations/*.sql`
4. `chain/migrations/*.sql`

Implementation options (pick one — both satisfy the rule):

- **Option A (preferred):** extend the postgres `initdb.d` mount to cover all four directories, either by adding additional bind mounts or by symlinking/copying into a single staging dir. Pros: works on first-ever container start; no script ordering needed.
- **Option B:** generalize `migrate.sh` to iterate `event-store`, `hsid`, `outbox`, `chain` in that order, and have `bootstrap.sh` call it as a new phase between `[2/5]` (core up) and `[3/5]` (init sidecars).

Either option must be paired with an idempotency guard (existing migrations include `IF NOT EXISTS` / `ON CONFLICT` clauses; verify before merging).

The playbook at `docs/infrastructure/bootstrap-and-operations-playbook.md` §7.1 currently documents this gap with a manual workaround. Once the rule is implemented, **delete that workaround block.**

## VALIDATION

- After `bootstrap.sh` on a clean volume (`docker compose ... down -v && bootstrap.sh`), the following queries must succeed:
  ```sql
  SELECT 1 FROM whyce.events       LIMIT 1;
  SELECT 1 FROM whyce.outbox       LIMIT 1;
  SELECT 1 FROM whyce.chain_anchor LIMIT 1;
  SELECT nextval('whyce.hsid_root_seq');  -- or whatever sequence 001_hsid_sequences.sql defines
  ```
- Runtime smoke command must complete without `HSID FATAL`.
- `scripts/infrastructure-check.sh` should be extended to verify all four migration directories have been applied (assert table presence).

## SEVERITY JUSTIFICATION

S2 (structural): blocks deterministic reproducible bootstrap from zero, which is principle #2 of the playbook and a phase 1 gate requirement. Not S0/S1 because a manual workaround exists and the canonical layout is correct — only the wiring is incomplete.
