# Finding 1 — Event File Naming Inconsistency: control-system/access-control/role

## CLASSIFICATION
domain

## SOURCE
Post-execution audit sweep of `src/domain/control-system/` against delivery template 01-domain-skeleton.md

## DESCRIPTION
`src/domain/control-system/access-control/role/event/RoleDefinedEvent.cs` contains three domain events:
- `RoleDefinedEvent`
- `RolePermissionAddedEvent`
- `RoleDeprecatedEvent`

The filename `RoleDefinedEvent.cs` names only the first event. Every other multi-event file in control-system uses `*Events.cs` naming (e.g., `PermissionEvents.cs`, `AuthorizationEvents.cs`, `SystemAlertEvents.cs`). This is a structural naming inconsistency.

The fix (rename to `RoleEvents.cs`) was deliberately deferred because CLAUDE.md $5 prohibits file renames.

## PROPOSED_RULE
Every event file containing more than one domain event class MUST be named `{Domain}Events.cs`, not after any individual event. A file named `{Concept}Event.cs` MUST contain exactly one event class. Scan: any file under `src/domain/**/event/` whose name matches `*Event.cs` (singular) and contains more than one `record` declaration is a violation.

## SEVERITY
S3 — naming drift, does not affect runtime behaviour or replay safety. Remediate in the next dedicated naming-cleanup pass for control-system.
