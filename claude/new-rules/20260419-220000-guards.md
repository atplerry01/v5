# R2.E.3 — Partition Skew Visibility (2026-04-19)

Captured per CLAUDE.md $1c. Third R2.E deliverable — builds on R2.E.1 rebalance observability + R2.E.2 lag histogram. Closes the §12 "Hot-partition detection" / "Hot-key detection" / "Partition skew visibility" rows from the post-R2.A.D gap-matrix re-survey.

**Insight:** the R2.E.2 `consumer.lag_messages` histogram tagged `{topic, worker, partition}` ALREADY surfaces hot-partition detection — a partition whose lag percentile (P95 / P99) is anomalously high relative to its siblings IS a hot partition by definition. What's missing is the per-partition throughput counter needed to compute skew as the max:mean ratio across partitions. R2.E.3 adds that single signal without wiring new call sites.

## R-CONSUMER-PARTITION-SKEW-01 — Per-Partition Throughput Counter

- **CLASSIFICATION:** runtime / event-fabric / consumer / observability
- **SEVERITY:** S3
- **DESCRIPTION:**
  - `KafkaLagObservability.Record` increments `consumer.messages_processed` (Counter<long>) on every successful call, tagged identically to `consumer.lag_messages`: `{topic, worker, partition}`. Operator dashboards compute partition skew as the max:mean or max:min ratio across the partition tag values at a fixed time window.
  - The counter is co-located with the lag histogram in the same `Record` method body so the two signals stay in lockstep: every message that contributes a lag observation ALSO contributes a throughput tick. Divergence (lag recorded but throughput missed, or vice versa) is impossible by construction.
  - `consumer.lag_unknown` (R2.E.2) path does NOT increment `messages_processed` — semantic parity: the lag-unknown path didn't successfully observe the message's position, so it also shouldn't claim throughput.
- **CHECK:**
  - `KafkaLagObservability.cs` exposes `consumer.messages_processed` via the same `Whycespace.Kafka.Consumer` meter.
  - No additional call sites needed — the 11 worker files already call `Record` after every successful `Consume` per R2.E.2.

## R-CONSUMER-HOT-PARTITION-DETECTION-01 — Lag Histogram IS the Hot-Partition Signal

- **CLASSIFICATION:** runtime / event-fabric / consumer / observability
- **SEVERITY:** S3
- **DESCRIPTION:**
  - Hot-partition detection is NOT a separate metric — it is a derived signal from the `consumer.lag_messages` histogram tagged `{partition}`. A partition whose lag P95 / P99 is > N× the median across partitions IS a hot partition. Threshold `N` lives in operator alerting config (Prometheus / Grafana), NOT in the adapter.
  - This is deliberate: adapter-level thresholds create deployment-specific configuration drift. The raw signal (lag per partition) is universal; the alert threshold is deployment-specific and belongs in the operator surface.
  - Correspondingly, R2.E.3 does NOT add a new `consumer.hot_partition_detected` event or counter. The existing metrics compose.
- **CHECK:**
  - Documentation entry in guard/audit references this derived-signal shape. No new adapter code expected.

## R-CONSUMER-HOT-KEY-DETECTION-DEFERRED-01 — Hot-Key Detection Deferred

- **CLASSIFICATION:** runtime / event-fabric / consumer / observability / scope
- **SEVERITY:** S3
- **DESCRIPTION:**
  - Hot-key detection (identifying a single key that drives most messages within a partition) requires stateful tracking — count-min-sketch or reservoir sampling — with reset semantics on rebalance and memory budget per consumer. This is a non-trivial future workstream (R2.E.4 or R3) and is explicitly out of scope for R2.E.3.
  - Why deferred: (a) unbounded-cardinality keys (aggregate IDs) make tag-based metrics infeasible, (b) operator value is lower than partition-skew + lag combined (hot keys manifest as hot partitions first, so lag-per-partition already catches them indirectly).
- **CHECK:**
  - Gap-matrix §12 "Hot-key detection" remains ABSENT with a note referencing this deferral rule.
