# DET-SEED-DERIVATION-01 — ID seed derivation must exclude non-deterministic inputs

## DESCRIPTION

The deterministic ID engine `IIdGenerator.Generate(seed)` is the canonical seam for producing reproducible identifiers. The engine itself is deterministic — but determinism is only as strong as its input. Three sites in the phase-1-gate landing call the seam correctly while constructing the seed string from `_clock.UtcNow.Ticks`, defeating the entire mechanism. Same logical command, different IDs on replay.

Affected sites at time of capture:
- `src/platform/api/controllers/TodoController.cs:45`
- `src/runtime/dispatcher/SystemIntentDispatcher.cs:32-35`
- `src/systems/midstream/wss/WorkflowDispatcher.cs`

## PROPOSED_RULE

When invoking `IIdGenerator.Generate(seed)` (or any future seam producing a deterministic identifier from a seed string), the seed MUST be composed exclusively of stable command coordinates:

**Allowed seed components:**
- Aggregate identifier
- Command type name
- Aggregate version (or expected version)
- Correlation id and causation id (themselves derived deterministically)
- Domain/context discriminators that are themselves deterministic

**Forbidden seed components:**
- `IClock.UtcNow` / `DateTime.*` / `Stopwatch.*` — any wall-clock or monotonic-clock value
- `Guid.NewGuid()` / `Random.*` / `RandomNumberGenerator.*`
- Process id, thread id, machine name, environment variables
- Hashes of any of the above

## ENFORCEMENT

- Static check: search `IIdGenerator.Generate(` calls; flag any seed-string interpolation containing `Clock`, `Now`, `Ticks`, `Guid`, `Random`.
- Architecture test: extend `WbsmArchitectureTests` with a Roslyn rule that fails the build on the above pattern.

## SEVERITY

S1 — architectural, blocks merge. Non-deterministic IDs propagate downstream into chain anchors, projection keys, and idempotency tokens, where they silently break replay determinism, projection idempotency, and audit chain integrity at once.
