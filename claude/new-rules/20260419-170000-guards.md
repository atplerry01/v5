# R2.A.D.3d — Redis Circuit Breaker (2026-04-19)

Captured per CLAUDE.md $1c. R2.A.D.3b shipped the Kafka breaker. R2.A.D.3d adds the Redis breaker — the fourth and final dependency-level breaker in the R2.A.D story (Postgres pool R2.A.D.3c still pending design review). Shared between the two Redis consumers so a Redis outage trips one breaker and posture reflects it uniformly.

## R-REDIS-BREAKER-01 — Shared "redis" breaker protects all Redis operations

- **CLASSIFICATION:** runtime / resilience / redis
- **SEVERITY:** S1
- **DESCRIPTION:**
  - One `ICircuitBreaker` instance named `"redis"` registered in host composition alongside the `IConnectionMultiplexer`. Registered as plain `ICircuitBreaker` so `CircuitBreakerRegistry` picks it up automatically.
  - Two consumers share this breaker:
    1. `RedisExecutionLockProvider.TryAcquireAsync` + `ReleaseAsync` — wraps `StringSetAsync` and `ScriptEvaluateAsync` Redis calls.
    2. `RedisHealthSnapshotProvider.GetSnapshotAsync` — wraps the `PingAsync` probe.
  - Both providers already swallow Redis exceptions with fail-closed defaults (`return false` for lock, `IsHealthy: false` snapshot). The breaker sits INSIDE the outer try/catch so it can count Redis failures; `CircuitBreakerOpenException` is caught at the same boundary and folds into the same fail-closed default.
  - Redis failure observation point: inside the breaker's lambda, Redis calls throw normally. The breaker counts these as failures per R-CIRCUIT-BREAKER-01. The outer catch (pre-R2.A.D.3d) still returns the fail-closed default — no new exception propagation paths.
- **CHECK:**
  - `sp.GetService<ICircuitBreakerRegistry>()?.Get("redis")` returns the breaker.
  - Both providers accept `ICircuitBreaker` via constructor; wrap their Redis calls in `_breaker.ExecuteAsync(...)`.
  - Fail-closed contract preserved: `TryAcquireAsync` still returns `false` on breaker-open or Redis exception; `GetSnapshotAsync` still returns `IsHealthy: false` snapshot on both paths.
  - Aggregator whitelist maps `"redis"` → `"redis_breaker_open"` so health-posture surfaces the signal alongside `"redis_unhealthy"` / `"redis_degraded_latency"` from the existing `RedisHealthCheck`.

## R-REDIS-BREAKER-OPEN-FAIL-CLOSED-01 — Open-state preserves fail-closed contract

- **CLASSIFICATION:** runtime / resilience / redis / boundary
- **SEVERITY:** S1
- **DESCRIPTION:**
  - `RedisExecutionLockProvider.TryAcquireAsync` MUST return `false` on `CircuitBreakerOpenException`. The existing runtime control-plane translates false into the canonical `"execution_lock_unavailable"` `CommandResult.Failure(DependencyUnavailable)` — no new exception propagation path appears at the API edge.
  - `RedisExecutionLockProvider.ReleaseAsync` MUST be a no-op on `CircuitBreakerOpenException`. Release is best-effort; the lease expires naturally on TTL regardless. Matches existing "swallow Redis exceptions" contract.
  - `RedisHealthSnapshotProvider.GetSnapshotAsync` MUST return an unhealthy snapshot on `CircuitBreakerOpenException`. Operators see `IsHealthy: false` with `PingLatencyMs: null` — same shape as a transport failure, additional breaker signal surfaces via the aggregator.
- **CHECK:**
  - `CircuitBreakerOpenException` never escapes either provider.
  - `RedisExecutionLockProvider.cs` and `RedisHealthSnapshotProvider.cs` contain `catch (CircuitBreakerOpenException` → fail-closed default. Architecture guard is simple grep.
