# R3.A.4 — Workflow Cancellation Lifecycle Event (2026-04-20)

Captured per CLAUDE.md $1c. Fourth R3 deliverable. Closes the §13 "Workflow cancellation" PARTIAL row.

**Why this matters:** caller-driven cancellation (request abort, host shutdown, explicit cancel) today propagates the token but emits no domain event. The event stream looks like a workflow that simply stopped receiving step-completed events mid-flight — indistinguishable from a crashed workflow. Auditors cannot differentiate deliberate cancellation from infrastructure failure.

**Scope discipline:** this is a PURE lifecycle-event closure. R3.A.4 does NOT add a "cancel command" surface, a cancellation-request API, or human-approval wait semantics. It only emits an event on the existing caller-cancel path. A cancel-from-external-trigger seam is R3.A.3 (suspend/resume) scope.

## R-WORKFLOW-CANCELLATION-EVENT-01 — WorkflowExecutionCancelledEvent + Cancelled Status

- **CLASSIFICATION:** domain / workflow / lifecycle
- **SEVERITY:** S2
- **DESCRIPTION:**
  - New domain record `WorkflowExecutionCancelledEvent(AggregateId, string? StepName, string Reason)` in `src/domain/orchestration-system/workflow/execution/event/`. `StepName` is nullable because cancellation MAY happen before any step started (empty workflow cancelled). `Reason` carries "caller_cancellation" (canonical short tag) plus the raw message of the propagating `OperationCanceledException` for audit.
  - New shared-contract schema `WorkflowExecutionCancelledEventSchema(Guid AggregateId, string? StepName, string Reason)` in `WorkflowExecutionEventSchemas.cs` — mirrors the pattern for the other 5 lifecycle events.
  - `WorkflowExecutionStatus` enum gains `Cancelled` value. Ordering: `NotStarted`, `Running`, `Completed`, `Failed`, `Cancelled`. Append-only preserves stored enum integers.
  - `WorkflowExecutionAggregate.Apply` gains a `case WorkflowExecutionCancelledEvent` that transitions `_status → Cancelled`. No additional fields captured — the reason is observability, not a replay-bearing value.
  - New error: `WorkflowExecutionErrors.CannotCancelAfterTerminal` — guards the Cancelled → anything transition (cancellation from Completed/Failed is illegal; would violate the terminal-state invariant).
- **CHECK:**
  - All 4 files exist: event record, schema, status enum updated, aggregate Apply case added.

## R-WORKFLOW-CANCELLATION-FACTORY-01 — Canonical Construction via Lifecycle Factory

- **CLASSIFICATION:** runtime / engine / workflow / lifecycle
- **SEVERITY:** S2
- **DESCRIPTION:**
  - Follow the existing `WorkflowLifecycleEventFactory` pattern (E-LIFECYCLE-FACTORY-CALL-SITE-01): `WorkflowLifecycleEventFactory.Cancelled(Guid workflowExecutionId, string? stepName, string reason)` returns a new `WorkflowExecutionCancelledEvent`. Engine calls the factory; the aggregate never exposes a `Cancel()` command method (same discipline as `Resume` — engine-guard rule 3 prohibits T1M from calling aggregate mutation methods).
- **CHECK:**
  - `WorkflowLifecycleEventFactory.Cancelled` method exists.
  - `WorkflowExecutionAggregate` has NO public `Cancel(...)` method (grep confirms zero hits).

## R-WORKFLOW-CANCELLATION-ENGINE-EMISSION-01 — Engine Emits Before Re-Throwing

- **CLASSIFICATION:** runtime / engine / workflow / cancellation
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `T1MWorkflowEngine.ExecuteAsync`'s `catch (OperationCanceledException) when (cancellationToken.IsCancellationRequested)` branch MUST emit `_lifecycleFactory.Cancelled(...)` into `context.EmitEvent(...)` BEFORE the `throw;`. The re-throw contract is preserved — callers still receive the OCE unchanged.
  - Reason string format: `"caller_cancellation: {ex.GetType().Name}: {ex.Message}"`. The `caller_cancellation` tag is canonical and short so dashboards can filter on it; the BCL info follows for audit detail.
  - `stepName` passed to the factory is `stepDefinition.StepName` from the surrounding for-loop — the step that was in-flight when cancellation arrived. Nullable null when the cancellation arrived before any step started (but this branch in the engine only runs inside the step-dispatch try block, so a step is always defined).
  - Outcome recording (R3.A.1 `cancelled` tag) is unchanged — this rule is additive: metrics + NEW domain event + re-throw.
- **CHECK:**
  - The cancel catch branch in `WorkflowEngine.cs` contains `context.EmitEvent(_lifecycleFactory.Cancelled(...)` BEFORE `throw;`.
  - Architecture test pins the emission-before-throw ordering via source scan.

## R-WORKFLOW-CANCELLATION-REPLAY-DETERMINISM-01 — Cancelled Status Is a Terminal Replay State

- **CLASSIFICATION:** runtime / engine / workflow / determinism
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `Cancelled` is a TERMINAL status — a cancelled workflow cannot resume. Replay of a stream ending in `WorkflowExecutionCancelledEvent` lands at `Status = Cancelled` and remains there. This matches the existing `Completed` status discipline.
  - `WorkflowLifecycleEventFactory.Resumed` already guards against non-Failed states (`aggregate.Status != Failed → CannotResumeUnlessFailed`). No change needed — the existing guard correctly blocks resume from Cancelled.
  - A cancelled workflow that the operator later wants to re-run is a NEW workflow execution (new aggregate id), not a resume. This preserves the event-sourcing discipline — cancelled events are immutable history.
- **CHECK:**
  - `WorkflowLifecycleEventFactory.Resumed` guard rejects `WorkflowExecutionStatus.Cancelled` just as it rejects `Completed` — the existing `status != Failed` check already covers this.
  - No new `ResumeFromCancelled` method or similar.

## Out of scope for R3.A.4

- **Cancel command surface** — no `WorkflowCancelCommand` or admin-initiated cancellation. Current cancellation path is caller-driven via `CancellationToken`. External-trigger cancellation is R4 operator surface scope.
- **Step-level cancellation granularity** — the event records the step name that was in-flight, but does not distinguish "soft cancel — let step finish" from "hard cancel — interrupt step." Current behaviour is hard cancel via CTS; softer semantics are a future workstream.
- **Cancellation-initiated compensation** — a cancelled workflow may need compensation for completed-but-now-rolled-back steps. R3.A.4 emits the event; compensation orchestration on top of it is separate.
