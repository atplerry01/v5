## Why this is safer than invasive-rename-in-CS.5

1. **Atomicity.** A structural realignment gate should realign structure. Adding
   invasive identifier renames mixes two kinds of change in one commit, making
   rollback harder.
2. **Replay stability.** CS.0 baseline verified zero live DomainRoute callers
   and zero projections/engines/kafka refs for content-system, but any
   persisted event streams from integration tests encode class-name type
   metadata. Structural-only changes preserve decodability.
3. **Band split consistency.** §CD-17 split README work into Band-S (structural,
   in-gate) and Band-F (semantic, CS.13). Extending the same band split to
   identifier renames is a coherent generalisation.
