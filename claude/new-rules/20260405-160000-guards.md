# New Rule Findings — Phase 1 Batch 1 Audit Sweep (2026-04-05)

## Finding 1: Non-Deterministic SystemClock

- **CLASSIFICATION**: behavioral / determinism
- **SOURCE**: `src/platform/host/Program.cs` — `SystemClock` class
- **DESCRIPTION**: `SystemClock` uses `DateTimeOffset.UtcNow` directly. Violates BDIM-24 / DETDIM-01. Pre-existing code, not introduced by this execution.
- **PROPOSED_RULE**: All `IClock` implementations in sandbox mode MUST accept an injectable time source or use deterministic seed. Production implementations may use system time with explicit opt-in.
- **SEVERITY**: S0 (CRITICAL)

## Finding 2: Non-Deterministic ID Generation in SystemIntentDispatcher

- **CLASSIFICATION**: behavioral / runtime / determinism
- **SOURCE**: `src/runtime/pipeline/SystemIntentDispatcher.cs` — lines 25-26
- **DESCRIPTION**: `CorrelationId` and `CausationId` generated via `Guid.NewGuid()`. Violates BDIM-24 / DETDIM-01. Has existing TODO comment acknowledging the issue.
- **PROPOSED_RULE**: All correlation/causation ID generation MUST use an `IIdGenerator` abstraction. `Guid.NewGuid()` is prohibited in runtime pipeline.
- **SEVERITY**: S0 (CRITICAL)

## Finding 3: Non-Deterministic InMemoryChainAnchor

- **CLASSIFICATION**: runtime / infrastructure / determinism
- **SOURCE**: `src/platform/host/Program.cs` — `InMemoryChainAnchor` class
- **DESCRIPTION**: Uses `Guid.NewGuid()` for `BlockId` and `DateTimeOffset.UtcNow` for timestamps. Chain anchoring must be deterministically reproducible.
- **PROPOSED_RULE**: Chain anchor `BlockId` MUST be derived from deterministic hash of events + previous block. Timestamps MUST come from `IClock`.
- **SEVERITY**: S1 (HIGH)
