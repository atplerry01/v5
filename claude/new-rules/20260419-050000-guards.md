# R2.A.1 — Retry Primitive Discipline (2026-04-19)

Captured per CLAUDE.md $1c and per R2 entry criteria in [`runtime-upgrade-plan.md`](../project-topics/v2b/runtime-upgrade-plan.md) §3:

> "Guard additions for new primitives drafted in `claude/new-rules/` before code lands."

These four rules lock the semantics of `IRetryExecutor` (the R2 canonical retry primitive, decision D1 LOCKED as "Custom" over Polly) and its consumers. They apply to the retry-executor code landing in R2.A.1 and to every retry site migrated in R2.A.2+.

---

## R-RETRY-DET-01 — Retry executor is fully replay-deterministic (S0)

- **CLASSIFICATION:** runtime / resilience / determinism
- **SEVERITY:** S0 (constitutional extension — violates replay determinism)
- **DESCRIPTION:**
  - `IRetryExecutor.ExecuteAsync` MUST be a pure function of (input operation, policy, operation coordinates, `IClock.UtcNow` reads at attempt boundaries, `IRandomProvider.NextDouble` seeded from operation coordinates). No other entropy source.
  - **Forbidden:** `DateTime.UtcNow`, `DateTimeOffset.UtcNow`, `Environment.TickCount`, `Stopwatch` (for values that land in events), `Random` / `Random.Shared` / `RandomNumberGenerator` inside the executor.
  - **Permitted:** `Task.Delay(backoff, ct)` for the actual wait (replay does NOT re-execute sleeps — the replay path reads events from store and skips wall-clock waits entirely). The DELAY VALUE must come from the deterministic backoff + jitter computation; the wait itself is free.
  - Every `IClock.UtcNow` read MUST happen at a deterministic point (before/after each attempt) and the read value MUST be captured in the returned `RetryAttemptRecord` so replay can verify identity.
- **CHECK:**
  - Architecture test scans `src/runtime/resilience/**` for forbidden tokens.
  - Unit test: running the same operation twice with the same `RetryOperationContext` produces identical `RetryResult` (same attempts, same delays, same jitter) when the inner operation returns the same outcomes.

## R-RETRY-CAT-01 — Retry eligibility is governed by `RuntimeFailureCategory` (S1)

- **CLASSIFICATION:** runtime / resilience
- **SEVERITY:** S1
- **DESCRIPTION:**
  - Retry eligibility MUST be computed from `RuntimeFailureCategory`, never from error strings.
  - Canonical table (R2.A.1 implementation):
    - **Retryable (bounded):** `Timeout`, `DependencyUnavailable`, `ConcurrencyConflict`, `ResourceExhausted`, `ExecutionFailure`.
    - **Retryable (policy-deferred special):** a new `PolicyEvaluationDeferred` category (R2.A.2) — distinct from `PolicyDenied`.
    - **Non-retryable (terminal reject):** `AuthorizationDenied`, `PolicyDenied`, `ValidationFailed`, `RuntimeGuardRejection`, `InvalidState`, `PoisonMessage`.
    - **Non-retryable (propagate):** `Cancellation`.
  - String-based permanent-failure heuristic in `RetryPolicy.IsPermanentFailure` is deprecated by R2.A.1. Existing `PostToLedgerStep` caller migrates in R2.A.2.
- **CHECK:** unit test covers every `RuntimeFailureCategory` value against the retryability predicate. New categories force a compile-time exhaustive switch.

## R-RETRY-CAP-01 — Bounded attempts; no infinite retry (S0)

- **CLASSIFICATION:** runtime / resilience / safety
- **SEVERITY:** S0 (DoS amplifier if violated — closes POL-FAIL-CLASS-01's "infinite retry on policy unavailable" concern)
- **DESCRIPTION:**
  - Every `IRetryExecutor.ExecuteAsync` call MUST terminate in ≤ `policy.MaxAttempts` attempts (default 3) regardless of failure category.
  - After exhaustion, the executor returns `RetryOutcome.Exhausted` with the final attempt's error preserved. It MUST NOT loop further.
  - Policy-deferred outcomes (R2.A.2 `PolicyEvaluationDeferred`) obey the same cap — on the final attempt, a defer converts to a terminal `FAIL_CLOSED` per POL-FAIL-CLASS-01.
- **CHECK:** unit test — injected inner operation always returns `DependencyUnavailable`; executor completes in exactly `MaxAttempts` calls; final `Outcome == Exhausted`.

## R-RETRY-EVIDENCE-01 — Every retry emits auditable evidence (S1)

- **CLASSIFICATION:** runtime / resilience / audit
- **SEVERITY:** S1
- **DESCRIPTION:**
  - `RetryResult` MUST carry an `IReadOnlyList<RetryAttemptRecord>` with one record per attempt (including the first).
  - Each record carries: attempt number (1-based), `StartedAt` (from `IClock`), `CompletedAt` (from `IClock`), `DelayBeforeAttempt` (deterministic backoff+jitter, zero for attempt 1), `FailureCategory` (null on the winning attempt), `Error` (safe message).
  - When a caller persists retry outcomes (R2.A.2+), they MUST emit `RetryAttemptedEvent` / `RetryExhaustedEvent` sourced from this record — never reconstructed from scratch.
- **CHECK:** unit test — returned `RetryResult.Attempts.Count == AttemptsMade`; every `DelayBeforeAttempt` is reproducible from the same seed.
