# R3.A.3 — Workflow Suspend Lifecycle Event (2026-04-20)

Captured per CLAUDE.md $1c. Fifth R3 deliverable. Closes the last non-terminal §13 row (Workflow suspend / resume) and unblocks Human-approval wait-state (§13 ABSENT).

**Why this matters:** suspending a workflow is how long-running processes wait for external signal (operator approval, external dependency becoming ready, a timer). Today the engine only supports Failed-then-Resumed recovery — there is no "pause intentionally, resume later" path. R3.A.3 ships the event + aggregate state machine; the actual suspend-trigger seam (API to caller or internal yield point) is follow-up.

**Scope boundary (explicit):** R3.A.3 adds the EVENT + STATUS + FACTORY + APPLY. It does NOT add:
- A caller-initiated suspend command/API (R4 operator-surface scope).
- A step-yield mechanism (the ability for a step to say "I'm done for now; wake me on signal X"). Future R3 scope.
- A suspend timeout (automatic cancellation of long-suspended workflows). Future — composes on R3.A.1 timeouts.

This rule is deliberately narrow to match R3.A.4's closure pattern: land the event surface so downstream features can compose on it.

## R-WORKFLOW-SUSPEND-EVENT-01 — WorkflowExecutionSuspendedEvent + Suspended Status

- **CLASSIFICATION:** domain / workflow / lifecycle
- **SEVERITY:** S2
- **DESCRIPTION:**
  - New domain record `WorkflowExecutionSuspendedEvent(AggregateId, string? StepName, string Reason)` in `src/domain/orchestration-system/workflow/execution/event/`. `StepName` is nullable so the event is usable for workflow-level suspend (pre-first-step or post-last-step) in addition to mid-step suspend.
  - New shared-contract schema `WorkflowExecutionSuspendedEventSchema(Guid AggregateId, string? StepName, string Reason)` in `WorkflowExecutionEventSchemas.cs`.
  - `WorkflowExecutionStatus` enum gains `Suspended` value (ordinal 5, appended — preserves stored enum integers post-R3.A.4's Cancelled at 4).
  - `Suspended` is NON-TERMINAL: from Suspended the workflow can transition back to Running via `WorkflowExecutionResumedEvent` (see R-WORKFLOW-SUSPEND-RESUME-GUARD-01). The four terminal statuses remain `Completed`, `Cancelled`, and `Failed-after-exhaustion`. `Failed` is resumable-but-has-been-so-from-the-start; R3.A.3 adds `Suspended` as a second resumable state.
  - `WorkflowExecutionAggregate.Apply` gains a `case WorkflowExecutionSuspendedEvent` that transitions `_status → Suspended`. Symmetric to the R3.A.4 `Cancelled` case in shape.
- **CHECK:**
  - All 4 files exist: event record, schema, status enum updated, aggregate Apply case added.

## R-WORKFLOW-SUSPEND-FACTORY-01 — Canonical Construction via Lifecycle Factory

- **CLASSIFICATION:** runtime / engine / workflow / lifecycle
- **SEVERITY:** S2
- **DESCRIPTION:**
  - Follow the `Resumed` / `Cancelled` factory pattern: `WorkflowLifecycleEventFactory.Suspended(Guid workflowExecutionId, string? stepName, string reason)` constructs the event. Aggregate has NO public `Suspend(...)` method — engine-guard rule 3 still holds.
  - The factory enforces a precondition: the aggregate's current status MUST be `Running` (suspending from NotStarted / Failed / Completed / Cancelled / already-Suspended is illegal). Matches the discipline of `Resumed` which guards against non-Failed/Suspended statuses. New error: `WorkflowExecutionErrors.CannotSuspendUnlessRunning`.
- **CHECK:**
  - `WorkflowLifecycleEventFactory.Suspended` method exists with the `Status != Running` guard.
  - `WorkflowExecutionAggregate` has NO public `Suspend(...)` method.

## R-WORKFLOW-SUSPEND-RESUME-GUARD-01 — Resume Accepts Failed OR Suspended

- **CLASSIFICATION:** runtime / engine / workflow / lifecycle
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `WorkflowLifecycleEventFactory.Resumed` (shipped pre-R3.A.3) currently guards against non-`Failed` statuses. R3.A.3 extends the guard: resume is legal from either `Failed` OR `Suspended`. Terminal statuses (`Completed`, `Cancelled`) remain un-resumable.
  - The existing `WorkflowExecutionErrors.CannotResumeUnlessFailed` is renamed to `WorkflowExecutionErrors.CannotResumeUnlessFailedOrSuspended` with updated message. Renaming is safe because the error is an internal invariant string (not a wire-format contract).
  - `WorkflowExecutionResumedEvent` itself is unchanged — it simply moves the aggregate `Running` regardless of whether the previous status was `Failed` or `Suspended`. Replay reproduces the same end state either way.
- **CHECK:**
  - `WorkflowLifecycleEventFactory.Resumed` guard allows both `Failed` and `Suspended`.
  - Unit tests of the guard exist OR an architecture test pins the status-pair check.

## R-WORKFLOW-SUSPEND-SCHEMA-REGISTRATION-01 — Event Deserializer Recognizes the New Schema

- **CLASSIFICATION:** runtime / event-fabric / registration
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `WorkflowExecutionSchemaModule` registers `WorkflowExecutionSuspendedEventSchema` alongside the existing 6 lifecycle schemas (Started, StepCompleted, Completed, Failed, Resumed, Cancelled). Payload mapper added symmetrically.
- **CHECK:**
  - `WorkflowExecutionSchemaModule` registers the new schema + payload mapper.

## R-WORKFLOW-SUSPEND-REPLAY-DETERMINISM-01 — Suspend Is a Replay-Stable State

- **CLASSIFICATION:** runtime / engine / workflow / determinism
- **SEVERITY:** S2
- **DESCRIPTION:**
  - Replay of a stream ending in `WorkflowExecutionSuspendedEvent` lands at `Status = Suspended` and remains there until a `WorkflowExecutionResumedEvent` appears downstream. The suspend event carries no new replay-bearing fields beyond the canonical envelope — `StepName` + `Reason` are audit/observability data, identical in shape to the Cancelled event.
  - No new replay discriminator introduced. Two streams that land at the same end state (whether via `... Failed → Resumed → ... → Suspended` or `... Suspended → Resumed → Suspended` chain) are replay-equivalent.
- **CHECK:**
  - No new mutation in Apply beyond status transition. Status is the only aggregate field changed by `Apply(WorkflowExecutionSuspendedEvent)`.

## Out of scope for R3.A.3

- **Caller-initiated suspend surface** — no `WorkflowSuspendCommand` or step-yield API. Current suspend path requires a future caller (operator command, timer-trigger, external signal bridge). Composes on the event shape landed here.
- **Suspend timeout** (automatic cancellation of long-suspended workflows). Future R3 item — would compose on R3.A.1 execution timeout semantics.
- **Observability signal** (`workflow.execution.suspended` counter). The existing R3.A.1 execution-duration histogram only records at ExecuteAsync EXIT — a suspended workflow has not exited. A future observability pass would add a per-state-transition counter. Out of scope for R3.A.3 because the engine does not yet emit suspend events; the rule lands the event type, not the instrumentation.
