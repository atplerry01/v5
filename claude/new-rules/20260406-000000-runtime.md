## FINDING: Workflow State Persistence Observer Pattern

CLASSIFICATION: runtime / workflow-state / persistence
SOURCE: Audit sweep of WBSM v3.5 Workflow State Persistence implementation
DESCRIPTION: The observer pattern (IWorkflowStepObserver) allows runtime to persist workflow state during engine execution via shared contract callback. This is an intentional deviation from the strict "persist only post-execution" rule. Workflow recovery requires after-each-step persistence; deferring all persistence to post-execution would make mid-workflow failures unrecoverable.
PROPOSED_RULE: Runtime MAY persist workflow state during engine execution via shared contract observers when the persistence is required for workflow recovery. The observer MUST be a shared contract (not runtime import). The engine MUST NOT know about persistence — it only invokes the contract callback. The observer implementation MUST live in runtime.
SEVERITY: S2 (structural — intentional architectural decision with documented rationale)

CLASSIFICATION: runtime / workflow-state / persistence
SOURCE: Infrastructure audit of WorkflowStateRepository implementation
DESCRIPTION: WorkflowStateRepository uses ConcurrentDictionary as an in-memory store. This is a placeholder implementation. Production MUST use Postgres (migration provided in scripts/migrations/001_create_workflow_state.sql).
