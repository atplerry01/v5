# NEW RULE — IDEMPOTENCY MUST NOT MASK AUTH FAILURES

CLASSIFICATION: runtime / middleware / ordering
SOURCE: E2E lifecycle validation (todo update negative test), 2026-04-07
SEVERITY: S1

## DESCRIPTION
Idempotency checks MUST NOT execute before authentication and policy evaluation. When they do, unauthorized requests can be rejected with a misleading 400 ("Duplicate command detected") instead of 401, masking the true auth failure and producing false signals in audits.

Observed violation:
- Unauthorized `POST /api/todo/update` returned `400 Duplicate command detected`
- Expected: `401 Unauthorized`
- Idempotency middleware fired before any identity check

## PROPOSED_RULE
The runtime middleware pipeline MUST enforce this order:

    Tracing → Metrics → ContextGuard → Validation → Auth → Policy → Idempotency → ExecutionGuard → Execution

Idempotency MUST come AFTER identity validation and policy evaluation. The `RuntimeControlPlaneBuilder` should reject any pipeline that places `IdempotencyMiddleware` before `PolicyMiddleware`.

## SUSPECTED LOCATIONS
- src/platform/host/Program.cs (middleware pipeline construction — verify ordering)
- src/runtime/middleware/RuntimeControlPlaneBuilder (add ordering invariant check)
