# R3.A.2 — Workflow Step Retry Handling (2026-04-20)

Captured per CLAUDE.md $1c. Second R3 deliverable. Closes the §13 "Workflow retry handling" PARTIAL row. Composes on R3.A.1's observability (adds one new signal on the same meter) and uses the existing `WorkflowOptions` record.

**Why this matters:** pre-R3.A.2 the engine terminates on FIRST step failure. Transient step failures (DB connection blip, Kafka lag pulse) currently force the whole workflow to fail + re-drive through external orchestration. R3.A.2 adds bounded in-execution retry so transient failures self-heal without external intervention, while keeping timeouts and cancellations non-retryable (they are already bounded signals).

## R-WORKFLOW-STEP-RETRY-01 — Bounded Step Retry Inside ExecuteAsync

- **CLASSIFICATION:** runtime / engine / workflow / retry
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `T1MWorkflowEngine.ExecuteAsync` wraps each step invocation in a retry loop bounded by `WorkflowOptions.StepRetryMaxAttempts`. Value 1 means "one attempt, no retry" (preserves pre-R3.A.2 behaviour). Value N means one initial attempt + up to N-1 retries.
  - **Retryable failures:**
    - `stepResult.IsSuccess == false` — soft failure via return value.
    - Step throws any exception EXCEPT `OperationCanceledException` (caller cancel), `WorkflowTimeoutException` (already classified), or the linked-CTS timeout translations the engine produces.
  - **Non-retryable failures:**
    - Caller-driven `OperationCanceledException` — propagates as-is (host shutdown / request abort).
    - Per-step timeout (`stepCts.IsCancellationRequested`) — already bounded; retrying would double the effective deadline.
    - Execution-level timeout (`executionCts.IsCancellationRequested`) — total deadline exceeded; retry is meaningless.
  - **Backoff schedule:** `delay = min(StepRetryMaxBackoffMs, StepRetryBaseBackoffMs × 2^(attempt-1))`. No jitter — in-execution retries are scoped to one process; cross-process jitter belongs to the `.retry` Kafka tier (R2.A.3d). Delay sleeps under `executionCts.Token` so caller cancellation + `MaxExecutionMs` interrupt waiting retries.
  - **Per-attempt CTS refresh:** each attempt gets a fresh per-step linked CTS with `PerStepTimeoutMs`. A retry is a distinct attempt — it gets the full per-step window, not the remainder of the previous one. The execution-level CTS still caps cumulative work.
- **CHECK:**
  - `T1MWorkflowEngine.cs` contains a retry loop around the step dispatch with the described guards.
  - Unit-testable via a fake step that fails N-1 times then succeeds. Integration test deferred to R5 chaos framework.

## R-WORKFLOW-STEP-RETRY-NON-RETRYABLE-EXCLUSION-01 — Timeouts and Cancellations Never Retry

- **CLASSIFICATION:** runtime / engine / workflow / retry / boundary
- **SEVERITY:** S1
- **DESCRIPTION:**
  - Timeouts (per-step or execution) and caller-driven cancellation are EXPLICITLY non-retryable. Retrying them would either:
    - Double the effective deadline (per-step timeout → retry attempts a step that is already known to exceed the allowed window, burning more work toward the execution-level cap for no benefit), or
    - Ignore caller intent (caller cancelled — retrying after cancel is a contract violation).
  - The engine's existing `catch` filters already distinguish these cases. The retry loop only kicks in on the two RETRYABLE branches: soft failure (`!stepResult.IsSuccess`) and generic exception (non-cancel, non-timeout).
- **CHECK:**
  - The retry loop body does NOT wrap the `catch (OperationCanceledException)` timeout-translation filters.
  - Each of those `catch` clauses continues to `throw new WorkflowTimeoutException(...)` or re-throw the caller cancel unchanged.

## R-WORKFLOW-STEP-RETRY-OBSERVABILITY-01 — Retry-Attempts Counter + Per-Attempt Step-Duration Emissions

- **CLASSIFICATION:** runtime / engine / workflow / observability / retry
- **SEVERITY:** S2
- **DESCRIPTION:**
  - New `workflow.step.retry_attempts` counter (`Counter<long>`) on the existing `Whycespace.Workflow` meter. Tagged `{workflow_name, step_name}`. Incremented ONCE per retry (i.e., on each attempt AFTER the first). A step that succeeds on its first try contributes zero to this counter; a step that succeeds on attempt 3 contributes 2.
  - `workflow.step.duration` (shipped in R3.A.1) continues to emit ONCE PER ATTEMPT — each attempt records its own duration with the attempt's outcome (`success` / `failed` / `timeout_step` / `timeout_execution` / `cancelled`). Histogram sample count = total step attempts, not step completions.
  - `workflow.execution.duration` + `workflow.execution.completed` (R3.A.1) unchanged — one sample/increment per `ExecuteAsync` exit, regardless of how many retries occurred internally.
- **CHECK:**
  - `workflow.step.retry_attempts` counter is declared on the engine's meter and incremented exactly in the retry-loop branch (between attempts).
  - Architecture test pinned via signal-name source-scan.

## R-WORKFLOW-STEP-RETRY-REPLAY-DETERMINISM-01 — Retries Are NOT Audit Events

- **CLASSIFICATION:** runtime / engine / workflow / retry / determinism
- **SEVERITY:** S2
- **DESCRIPTION:**
  - In-execution step retries are observability events ONLY — they emit metrics but do NOT emit new domain lifecycle events. The event stream sees the final outcome: `WorkflowStepCompletedEvent` on success (regardless of attempt count) or `WorkflowExecutionFailedEvent` on exhaustion.
  - Replay consequences:
    - Replay of the event stream reproduces the final outcome regardless of how many attempts the live execution made. Attempt count is NOT a replay input.
    - A replay-rehydrated workflow that resumes via `WorkflowLifecycleEventFactory.Resumed` starts FRESH — the retry attempt counter resets to 1 for the step that previously failed.
  - This intentionally keeps in-execution retry out of the audit surface. Cross-process retry (the `.retry` Kafka tier, R2.A.3d) IS audit-visible because the message round-trips through Kafka with `retry-attempt-count` headers. The two tiers compose but do not conflate.
- **CHECK:**
  - No `WorkflowStepRetriedEvent` is emitted by `T1MWorkflowEngine` — a code scan confirms only the existing 4 lifecycle events (Started, StepCompleted, Completed, Failed, Resumed) surface.
  - Replay tests (if added) confirm that the rehydrated state from a stream with a failed-then-succeeded step matches the state produced by a single-attempt success.

## Out of scope for R3.A.2

- **Per-step retry policy overrides** (e.g. specific steps opt out of retry, or have their own attempt count). Deferred — current configuration is workflow-global. Per-step overrides via `WorkflowStepDefinition` would be R3.A.2.1.
- **Retry failure classification** (which exception types are transient vs terminal). Current rule: "anything that isn't timeout/cancel is retryable". Finer classification is R3.A.5 scope (exception-path handling).
- **`WorkflowStepRetriedEvent` domain event** — deferred. Audit-visible retry is the `.retry` Kafka tier's job.
- **Retry metric histograms for backoff delays** — the backoff formula is deterministic so operator dashboards can compute it; no new histogram needed.
