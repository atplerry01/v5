# R2.E.1 — Consumer Rebalance Safety (2026-04-19)

Captured per CLAUDE.md $1c. First R2.E deliverable — closes the largest remaining ABSENT cluster from the 2026-04-19 post-R2.A.D gap-matrix re-survey (§11 consumer rebalance safety + §12 partition ownership discipline + §12 safe consumer-group rebalance behavior).

## R-CONSUMER-REBALANCE-01 — Every Kafka Consumer Worker Registers Rebalance Handlers

- **CLASSIFICATION:** runtime / event-fabric / consumer
- **SEVERITY:** S1
- **DESCRIPTION:**
  - Every `IConsumer<TKey,TValue>` created via `ConsumerBuilder<TKey,TValue>` MUST register three rebalance handlers before `.Build()`:
    1. `SetPartitionsAssignedHandler` — emits `consumer.rebalance.assigned` counter tagged `{topic, partition_count}` + structured log line.
    2. `SetPartitionsRevokedHandler` — emits `consumer.rebalance.revoked` counter + structured log line BEFORE the partition is released, so in-flight per-message commits have flushed (our existing per-message `consumer.Commit(result)` pattern satisfies this by construction — no in-memory buffer across messages).
    3. `SetPartitionsLostHandler` — emits `consumer.rebalance.lost` counter (distinct from revoked — indicates session timeout, a red-flag signal).
  - Handler implementation lives in a shared `src/platform/host/adapters/KafkaRebalanceObservability.cs` helper so the 11 consumer worker files don't duplicate the wiring. Each worker calls `KafkaRebalanceObservability.Attach(builder, topic, workerName, logger)` during construction.
  - Metrics live on a new `Whycespace.Kafka.Consumer` meter (separate from `Whycespace.Projection.Consumer` because rebalance observability applies to ALL consumer workers, not just projection consumers).
- **CHECK:**
  - `grep -rn "new ConsumerBuilder<" src/platform/host/adapters/` should show every usage followed by a `KafkaRebalanceObservability.Attach(...)` call on the same builder. Architecture test pins this.
  - Three counters named `consumer.rebalance.{assigned,revoked,lost}` appear in any MeterListener scrape.

## R-CONSUMER-REBALANCE-COOPERATIVE-STICKY-01 — Cooperative Sticky Assignment Strategy

- **CLASSIFICATION:** runtime / event-fabric / consumer / config
- **SEVERITY:** S2
- **DESCRIPTION:**
  - Every `ConsumerConfig` MUST set `PartitionAssignmentStrategy = PartitionAssignmentStrategy.CooperativeSticky`. The default `Range` / `RoundRobin` strategies rebalance stop-the-world — all partitions are revoked and reassigned every rebalance event, which amplifies downtime during horizontal scale events.
  - Cooperative sticky reassigns only the delta; instances that remain in the consumer group keep their existing partitions across rebalances, cutting rebalance blast radius dramatically.
  - This flag flip is safe because our per-message `consumer.Commit(result)` pattern has no cross-message state — there is no batch-buffer that needs flushing differently between the two strategies.
- **CHECK:**
  - `grep -rn "PartitionAssignmentStrategy" src/` returns `= PartitionAssignmentStrategy.CooperativeSticky` at every `ConsumerConfig` construction site.
  - Architecture test pins the strategy across all consumer workers.

## R-CONSUMER-REBALANCE-OBSERVABILITY-HELPER-01 — Canonical Helper Type

- **CLASSIFICATION:** runtime / event-fabric / consumer / api
- **SEVERITY:** S3
- **DESCRIPTION:**
  - `KafkaRebalanceObservability.Attach(ConsumerBuilder<TKey,TValue> builder, string topic, string workerName, ILogger? logger)` is the single canonical attach point. Workers MUST use this helper — no ad-hoc inline rebalance handlers.
  - Helper is stateless (static class) and takes no runtime state other than `logger`. All observability is emitted via the shared `Whycespace.Kafka.Consumer` meter.
  - Workers that don't register their own rebalance handlers get the canonical ones for free by calling `Attach`. Future rebalance-protocol changes land in one file.
- **CHECK:**
  - `src/platform/host/adapters/KafkaRebalanceObservability.cs` exists and is referenced by every consumer worker.
