# R3.A.5 — Workflow Step Exception Classification (2026-04-20)

Captured per CLAUDE.md $1c. Third R3 deliverable (skipping R3.A.3 suspend + R3.A.4 cancellation-events — both require domain-event surface expansion and are larger than R3.A.5). Closes the §13 "Exception-path handling" PARTIAL row.

**Why this matters:** post-R3.A.2 the engine retries every non-cancel, non-timeout exception. This is conservative but wasteful for clearly-terminal failures — retrying a `ValidationFailed` step burns attempt budget without changing the outcome. Terminal-vs-transient classification lets the engine fast-fail on non-retryable causes while preserving the R3.A.2 backoff for genuinely transient failures.

**Key design choice:** reuse the existing R1 `RuntimeFailureCategory` taxonomy + R2 `RuntimeExceptionMapper` seam. No new enum, no new abstraction — the classification already exists; R3.A.5 adds the retry-decision mapping on top of it.

## R-WORKFLOW-STEP-EXCEPTION-CLASSIFICATION-01 — Classifier Routes via RuntimeExceptionMapper

- **CLASSIFICATION:** runtime / engine / workflow / retry / classification
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `WorkflowStepFailureClassifier.Classify(Exception)` returns a `WorkflowStepFailureClassification` enum: `Retryable` or `Terminal`.
  - Implementation:
    1. `var mapped = RuntimeExceptionMapper.Map(ex);` — reuse the R1 mapper to land on a `RuntimeFailureCategory`.
    2. Consult a compile-time `HashSet<RuntimeFailureCategory>` of **terminal** categories: `{ValidationFailed, AuthorizationDenied, PolicyDenied, PoisonMessage, RuntimeGuardRejection, InvalidState, Cancellation}`.
    3. If the mapped category is in the set → `Terminal`. Else → `Retryable`.
  - The "retryable by default" posture preserves the R3.A.2 behaviour for unknown / `ExecutionFailure` / `Unknown` — they continue to retry, matching the pre-R3.A.5 conservative default.
  - The classifier is a static pure function — no state, no DI, testable in isolation. Lives in `src/engines/T1M/core/workflow-engine/WorkflowStepFailureClassifier.cs`.
  - **Not extending** `RuntimeExceptionMapper` directly: the mapper's purpose is exception → category. Retry decision is a workflow-specific concern layered on top, not a universal one (e.g., command-dispatch retry semantics may differ).
- **CHECK:**
  - `WorkflowStepFailureClassifier.cs` exists.
  - The terminal-category set contains exactly the 7 values listed above; unit-testable via source-scan.

## R-WORKFLOW-STEP-EXCEPTION-ENGINE-WIRING-01 — Engine Consults Classifier Before Retrying

- **CLASSIFICATION:** runtime / engine / workflow / retry / integration
- **SEVERITY:** S2
- **DESCRIPTION:**
  - In the engine's `catch (Exception hardEx)` branch (R3.A.2 retry path), consult `WorkflowStepFailureClassifier.Classify(hardEx)` BEFORE deciding whether to retry.
  - If `Terminal`:
    - Record step duration with outcome `failed` (terminal failures still tag as `failed` in the existing 5-outcome vocabulary — no new outcome value introduced).
    - Increment the new `workflow.step.terminal_failures` counter tagged `{workflow_name, step_name, category}` where `category` is the `RuntimeFailureCategory` enum value string (low cardinality by construction — 14 enum values).
    - Record execution outcome, emit `Failed` lifecycle event with a classification-aware reason (e.g., `"Step 'X' failed terminally with category=ValidationFailed: ArgumentException: message"`).
    - Return immediately — do NOT retry, do NOT increment `workflow.step.retry_attempts`.
  - If `Retryable`: existing R3.A.2 retry logic applies unchanged.
- **CHECK:**
  - `T1MWorkflowEngine.cs` contains a `WorkflowStepFailureClassifier.Classify(` call inside the `catch (Exception hardEx)` branch.
  - Terminal path emits a `Failed` lifecycle event with category string in the reason.
  - Architecture test pins the classifier call site + the new counter emission site.

## R-WORKFLOW-STEP-EXCEPTION-COUNTER-01 — Terminal-Failures Counter

- **CLASSIFICATION:** runtime / engine / workflow / observability
- **SEVERITY:** S2
- **DESCRIPTION:**
  - New `workflow.step.terminal_failures` (Counter<long>) on the existing `Whycespace.Workflow` meter. Tagged `{workflow_name, step_name, category}`. Incremented exactly once per terminal-classified step failure.
  - **Why a separate counter rather than extending the outcome vocabulary**: the 5-outcome vocabulary is already shipped and consumed by operator dashboards. Adding `failed_terminal` would either break dashboards (new tag value) or dilute meaning (`failed` today means `failed`; tomorrow might mean "failed but retryable failed"). A separate counter keeps the duration-histogram outcome stable and gives operators a pure terminal-failure signal.
  - Counter-histogram divergence is intentional here: the histogram continues to count EVERY failure (transient + terminal); the terminal-failures counter counts only the terminal subset. Operators compute `terminal_rate = terminal_failures / step_duration_failed_count`.
- **CHECK:**
  - `T1MWorkflowEngine.cs` declares and emits the counter. Architecture test pins the signal name.

## R-WORKFLOW-STEP-EXCEPTION-REPLAY-DETERMINISM-01 — Terminal vs Retryable Does Not Affect Replay

- **CLASSIFICATION:** runtime / engine / workflow / determinism
- **SEVERITY:** S2
- **DESCRIPTION:**
  - The classification is a RUNTIME optimization — it affects whether the engine retries in-execution, but the final outcome event (`WorkflowStepCompletedEvent` on success, `WorkflowExecutionFailedEvent` on failure) is IDENTICAL regardless of whether the failure was classified terminal-or-retryable-then-exhausted.
  - Replay consequences: same as R3.A.2 — the event stream replays the final outcome. The attempt count, classification, and in-execution backoff schedule are all runtime-only state.
  - The `Failed` event's reason string carries classification info (`"failed terminally with category=X"` vs `"exhausted N attempt(s)"`) for operator clarity but this is a human-readable field, not a replay discriminator — two runs that land at the same failed-event state are replay-equivalent even if one emitted via terminal classification and the other via retry exhaustion.
- **CHECK:**
  - No new domain event types introduced by R3.A.5 (only observability metric).

## Out of scope for R3.A.5

- **Per-step classification override** — some workflows may want to treat `ValidationFailed` as retryable (e.g., the validation may refer to a transiently missing reference). Deferred — current classification is global. Per-step override via `WorkflowStepDefinition` would be R3.A.5.1.
- **Suspend on terminal failure** (human-approval wait-state) — R3.A.3 scope, requires domain-event surface expansion.
- **Classification on SOFT failures** (`stepResult.IsSuccess == false`) — soft failures don't carry an exception; their error string is opaque. Deferred unless a soft-failure classification contract lands.
