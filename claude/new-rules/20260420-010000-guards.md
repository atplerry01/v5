# R3.A.1 — Workflow Observability Closure (2026-04-20)

Captured per CLAUDE.md $1c. First R3 deliverable. Closes the §13 "Workflow observability" PARTIAL row from the post-Phase-B re-survey. Scope is intentionally narrow — instrumentation only, no behaviour change.

**Why this is the right first R3 slice:**
- R2 closed with circuit breakers, consumer lag, partition skew, and retry-tier observability shipped on their respective meters. Workflow runtime is the only remaining §13 subsystem without parallel duration / outcome metrics on its meter.
- The seam already exists (`Whycespace.Workflow` meter, declared in `WorkflowAdmissionGate`). R3.A.1 extends it — no new meter, no new composition wiring.
- All timing is observability-only — no values flow into audit events or replay-deterministic paths, so `Stopwatch.GetElapsedTime` is the idiomatic source (replay-determinism rule targets audit data, not instrumentation).

## R-WORKFLOW-OBSERVABILITY-01 — Execution + Step Duration Histograms

- **CLASSIFICATION:** runtime / engine / workflow / observability
- **SEVERITY:** S2
- **DESCRIPTION:**
  - Two histograms on the existing `Whycespace.Workflow` meter:
    - `workflow.execution.duration` (unit: `ms`) — recorded once per `T1MWorkflowEngine.ExecuteAsync` exit, on EVERY path (success, failed, timeout_step, timeout_execution, cancelled). Tags: `{workflow_name, outcome}`.
    - `workflow.step.duration` (unit: `ms`) — recorded once per step completion (success OR failure). Tags: `{workflow_name, step_name, outcome}` where `outcome ∈ {success, failed, timeout, cancelled}`.
  - Both are `Histogram<double>` — floating-point ms so sub-millisecond steps (the majority of domain step work) don't round to zero.
  - Start markers use `Stopwatch.GetTimestamp()`; elapsed via `Stopwatch.GetElapsedTime(startTicks).TotalMilliseconds`. Monotonic, unaffected by wall-clock skew, NOT subject to the `IClock` replay-determinism rule because observability durations never flow into audit events or replay decisions.
- **CHECK:**
  - `T1MWorkflowEngine.cs` emits both histograms. Architecture test verifies the histograms exist on the `Whycespace.Workflow` meter (via source scan of `Meter.CreateHistogram<double>("workflow.execution.duration"`).
  - All 5 outcome paths tagged: success, failed, timeout_step, timeout_execution, cancelled. Code inspection confirms each `throw`/`return`/normal-exit path records an outcome before leaving the method.

## R-WORKFLOW-OBSERVABILITY-COMPLETION-COUNTER-01 — Execution Outcome Counter

- **CLASSIFICATION:** runtime / engine / workflow / observability
- **SEVERITY:** S2
- **DESCRIPTION:**
  - `workflow.execution.completed` counter (`Counter<long>`) on the same `Whycespace.Workflow` meter, incremented once per `ExecuteAsync` exit. Tags: `{workflow_name, outcome}`. Complements the duration histogram so operators can compute per-outcome rates (successes/minute, failure rate) without deriving from the histogram sample count.
  - Outcome vocabulary: `success`, `failed`, `timeout_step`, `timeout_execution`, `cancelled` (caller-driven cancel). Exactly one increment per ExecuteAsync return.
- **CHECK:**
  - `workflow.execution.completed` counter is present. Same 5 outcome tags as the duration histogram.

## R-WORKFLOW-OBSERVABILITY-DETERMINISM-NOTE-01 — Stopwatch is Permitted for Observability Timing

- **CLASSIFICATION:** runtime / engine / workflow / observability / scope
- **SEVERITY:** S3
- **DESCRIPTION:**
  - The replay-determinism discipline (`R-RETRY-DELIVER-AFTER-DETERMINISM-01` and the wider IClock rule) targets **audit-bearing** timestamps — values that flow into domain events, retry-headers, idempotency keys, or any replay-input. Observability instrumentation (duration histograms, latency counters) is explicitly out of scope: metrics are derivable from runtime behaviour and never influence replay outcomes.
  - `Stopwatch.GetTimestamp` / `GetElapsedTime` is the canonical source for observability timing because it is monotonic and unaffected by wall-clock skew. Using `IClock` here would conflate two orthogonal concerns.
  - This rule makes the exemption explicit so a future audit sweep doesn't flag Stopwatch usage in the engine as a determinism violation.
- **CHECK:**
  - `T1MWorkflowEngine.cs` contains `Stopwatch.GetTimestamp()` and `Stopwatch.GetElapsedTime(...)` calls scoped to the duration histograms only.
  - No `DateTime.UtcNow`, `DateTimeOffset.UtcNow`, or `System.Random` appears anywhere in the engine.

- **Workflow-level lag** (how stale is a workflow's progress?) — deferred; the consumer-fabric lag histograms already give a proxy for event-driven workflows.
