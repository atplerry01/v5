# R2.E.2 — Consumer Lag Awareness (2026-04-19)

Captured per CLAUDE.md $1c. Second R2.E deliverable — builds on the R2.E.1 rebalance observability seam. Closes the §11 "Consumer lag awareness" row from the 2026-04-19 post-R2.A.D gap-matrix re-survey. Lag is the canonical "is my consumer falling behind?" signal; without it, operators can't distinguish a healthy slow topic from a falling-behind consumer and can't size horizontal scale-out decisions.

## R-CONSUMER-LAG-01 — Per-Message Lag Sampling After Every Successful Consume

- **CLASSIFICATION:** runtime / event-fabric / consumer / observability
- **SEVERITY:** S2
- **DESCRIPTION:**
  - Every consumer worker MUST call `KafkaLagObservability.Record(consumer, consumeResult, workerName, topicTag)` after each successful `Consume()` that returned a non-null `ConsumeResult<TKey,TValue>`.
  - The helper queries `consumer.GetWatermarkOffsets(consumeResult.TopicPartition)` — a fast in-process read of the cached high-watermark from the last FetchResponse, NOT a broker round-trip — and emits the offset delta as `consumer.lag_messages` histogram on the shared `Whycespace.Kafka.Consumer` meter. Tagged low-cardinality `{topic, worker, partition}` (partition included because standard Kafka-exporter vocabulary uses it; partition counts are typically 3-12, acceptable cardinality).
  - Lag = `HighWatermark - ConsumedOffset - 1`. The `-1` is canonical: when we consume offset N, the high-watermark is at-least N+1, so lag after consuming the last available message is zero.
  - Empty-poll returns (`result is null`) do NOT emit lag — there is no offset to measure against. The rebalance observability seam already covers the "consumer is alive but idle" case.
  - Poison-routed and DLQ-committed messages DO emit lag before the commit — the lag signal reflects "we saw this offset; how far behind are we?" which is meaningful regardless of routing outcome.
- **CHECK:**
  - `grep -rn "result = consumer.Consume(" src/platform/host/adapters/` shows every hit followed by a `KafkaLagObservability.Record(` call on the same result. Architecture test pins this.
  - Histogram `consumer.lag_messages` appears on the `Whycespace.Kafka.Consumer` meter alongside the R2.E.1 `consumer.rebalance.*` counters.

## R-CONSUMER-LAG-HELPER-01 — Canonical Helper Type

- **CLASSIFICATION:** runtime / event-fabric / consumer / api
- **SEVERITY:** S3
- **DESCRIPTION:**
  - `KafkaLagObservability.Record<TKey,TValue>(IConsumer<TKey,TValue> consumer, ConsumeResult<TKey,TValue> result, string workerName, string topicTag)` is the single canonical attach point for lag observability. Stateless static class at `src/platform/host/adapters/KafkaLagObservability.cs`. Lives in the same `Whycespace.Platform.Host.Adapters` namespace as `KafkaRebalanceObservability` so both observability seams have parallel shape.
  - Helper swallows `KafkaException` from `GetWatermarkOffsets` — it is a best-effort read and MUST NOT disrupt the consume loop if the watermark cache is transiently unavailable (e.g. immediately after a rebalance assigned event before the first fetch response). A failure emits a separate `consumer.lag_unknown` counter so the operator can distinguish "zero lag" from "couldn't read lag."
  - Helper takes `topicTag` as a separate parameter (not derived from `result.Topic`) so multi-topic workers can report under a stable canonical identity (typically the first topic) matching the R2.E.1 convention.
- **CHECK:**
  - `src/platform/host/adapters/KafkaLagObservability.cs` exists.
  - Method signature matches the R2.E.1 `KafkaRebalanceObservability.Attach` shape (same workerName / topicTag convention).

## R2E2-ALL-WORKERS-WIRED-01 — Every Worker Wired

- **CLASSIFICATION:** runtime / event-fabric / consumer
- **SEVERITY:** S2
- **DESCRIPTION:**
  - All 11 consumer workers (the same set wired in R2.E.1 for rebalance observability) MUST call `KafkaLagObservability.Record` after each successful `Consume()`. No exceptions — lag is a universal observability requirement, not a per-worker optional signal.
- **CHECK:**
  - Architecture test `Every_Consumer_Consume_is_followed_by_KafkaLagObservability_Record` passes. Same scan pattern as R2.E.1's `Every_ConsumerBuilder_Build_is_preceded_by_KafkaRebalanceObservability_Attach` but targets the consume/record pair.
