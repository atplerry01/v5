## FINDING 1 — Integration test project does not compile (S0)

### Description

[tests/integration/operational-system/sandbox/todo/TodoPipelineTests.cs](../../tests/integration/operational-system/sandbox/todo/TodoPipelineTests.cs)
references a `RuntimeCommandDispatcher` constructor signature that no
longer exists. The test instantiates the dispatcher as:

```csharp
new RuntimeCommandDispatcher(middlewares, registry, services, _eventStore, _chainAnchor, _outbox)
```

at lines 26-46 (and again in the `TodoE2EPipelineTests` class), but the
real type at
[src/runtime/dispatcher/RuntimeCommandDispatcher.cs:28-44](../../src/runtime/dispatcher/RuntimeCommandDispatcher.cs#L28-L44)
declares:

```csharp
public RuntimeCommandDispatcher(
    IEngineRegistry engineRegistry,
    IServiceProvider serviceProvider,
    IEventStore eventStore,
    IWorkflowEngine workflowEngine,
    IWorkflowRegistry workflowRegistry,
    IWorkflowStateRepository workflowStateRepository,
    IClock clock)
```

`dotnet build tests/integration/Whycespace.Tests.Integration.csproj`
fails with `CS0246: The type or namespace name 'RuntimeCommandDispatcher'
could not be found`. There is no test-local shim and no global using that
would resolve the symbol — the integration test project is **red** and has
been since whatever refactor changed the dispatcher signature.

### Impact

- Phase 2 cannot be witnessed at runtime. The static audits
  (`runtime-order.audit.md`, `replay-determinism.audit.md`) produce a PASS
  by inspection, but no integration test exercises the locked order or the
  replay equality.
- Any new test added to this project will compile-fail alongside the
  existing one — `dotnet test` cannot even reach test discovery.
- The "TodoE2EPipelineTests" class in the same file (the canonical "full
  flow" example) is also broken and has been silently broken in the repo.
  CI presumably is not running this project, or is, and its red status is
  being ignored.

### Proposed action

This finding should become a **dedicated repair prompt**, scoped to:

1. Rewrite `TodoPipelineTests.cs` against the real `RuntimeCommandDispatcher`
   constructor. This requires test doubles for `IWorkflowEngine`,
   `IWorkflowRegistry`, and `IWorkflowStateRepository` (none currently
   exist in the file).
2. Decide whether the test should drive the dispatcher directly (the
   current pattern) or drive the full `RuntimeControlPlaneBuilder` pipeline
   (the more useful pattern, and what the Phase 2 validation actually
   needs).
3. Retrofit `InMemoryChainAnchor` (line 318) and any other doubles that
   currently use `Guid.NewGuid()` / `DateTimeOffset.UtcNow` to take
   `IClock` and `IIdGenerator`. This is a precondition for the replay
   determinism test.
4. Add a CI gate that fails if `tests/integration/` does not build, so
   this regression cannot recur silently.

### Severity

**S0 — CRITICAL.** A broken test suite that has been broken long enough
to be normalized is one of the highest-impact forms of test rot. Every day
the project is red is a day no integration regression can be caught.

## FINDING 2 — Phase 2 Validation prompt's literal hash-equality assertion conflicts with EventReplayService design (S2)

### Description

The Phase 2 Validation prompt (TG2.3 STEP 3) asserts:

```
ExecutionHashes_replay == ExecutionHashes_original
DecisionHashes_replay == DecisionHashes_original
```

The current
[EventReplayService.cs:57-59](../../src/runtime/event-fabric/EventReplayService.cs#L57-L59)
deliberately overwrites these fields with sentinel values during replay:

```csharp
ExecutionHash = "replay",
PolicyHash = "replay",
Timestamp = DateTimeOffset.MinValue,
```

The sentinel exists by design to distinguish a projection-rebuild envelope
from a fresh-execution envelope downstream. Taking the prompt literally
would require deleting this sentinel and reconstructing or loading the
original envelope hashes from the event store on replay. That is a
**design change**, not a fix.

### Resolution applied in this sweep

The `replay-determinism.audit.md` separates two notions of replay:

- **Type A — Re-execution.** Run the same commands twice through the full
  pipeline with deterministic clock and id generator. This is the property
  that should be tested. PASS by static inspection.
- **Type B — Projection rebuild.** Use `EventReplayService` to load events
  from the store and rehydrate projections. Envelope hash fields are
  sentinel values by design.

The audit treats the prompt's assertion as the spirit of Type A, not the
letter of Type B. This interpretation should be confirmed with the prompt
author.

### Proposed action

Either:

1. **Confirm the spirit interpretation** (recommended). Update any future
   prompt that talks about replay to distinguish "re-execution" from
   "projection rebuild." Document the sentinel pattern in
   `replay-determinism.audit.md` (already done).
2. **Or change the design** to load original envelope fields from the
   event store on rebuild. This is non-trivial and would change the
   semantics of `Whyce.Runtime.EventFabric.EventEnvelope` for downstream
   consumers. Not recommended without a strong driving requirement.

### Severity

**S2 — MEDIUM.** Documentation/interpretation gap, not a code defect.

## FINDING 3 — Phase 2 Validation prompt's "canonical 7-step order" is a lossy subset of the real 11-stage order (S2)

### Description

The Phase 2 Validation prompt (TG1.1) defines the canonical execution
order as 7 steps:

```
1. Guard (pre)
2. Policy evaluation
3. Guard (post-policy)
4. Execution (engine/workflow)
5. Persist events (EventStore)
6. Chain anchoring (WhyceChain)
7. Kafka publish (Outbox)
```

The actual locked order in the codebase is **8 middlewares + 3 fabric
stages = 11 stages**, encoded structurally in
[RuntimeControlPlaneBuilder.cs:98-115](../../src/runtime/control-plane/RuntimeControlPlaneBuilder.cs#L98-L115)
and [EventFabric.cs:87-93](../../src/runtime/event-fabric/EventFabric.cs#L87-L93):

```
1. TracingMiddleware
2. MetricsMiddleware
3. ContextGuardMiddleware
4. ValidationMiddleware
5. PolicyMiddleware
6. AuthorizationGuardMiddleware
7. IdempotencyMiddleware
8. ExecutionGuardMiddleware
→ engine
9. EventStoreService.AppendAsync
10. ChainAnchorService.AnchorAsync
11. OutboxService.EnqueueAsync
```

The prompt's 7-step order is *compatible* with the real 11-stage order
(the 7 are a labeling of the load-bearing subset), but writing the prompt's
version into a canonical guard or audit would be **drift** under
[CLAUDE.md $5](../../CLAUDE.md). Specifically, it would lose:

- The Tracing/Metrics observability wrapping (positions 1, 2)
- The split between ContextGuard and Validation in pre-policy (positions 3, 4)
- The split between AuthorizationGuard and Idempotency in post-policy (positions 6, 7)
- The ExecutionGuard final gate (position 8)

### Resolution applied in this sweep

`runtime-order.audit.md` and `runtime-order.guard.md` codify the **real**
11-stage order, with the prompt's 7-step labels included as commentary.
This is an anti-drift call: canonicalize what is actually in the code, not
what an external doc claims.

### Proposed action

Update future Phase 2 / Phase 3 prompts to reference the 11-stage order
when describing runtime invariants, and treat the 7-step version as a
human-readable summary only, not as a normative spec.

### Severity

**S2 — MEDIUM.** Documentation drift risk; prevented in this sweep but
worth addressing at the prompt-author level.
