# GUARD-LAYER-MODEL-01 — Canonical 4-Layer Guard Model (LOCKED)

**CLASSIFICATION:** guards
**SOURCE:** §0 Final Lock governance decision, 2026-04-14. Supersedes any prior 5-layer / 6-layer enumeration (quality + testing as separate top-level tiers).
**SEVERITY:** S1 (architectural)

## DESCRIPTION

The Whycespace guard system operates on a **4-layer canonical model**. Quality enforcement and Test/E2E validation are **subsystems of Runtime Enforcement**, not independent guard tiers. This rule locks the canonical layer count and assignment after the 2026-04-14 consolidation.

## PROPOSED_RULE

**GUARD-LAYER-MODEL-01 — 4-Layer Canonical Model**

1. The guard system has exactly **four** canonical layers, realised as four files directly under `claude/guards/`:
   - `constitutional.guard.md` — WHYCEPOLICY authority, determinism, identifier/hash/replay primitives.
   - `runtime.guard.md` — execution pipeline/ordering, engine/projection/prompt-container, dependency graph & layer boundaries, contracts boundary, **code quality enforcement**, **test & E2E validation**.
   - `domain.guard.md` — business truth: purity, structure, classification-suffix, DTO naming, behavioral, structural, and inlined domain-aligned (economic/governance/identity/observability/workflow).
   - `infrastructure.guard.md` — platform, systems, kafka, config safety, composition loader, program composition.

2. **Quality and Testing are subsystems of Runtime Enforcement** — they are not, and MUST NOT become, separate top-level guards.
   - Code quality (`clean-code`, `no-dead-code`, `stub-detection`) lives in `runtime.guard.md` §Code Quality Enforcement.
   - Test validation (`tests`, `e2e-validation`) lives in `runtime.guard.md` §Test & E2E Validation.

3. **FORBIDDEN:**
   - Any fifth top-level guard file (e.g., `quality.guard.md`, `testing.guard.md`, `security.guard.md`). Creating one is S1 architectural drift.
   - Any subdirectory under `claude/guards/` (including a re-introduction of `domain-aligned/`). Domain-aligned guards live as subsections of `domain.guard.md`.
   - Any documentation that refers to a "5-layer" or "6-layer" guard model. References MUST be updated to "4-layer canonical model."

4. **Placement doctrine (non-negotiable, precedent-setting):**
   - `dependency-graph`, `contracts-boundary` → Runtime (enforcement, NOT Domain).
   - `policy`, `policy-binding` → Constitutional (WHYCEPOLICY authority, NOT Runtime).
   - `clean-code`, `no-dead-code`, `stub-detection`, `tests`, `e2e-validation` → Runtime (enforcement/validation).
   - `economic`, `governance`, `identity`, `observability`, `workflow` → Domain (inlined as subsections, NOT separate files).
   - `composition-loader`, `program-composition` → Infrastructure (host-process assembly).

5. **Pre-execution guard loading per CLAUDE.md §1a** MUST load exactly these four files. Any stale fragmented guard discovered in the tree is evidence of drift and MUST be deleted and captured under `claude/new-rules/` per §1c.

## ENFORCEMENT

- Post-execution audit sweep MUST fail if `claude/guards/` contains anything other than the four canonical files (plus optional `README.md`).
- Any PR that creates a fifth top-level guard file, reintroduces subdirectories, or duplicates the WBSM GE-01..GE-05 block more than once within a single canonical file is auto-blocked at S1.
- Any documentation (CLAUDE.md, READMEs, audit definitions) referencing a 5/6-layer model or deleted guard filenames is drift and MUST be reconciled to this rule.

## RELATIONSHIP TO PRIOR RULES

- Extends and supersedes `GUARD-CANONICAL-STRUCTURE-01` (filed 2026-04-14 205905). That rule established the 4-file tree; this rule locks the layer *doctrine* and clarifies that Quality/Testing are Runtime subsystems (closing the ambiguity that produced the transient `quality.guard.md` / `testing.guard.md` strays during consolidation).
