# R2.A.D.3a — Chain Adapter Refactor (2026-04-19, post-compact)

Captured per CLAUDE.md $1c. R2.A.D.1 shipped `ICircuitBreaker`; R2.A.D.2 refactored OpaPolicyEvaluator (first consumer); R2.A.D.4 shipped the registry + aggregator integration. R2.A.D.3a does the same refactor for `WhyceChainPostgresAdapter` (second consumer) so the chain breaker flows through the registry alongside OPA, eliminating the last concrete-adapter reference in `RuntimeStateAggregator` for breakers.

## R-CHAIN-BREAKER-DELEGATION-01 — WhyceChainPostgresAdapter consumes ICircuitBreaker

- **CLASSIFICATION:** runtime / resilience / composition
- **SEVERITY:** S1
- **DESCRIPTION:**
  - `WhyceChainPostgresAdapter` constructor accepts an injected `ICircuitBreaker` parameter. The adapter does NOT own breaker state — it delegates to the contract.
  - Inline breaker state REMOVED: `_breakerLock`, `_consecutiveFailures`, `_openedAt` fields, `IsBreakerOpenInternal`, `RecordSuccess`, `RecordFailure` private methods. Architecture guard confirms these tokens are absent post-refactor (comment-stripped scan).
  - The DB transaction inside `AnchorAsync` is wrapped in `_breaker.ExecuteAsync<ChainBlock>(...)`. The lambda body preserves the existing transport-failure throw sites — the breaker counts these as failures.
  - `CircuitBreakerOpenException` thrown from `_breaker.ExecuteAsync` is CAUGHT at the adapter boundary and TRANSLATED to `ChainAnchorUnavailableException(reason: "breaker_open", retryAfterSeconds: breakerEx.RetryAfterSeconds, ...)` preserving the pre-refactor external contract.
  - The public `IsBreakerOpen` property is retained with adapted semantics: returns `_breaker.State != CircuitBreakerState.Closed`. Backward-compat for any remaining consumers — though R2.A.D.4 already moved the `RuntimeStateAggregator` off this getter onto the registry.
- **CHECK:**
  - Architecture test extends the existing pattern — `WhyceChainPostgresAdapter_Delegates_Breaker_State_To_ICircuitBreaker` mirrors the OPA arch test: comment-stripped grep confirms `_breakerLock` / `_consecutiveFailures` / `_openedAt` gone; `ICircuitBreaker` reference present; `catch (CircuitBreakerOpenException` translation present.
  - Host composition registers the chain breaker as a plain `ICircuitBreaker` under name `"chain-anchor"`. `ChainInfrastructureModule` consumes it via `ICircuitBreakerRegistry.Get("chain-anchor")`.
  - `RuntimeStateAggregator.CanonicalBreakerReasons` gains `"chain-anchor"` → `"chain_anchor_breaker_open"` entry. Concrete `_chainAdapter.IsBreakerOpen` check removed — the registry iteration now covers it.

## R-CHAIN-BREAKER-BOUNDARY-01 — CircuitBreakerOpenException does NOT leak from WhyceChainPostgresAdapter

- **CLASSIFICATION:** runtime / resilience / boundary
- **SEVERITY:** S1
- **DESCRIPTION:**
  - `WhyceChainPostgresAdapter.AnchorAsync` MUST NOT propagate `CircuitBreakerOpenException` to callers. The typed exception used by the chain adapter is `ChainAnchorUnavailableException` — existing callers (`ChainAnchorService`, API-edge exception handler) depend on that contract.
  - The translation happens at the adapter boundary. `CircuitBreakerOpenException.RetryAfterSeconds` is copied onto the new `ChainAnchorUnavailableException.RetryAfterSeconds` so Retry-After guidance flows through unchanged.
  - Caller-driven cancellation (`OperationCanceledException` with caller's CT) propagates unchanged — not a chain-store health signal per the existing invariant.
- **CHECK:**
  - Grep for `throw.*CircuitBreakerOpenException` inside `WhyceChainPostgresAdapter.cs` returns zero hits (breaker instance throws; adapter catches + translates; never throws it).
  - Existing chain-anchor tests continue to pass — the external contract is unchanged.
