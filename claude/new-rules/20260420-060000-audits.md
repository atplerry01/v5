## FAIL-2 detail

**Rule:** `Engines_do_not_call_Resume_on_workflow_aggregate_directly` architecture test in `tests/unit/architecture/WbsmArchitectureTests.cs`.

**Previous regex:** `\b(workflowExecution|aggregate)\.Resume\s*\(` (case-insensitive, applied to `src/engines/**`).

**False-positive sites:**
- `src/engines/T2E/economic/enforcement/lock/ResumeSystemLockHandler.cs:22` — `aggregate.Resume(new Timestamp(cmd.ResumedAt));` where `aggregate` is a `LockAggregate` with its own domain-level `Resume(Timestamp)` method.
- `src/engines/T2E/economic/enforcement/restriction/ResumeRestrictionHandler.cs:33` — analogous pattern on `RestrictionAggregate`.

**Invariant the test should guard:** T1M workflow engine MUST NOT call `.Resume()` on a workflow-execution aggregate directly (post-phase1.6-S1.2 E-LIFECYCLE-FACTORY-CALL-SITE-01). T2E domain handlers resuming their own aggregates are OUT OF SCOPE — those are legitimate domain commands.

**Fix applied:** Scan path narrowed from `src/engines/**` to `src/engines/T1M/**`. The regex is preserved (variable-name based), but the directory scope now matches the invariant's actual target.

**Code status pre-fix:** T1M engine code was ALREADY clean — the previous phase1.6-S1.2 refactor removed the `WorkflowExecutionReplayService.cs:111 aggregate.Resume()` violation. The test was FAILING against T2E false-positives, not against a T1M regression. The fix brings the TEST in line with reality; it does not relax any code invariant.

**Post-fix test result:** PASS (1 of 1 test, 24ms).
