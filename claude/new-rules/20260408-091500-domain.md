## DESCRIPTION

Two latent correctness issues found in projection handlers during H9 discovery,
deferred from the H7-H9 SAFE commit per user ruling 2026-04-08.

### Finding 1 — WorkflowExecutionProjectionHandler not replay-safe

File: src/projections/orchestration-system/workflow/handler/WorkflowExecutionProjectionHandler.cs

The handler throws `InvalidOperationException` when an existing read-model row
is missing for `WorkflowStepCompletedEventSchema`,
`WorkflowExecutionCompletedEventSchema`, and `WorkflowExecutionFailedEventSchema`
(lines 60-61, 78-79, 90-91).

If the started-event is dropped, arrives out of order, or the projection is
rebuilt from a partial offset, every subsequent event for that workflow will
crash the consumer. There is no upsert-on-missing fallback and no buffering
of out-of-order events.

Linked to H8a (event-store concurrency hardening) because both share the
"replay/ordering must be guaranteed" assumption that is currently not enforced.

### Finding 2 — In-place dictionary mutation in workflow handler

File: src/projections/orchestration-system/workflow/handler/WorkflowExecutionProjectionHandler.cs:66

`existing.StepOutputs[e.StepName] = e.Output` mutates the StepOutputs
dictionary on a record returned from `IWorkflowExecutionProjectionStore.GetAsync`.
If the store implementation caches references (e.g. an in-memory test store,
or any future caching layer), this mutation leaks across calls.

The `with` expression on the next line cannot fix this because dictionaries
are reference types — `with` shallow-copies the record, not its members.

The comment in the source acknowledges the workaround. It is a latent bug
waiting for a caching projection store implementation to surface it.

## PROPOSED_RULE

### For Finding 1
Projection handlers MUST be replay-safe. Specifically:
  - Lifecycle handlers MUST upsert on missing rows (not throw)
  - OR a documented out-of-order buffering mechanism MUST exist

This should become a projection.guard.md addendum.

### For Finding 2
Projection handlers MUST NOT mutate state returned from a projection store
in place. Either:
  - The store contract guarantees a fresh-copy return, OR
  - Handlers must construct a new collection before mutation.

This should become a projection.guard.md addendum.

## SEVERITY

S2 — both findings are latent. Currently masked because (a) the in-memory test
store does not cache references and (b) production message ordering has not yet
exercised the missing-started-event path.
