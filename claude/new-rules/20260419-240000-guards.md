# R2.A.3d — Kafka `.retry` Tier Infrastructure (2026-04-19)

Captured per CLAUDE.md $1c. Last R2.A subsystem closure item. Closes the §11 "Retry routing" row (currently PARTIAL — `TopicNameResolver.ResolveRetry` scaffolded in R2.A.3a but no producer/consumer flow). Locked by D2 (tiered `.events → .retry → .deadletter`).

**Why this matters:** without `.retry`, every transient handler failure either (a) gets retried by Kafka consumer redelivery in tight loop — which amplifies load on the failing dependency — or (b) goes straight to `.deadletter` after first failure, losing the transient-vs-permanent distinction. The `.retry` tier adds durable delay + bounded attempt count with exponential backoff, so transient failures self-heal without hot-looping.

## R-RETRY-TIER-01 — Four-Tier Shape Is Enforced in Routing Code

- **CLASSIFICATION:** runtime / event-fabric / retry
- **SEVERITY:** S1
- **DESCRIPTION:**
  - The D2-locked four-tier shape `{topic}.commands`, `{topic}.events`, `{topic}.retry`, `{topic}.deadletter` is enforced by THREE canonical code paths:
    1. `TopicNameResolver.ResolveRetry(topic)` — returns `.retry` name for an `.events` topic (R2.A.3a, already shipped).
    2. `TopicNameResolver.ResolveDeadLetter(topic)` — returns `.deadletter` name, including from `.retry` source (already shipped; R2.A.3a "retry → deadletter is a canonical tier transition").
    3. `KafkaRetryEscalator.EscalateAsync` (NEW) — the single seam through which handler failures transition from `.events` to `.retry` or from `.retry` to `.deadletter` based on attempt-count vs max-attempts.
  - Consumer error paths MUST use `KafkaRetryEscalator` — they MUST NOT inline publish to `.deadletter` directly for handler-throws (poison-message DLQ routing for header violations / schema mismatches stays direct since those are never retryable).
- **CHECK:**
  - Grep for `deadletterProducer.ProduceAsync` or inline `.deadletter` publishes in consumer error paths. Post-R2.A.3d integration, handler-throw paths route through `KafkaRetryEscalator.EscalateAsync` instead of direct `.deadletter`. Poison-message paths (absent header / bad JSON) continue direct because they are never retryable.

## R-RETRY-HEADERS-01 — Canonical Retry Header Contract

- **CLASSIFICATION:** runtime / event-fabric / retry / protocol
- **SEVERITY:** S1
- **DESCRIPTION:**
  - Four headers on every message in a `.retry` topic:
    - `retry-attempt-count` (UTF-8 encoded integer, 1-based) — counts the failed attempt that produced this retry-tier message. A message republished once carries `1`, twice `2`, etc.
    - `retry-max-attempts` (UTF-8 encoded integer) — carried so the retry consumer does not need separate configuration to know the ceiling. Default value is the composition-time `Kafka:Retry:MaxAttempts` but the header wins per-message. Allows per-event-type overrides in future R2.A.3d.1.
    - `retry-deliver-after-unix-ms` (UTF-8 encoded long) — the Unix-epoch-milliseconds at which the retry consumer SHOULD re-publish to the source `.events` topic. Computed as `IClock.UtcNow + ExponentialBackoff(attempt, baseSeconds, maxSeconds, jitter)` at escalation time.
    - `retry-source-topic` (UTF-8 encoded string) — the canonical `.events` topic name to re-publish to when deliver-after arrives. Carried because the retry consumer subscribes to multiple `.retry` topics (pattern subscription) and needs to know where to put the message back.
  - Original message headers (`event-id`, `aggregate-id`, `event-type`, `correlation-id`, `causation-id`) are preserved unchanged. The retry tier is a transport mechanism, not a re-envelope step.
  - Header name constants live in `src/platform/host/adapters/RetryHeaders.cs` — single source of truth.
- **CHECK:**
  - `RetryHeaders.cs` exists with four `public const string` fields.
  - `KafkaRetryEscalator.EscalateAsync` writes exactly these four headers on every `.retry` publish.

## R-RETRY-ESCALATOR-01 — Canonical Escalator Helper

- **CLASSIFICATION:** runtime / event-fabric / retry / api
- **SEVERITY:** S1
- **DESCRIPTION:**
  - `KafkaRetryEscalator.EscalateAsync(producer, topicNameResolver, sourceTopic, originalMessage, failureReason, attemptCount, maxAttempts, baseBackoff, maxBackoff, randomProvider, clock, kafkaBreaker?, ct)` is the single canonical seam. Returns a `RetryEscalationOutcome` record: `{ Tier: "retry" | "deadletter", NextAttemptCount, DeliverAfterUtc, TargetTopic }`.
  - Routing logic:
    - If `attemptCount + 1 > maxAttempts` → publish to `.deadletter` topic (via `topicNameResolver.ResolveDeadLetter(sourceTopic)`) with a `dlq-reason` header encoding the exhaustion. Return outcome `{Tier: "deadletter", ...}`.
    - Else → publish to `.retry` topic (via `topicNameResolver.ResolveRetry(sourceTopic)`) with the four R-RETRY-HEADERS-01 headers. Return outcome `{Tier: "retry", NextAttemptCount: attemptCount + 1, ...}`.
  - Backoff formula: `delaySeconds = min(maxBackoff, baseBackoff × 2^(attempt-1)) × (1 + jitter × 0.25)` where `jitter ∈ [0, 1)` is sourced from `randomProvider.Next(seed)` using the deterministic seed `$"{sourceTopic}:{eventId}:retry:{attemptCount}"`.
  - `ProduceAsync` wrapped in the shared `"kafka-producer"` breaker (R-KAFKA-BREAKER-01 scope extends to cover this fifth site). `CircuitBreakerOpenException` from Open breaker re-thrown — the caller (consumer error path) MUST decide what to do (typical answer: do NOT commit the source offset so Kafka redelivers when the broker recovers).
- **CHECK:**
  - `KafkaRetryEscalator.cs` exists.
  - Unit tests pin: deliver-after is replay-deterministic for the same `(sourceTopic, eventId, attempt)` input; exhaustion routes to DLQ; `CircuitBreakerOpenException` propagates unchanged.
  - Architecture test adds `KafkaRetryEscalator.cs` to the R-KAFKA-BREAKER-01 sanctioned `ProduceAsync` sites list.

## R-RETRY-CONSUMER-WORKER-01 — Dedicated Retry Consumer Worker

- **CLASSIFICATION:** runtime / event-fabric / retry / worker
- **SEVERITY:** S1
- **DESCRIPTION:**
  - `KafkaRetryConsumerWorker` (an `IHostedService`) subscribes to the union of canonical `.retry` topics declared in `KafkaCanonicalTopics.All.Where(t => t.EndsWith(".retry"))`. Single consumer instance with a dedicated consumer group `whyce.retry-consumer` (distinct from projection / saga / integration groups).
  - Per-message flow:
    1. Read `retry-attempt-count`, `retry-max-attempts`, `retry-deliver-after-unix-ms`, `retry-source-topic` headers. Any missing / unparseable → route to `.deadletter` as `RuntimeFailureCategory.PoisonMessage` (a retry-tier message without the contract headers is corrupt).
    2. `var now = IClock.UtcNow.ToUnixTimeMilliseconds();`
    3. If `now < retryDeliverAfterUnixMs`: calculate `delay = retryDeliverAfterUnixMs - now`. If `delay > DelayCeilingMs` (default 60_000), short-circuit — do NOT commit; let Kafka redeliver next poll cycle; sleep `min(delay, pollCeiling)` and re-consume. If `delay ≤ DelayCeilingMs`, sleep for `delay` ms then proceed to step 4.
    4. Republish the original message (preserved headers + payload) to the `retry-source-topic` via `KafkaRetryEscalator` (which handles the `.retry → .deadletter` exhaustion case too — at this point attempt-count has not yet incremented, so republish goes to `.events`). Use `IProducer<string, string>.ProduceAsync` wrapped in the shared breaker.
    5. Commit the `.retry` source offset.
  - Emits domain audit events `RetryAttemptedEvent` (on republish) and `RetryExhaustedEvent` (when escalator routes to `.deadletter`) through the standard outbox path. These events are NOT strictly required for R2.A.3d Phase A but unlock the §14 "Recovery evidence and observability" row when wired.
  - Uses the R2.E.1 rebalance observability helper + R2.E.2 lag observability helper just like the other 11 workers — consistency is load-bearing.
- **CHECK:**
  - `KafkaRetryConsumerWorker.cs` exists.
  - Wired as `IHostedService` in `KafkaInfrastructureModule.AddMessaging`.
  - Uses `KafkaRebalanceObservability.Attach` + `KafkaLagObservability.Record` (architecture tests already enforce for every `ConsumerBuilder`).

## R-RETRY-DELIVER-AFTER-DETERMINISM-01 — Deliver-After Is Replay-Deterministic

- **CLASSIFICATION:** runtime / event-fabric / retry / determinism
- **SEVERITY:** S1
- **DESCRIPTION:**
  - The `retry-deliver-after-unix-ms` value written by `KafkaRetryEscalator.EscalateAsync` MUST be replay-deterministic: the same `(sourceTopic, eventId, attemptCount)` input MUST produce the same deliver-after value on replay.
  - Inputs to the formula:
    - `IClock.UtcNow` at escalation time — deterministic per the R1 IClock contract.
    - `IRandomProvider.Next(seed)` where seed is `$"{sourceTopic}:{eventId}:retry:{attemptCount}"` — deterministic per R1 IRandomProvider contract + locked R2 retry-determinism invariant.
  - NO `DateTime.UtcNow` / `DateTimeOffset.UtcNow` / `System.Random` / `Stopwatch`-derived values may appear in the delay computation.
- **CHECK:**
  - Unit test: two `EscalateAsync` calls with identical inputs + fake clock + fake random produce byte-identical `retry-deliver-after-unix-ms` headers.
  - Source scan: `KafkaRetryEscalator.cs` does NOT contain `DateTime.UtcNow`, `DateTimeOffset.UtcNow`, `new Random`, or `Stopwatch.GetTimestamp`.

## R-RETRY-CONSUMER-INTEGRATION-01 — GenericKafkaProjectionConsumerWorker Routes Handler-Throws Through the Escalator

- **CLASSIFICATION:** runtime / event-fabric / retry / integration
- **SEVERITY:** S1
- **DESCRIPTION:**
  - The outer `catch (Exception ex)` in `GenericKafkaProjectionConsumerWorker.ExecuteAsync` currently logs + sleeps + re-polls (no offset advance → infinite tight-loop redelivery on persistent handler failure). Post-R2.A.3d it calls `KafkaRetryEscalator.EscalateAsync` with `attemptCount = 1` for the first failure and commits the source offset. Subsequent failures flow from `.retry` via the retry consumer, which increments attempts.
  - Poison-message paths (absent / blank / unparseable headers, JSON deserialize) continue to route directly to `.deadletter` via the existing `PublishToDeadletterAsync` — those are NEVER retryable regardless of attempt budget.
  - Integration for the 10 other consumer workers (saga / integration / feedback) is out of scope for R2.A.3d Phase A — deferred to Phase B. `GenericKafkaProjectionConsumerWorker` is the highest-multiplier integration (6 projection modules × 1 worker).
- **CHECK:**
  - The outer `catch (Exception ex)` block in `GenericKafkaProjectionConsumerWorker.cs` calls `KafkaRetryEscalator.EscalateAsync` and commits the source offset.
  - Poison-message branches (absent headers / JSON fail) keep calling `PublishToDeadletterAsync` directly — unchanged.
