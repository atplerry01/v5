# R2.A.D.4 â€” Breaker Registry + Health Posture Integration (2026-04-19)

Captured per CLAUDE.md $1c. R2.A.D.1 shipped `ICircuitBreaker`. R2.A.D.2 refactored `OpaPolicyEvaluator` (first consumer, via keyed DI). R2.A.D.4 introduces the canonical registry seam so:

## R-BREAKER-REGISTRY-01 â€” ICircuitBreakerRegistry canonical lookup + enumeration

- **CLASSIFICATION:** runtime / resilience / composition
- **SEVERITY:** S1
- **DESCRIPTION:**
  - `ICircuitBreakerRegistry` exposes: `Get(string name) â†’ ICircuitBreaker` (throws `KeyNotFoundException` for unknown names â€” named lookup is load-bearing at composition time), `TryGet(string name) â†’ ICircuitBreaker?` (null-tolerant variant for consumers that check existence), `GetAll() â†’ IReadOnlyCollection<ICircuitBreaker>` (iteration surface for the runtime-state aggregator).
  - Registry is a singleton registered once in host composition. It is NOT a runtime-mutable collection â€” breakers are registered at startup and never added/removed during runtime.
  - Names are stable identifiers (e.g. `"opa-policy-evaluator"`) â€” they MUST match the breaker's own `Name` property so downstream consumers (metrics, logs, health reasons) use a single string identity.
  - The registry owns the iteration invariant: `GetAll()` returns breakers in a deterministic order (sorted by name) so health-posture reason lists are stable across process restarts.
- **REPLACES:** R2.A.D.2's keyed-service registration (`AddKeyedSingleton<ICircuitBreaker>("opa-policy-evaluator", ...)`). The keyed approach required callers to know the key at composition time and couldn't enumerate registered breakers. Registry approach gives both named lookup and enumeration.
- **CHECK:**
  - `ICircuitBreakerRegistry` lives at `src/shared/contracts/runtime/` adjacent to `ICircuitBreaker`.
  - `CircuitBreakerRegistry` implementation at `src/runtime/resilience/`.
  - Unit tests pin: registration preserves Name; lookup returns exact instance; TryGet null on unknown; GetAll deterministic order; duplicate-name registration throws.

## R-BREAKER-HEALTH-POSTURE-01 â€” Aggregator iterates breakers via registry

- **CLASSIFICATION:** runtime / health / composition
- **SEVERITY:** S1
- **DESCRIPTION:**
  - `RuntimeStateAggregator` takes `ICircuitBreakerRegistry` as a constructor dependency. Both `ComputeFromResults` and `GetDegradedMode` iterate `registry.GetAll()` and emit one reason per non-Closed breaker.
  - Reason emission uses canonical naming: the breaker's `Name` is lowercased-snake-cased with `_breaker_open` appended. Example: breaker name `"opa-policy-evaluator"` â†’ reason `"opa_policy_evaluator_breaker_open"`.
  - **Backward compatibility:** the existing reason `"opa_breaker_open"` is RETAINED (not renamed to `"opa_policy_evaluator_breaker_open"`) because downstream dashboards, alert rules, and degraded-mode vocabulary already key on it. The aggregator maintains a short whitelist of known-canonical names to preserve strings:
    - `"opa-policy-evaluator"` â†’ `"opa_breaker_open"`
    - future known names added to the same whitelist.
  - The concrete `WhyceChainPostgresAdapter.IsBreakerOpen` check + `"chain_anchor_breaker_open"` reason is RETAINED for now â€” R2.A.D.3 will refactor Chain to consume `ICircuitBreaker`, and at that point the check moves to the registry path with a whitelist entry for `"chain-anchor"` â†’ `"chain_anchor_breaker_open"`.
  - `RuntimeDegradedMode.CanonicalReasons` vocabulary is NOT changed in R2.A.D.4 â€” no new reasons, no renamed reasons. The aggregator's reason list is still filtered through the existing vocabulary.
- **CHECK:**
  - Architecture test / unit test: aggregator with registry containing a non-Closed breaker named `"opa-policy-evaluator"` emits `"opa_breaker_open"` reason.
  - `ComputeFromResults` and `GetDegradedMode` emit identical reason strings for breaker signals (both iterate the same registry).
  - Aggregator no longer takes `OpaPolicyEvaluator` as a constructor dependency â€” the concrete adapter check is removed.
