## Decisions made before implementation

Recorded here so future readers do not have to reconstruct them from
git history.

1. **Path** — `tests/integration/failure-recovery/`, NOT a new
   `tests/integration/runtime/failure/` tree. The existing FR-1 test
   and `OutboxSharedTableCollection` already live in
   `failure-recovery/` and serialize against the shared `outbox`
   table. A parallel tree would split the suite across two
   namespaces and reintroduce the parallel-corruption risk MI-2 / FR-1
   already closed.

2. **FR-1 not duplicated** — `OutboxKafkaOutageRecoveryTest.cs`
   exhaustively covers scenario 1.1 (Kafka outage → retry → DLQ →
   recovery). Writing a parallel `KafkaFailureRecoveryTests.cs` would
   produce duplicate, conflicting coverage of the same SQL contract.

3. **Net new files** — 4 (FR-2 .. FR-5) instead of 5.

4. **Additive TestHost overload** — `TestHost.ForTodo` gained two
   optional parameters (`IPolicyEvaluator? policyEvaluator`,
   `IChainAnchor? chainAnchorOverride`) so FR-4 / FR-5 can inject
   throwing stubs without restructuring the fixture. Defaults preserve
   the pre-FR behavior byte-identically; every existing call site is
   unaffected.

## Per-scenario evidence

### FR-1 — Kafka outage → retry → DLQ → recovery (CANONICAL)

| Acceptance criterion | Proof |
|----------------------|-------|
| A1 No data loss      | Every seeded row reaches a terminal state (`published` or `deadletter`); final-state assertion. |
| A2 No duplicate      | Deadlettered row has `published_at = NULL`; recovery row reaches `published` exactly once. |
| A3 Retry & DLQ       | `retry_count` increments correctly; row promoted to `deadletter` at `retry_count + 1 >= MaxRetry`. |
| A5 Auto recovery     | Mid-retry recovery test: producer recovers on call N, row reaches `published` without operator intervention. |

Source-of-truth observability: `Whyce.Outbox` meter counters
(`outbox.published`, `outbox.failed`, `outbox.deadlettered`,
`outbox.dlq_published`) defined at
[`KafkaOutboxPublisher.cs:27-31`](../../../src/platform/host/adapters/KafkaOutboxPublisher.cs#L27-L31).

### FR-2 — Postgres failure recovery (NEW)

| Acceptance criterion | Proof |
|----------------------|-------|
| A1 No partial writes | `Connection_Drop_Mid_Batch_Rollbacks_To_Zero_Rows`: 5 INSERTs in a tx → dispose without commit → assert `COUNT(*) = 0` for the test correlation_id. |
| A2 No data loss      | `Recovery_After_Rollback_Reinserts_Exactly_Once`: replay produces every row in `pending`. |
| A3 No duplicates     | Same test: third pass with identical `idempotency_key` is absorbed by `ON CONFLICT DO NOTHING`; row count never doubles. |

Failure injection seam: raw `NpgsqlConnection` + `await using
NpgsqlTransaction` disposal without `CommitAsync`. Same precedent as
MI-2 / FR-1.

### FR-3 — Runtime crash recovery (NEW)

Δ from MI-2: MI-2 proves the **single-row** crash seam. FR-3 extends
to **multi-row claim release** (5 rows locked in one SELECT, all
released on rollback) and explicitly asserts no stuck locks via a
post-drain `CountClaimableForCorrAsync == 0`.

| Acceptance criterion | Proof |
|----------------------|-------|
| Rows reprocessed     | `publishedBy` `ConcurrentDictionary` contains every row id originally claimed by the crashed worker. |
| No duplicate publish | `TryAdd` returning `false` throws; faulted task surfaces in `WhenAll`. |
| No stuck locks       | After survivor drain, `CountClaimableForCorrAsync(corrId) == 0`. |

Failure injection seam: open tx, `SELECT ... FOR UPDATE SKIP LOCKED
LIMIT 5`, dispose without commit. Postgres releases all locks held by
the rolled-back tx — validated by 3 survivor workers racing the
recovery path with no duplication.

### FR-4 — Policy engine failure (NEW)

Contract proven: `PolicyEvaluationUnavailableException` thrown by
`IPolicyEvaluator` bubbles untouched through the runtime control plane,
and ZERO downstream side effects occur.

| Assertion                                     | Mechanism |
|-----------------------------------------------|-----------|
| Typed exception bubbles                       | `Assert.ThrowsAsync<PolicyEvaluationUnavailableException>` on `ControlPlane.ExecuteAsync`; verifies `Reason == "breaker_open"` and `CallCount == 1`. |
| No event persisted (Step 7 never ran)         | `host.EventStore.AllEvents(aggregateId)` is empty. |
| No chain anchor (Step 7 never ran)            | `host.ChainAnchor.Blocks` contains no block with the test correlation_id. |
| No outbox enqueue (Step 7 never ran)          | `host.Outbox.Batches` contains no batch with the test correlation_id. |

Failure injection seam: `ThrowingPolicyEvaluator` private nested class
implementing the canonical `IPolicyEvaluator` contract; injected via
the new `TestHost.ForTodo(policyEvaluator: ...)` overload.

### FR-5 — Chain anchor failure (NEW)

Contract proven: `IChainAnchor` failure causes Step 2 (event store
append) to commit, Step 3 (chain anchor) to throw, and Step 4 (outbox
enqueue) to NEVER run — i.e. the documented `EventFabric` sequencing
hard-stop is byte-perfect.

| Assertion                                     | Mechanism |
|-----------------------------------------------|-----------|
| Typed exception bubbles                       | `Assert.ThrowsAsync<InvalidOperationException>` on `ControlPlane.ExecuteAsync`; verifies stub `CallCount == 1` (chain stage actually executed). |
| Step 2 ran (event in source of truth)         | `host.EventStore.AllEvents(aggregateId)` is **non-empty**. |
| Step 4 never ran (outbox skipped)             | `host.Outbox.Batches` contains no batch with the test correlation_id. |

Failure injection seam: `ThrowingChainAnchor` private nested class
implementing `IChainAnchor`; injected via the new
`TestHost.ForTodo(chainAnchorOverride: ...)` overload.

NOTE on STRICT-mode flag: there is no STRICT mode flag in the current
implementation. Chain failure is **always** hard-stop per
[`ChainAnchorService.cs:133-155`](../../../src/runtime/event-fabric/ChainAnchorService.cs#L133-L155).
The "STRICT mode" branch listed in the §5.2.6 spec collapses to a
single behavior, which FR-5 covers in full.

## Observability validation (§5.2.6 §4)

The §5.2.6 spec requires that failure visibility be exposed via metrics
without introducing new metric names. This is satisfied by the existing
meters — no new instruments are introduced by the FR suite:

| Spec metric name (informal)        | Existing canonical instrument                                                          | Defined at |
|-------------------------------------|----------------------------------------------------------------------------------------|------------|
| `outbox.pending.count`              | `outbox.depth` (gauge from `OutboxDepthSampler`, `Whyce.Outbox` meter)                 | `OutboxDepthSampler.cs` |
| `outbox.retry.count`                | `outbox.failed` (counter, `Whyce.Outbox`)                                              | `KafkaOutboxPublisher.cs:29` |
| `outbox.deadletter.count`           | `outbox.deadlettered` (counter, `Whyce.Outbox`)                                        | `KafkaOutboxPublisher.cs:30` |
| `worker.failure.count`              | (worker liveness via `IWorkerLivenessRegistry`, surfaced by `RuntimeStateAggregator`)  | (HC-5)     |
| `db.connection.failure`             | `postgres.pool.acquisition_failures` (counter, `Whyce.Postgres`)                       | `PostgresPoolMetrics.cs:39` |
| `kafka.publish.failure`             | `outbox.failed` (counter, `Whyce.Outbox`) — Kafka publish failures route through this  | `KafkaOutboxPublisher.cs:29` |

Per $5 anti-drift, no new metric names are introduced. Operators
already running on the existing meter set get §5.2.6 visibility
without configuration changes.
