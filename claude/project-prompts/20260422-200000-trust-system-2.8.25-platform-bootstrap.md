# 2.8.25 — Platform Bootstrap (PlatformBootstrapService)

## TITLE
Phase 2.8.25: Platform Bootstrap — First-Operator Identity Creation at Host Startup

## CONTEXT
The trust-system registration pipeline (2.8.17–2.8.24) is fully wired. Every command passes
WHYCEPOLICY, every actor is non-anonymous (INV-202), and every dispatch is chain-anchored.

The missing piece is the very first actor: before any HTTP request can arrive, the platform
needs a known operator identity in the registry so that initial administration commands carry
a valid trust context. Without this, the first real operator has no way to authenticate.

`PlatformBootstrapService` solves this by running at host startup, using the system identity
scope, and dispatching `InitiateRegistrationCommand` for a configuration-supplied first-operator.
It is idempotent (checks the event store before dispatching), policy-bound (uses the existing
`whyce.trust.identity.registry.initiate` policy via `DispatchSystemAsync`), and produces no
hardcoded super-admin bypass — the operator goes through the same registration pipeline as any
other actor.

## OBJECTIVE
Implement `PlatformBootstrapService` as an `IHostedService` that:
1. Reads first-operator configuration from `Bootstrap:FirstOperator:Email` and
   `Bootstrap:FirstOperator:RegistrationType`
2. Derives a deterministic `RegistryId` via `IIdGenerator.Generate(seed)` from those values
3. Checks `IEventStore.LoadEventsAsync(registryId)` — if events already exist, skips dispatch
4. If absent, sets `SystemIdentityScope.Begin("system/platform-bootstrap", ...)` then calls
   `ISystemIntentDispatcher.DispatchSystemAsync(InitiateRegistrationCommand, ...)`
5. Throws if dispatch returns failure; propagates any exception to halt the host

## CONSTRAINTS
- No hardcoded credentials, emails, or actor IDs
- `Bootstrap:FirstOperator:Email` and `Bootstrap:FirstOperator:RegistrationType` MUST be present
  in config — throw `InvalidOperationException` on missing values (fail-fast)
- No `Guid.NewGuid()`, no `DateTime.UtcNow` — use `IIdGenerator` and `IClock`
- No anonymous execution — `SystemIdentityScope.Begin("system/platform-bootstrap")` before dispatch
- Idempotent — event store check prevents double-registration on restart
- Dispatch via `DispatchSystemAsync` only — not `DispatchAsync` (this is a system-originated command)
- Classification: trust-system

## EXECUTION STEPS
1. Write `PlatformBootstrapService.cs` in `src/platform/host/bootstrap/`
2. Register via `services.AddHostedService<PlatformBootstrapService>()` in `TrustSystemCompositionRoot`
3. Write unit tests in `tests/unit/trust-system/bootstrap/PlatformBootstrapServiceTests.cs`
4. Build and verify clean

## OUTPUT FORMAT
- `src/platform/host/bootstrap/PlatformBootstrapService.cs`
- Updated `src/platform/host/composition/trust/TrustSystemCompositionRoot.cs`
- `tests/unit/trust-system/bootstrap/PlatformBootstrapServiceTests.cs`

## VALIDATION CRITERIA
- Missing config throws `InvalidOperationException` before `StartAsync` completes
- Event store has events → dispatch is skipped (idempotent)
- Event store empty → dispatch called exactly once with deterministic `RegistryId`
- Same config twice → same `RegistryId` (determinism invariant)
- Dispatch failure → exception propagates (no silent swallow)
- No `Guid.NewGuid()` or `DateTime.UtcNow` in implementation
