## CLOSURE STATEMENT

Phase 1.5 §5.2.1 established **declared and observable overload
control** at the runtime HTTP edge and at every internal chokepoint
identified by Step A. All Step B S0 risks were converted away from
`UNBOUNDED-OPEN` shapes:

- **R-01** HTTP intake → `BOUNDED-OBSERVABLE` via PC-1 partitioned
  rate limiter.
- **R-02** outbox table buffer → `BOUNDED-OBSERVABLE` via PC-3 depth
  sampler + high-water-mark refusal.
- **R-06** OPA evaluator → `BOUNDED-OBSERVABLE` via PC-2 timeout +
  breaker + typed-exception edge.
- **R-03** chain anchor → `BOUNDED-OBSERVABLE` (observability half)
  via PC-5 wait/hold histograms.
- **R-04 / R-10** Npgsql pool → `BOUNDED-OBSERVABLE` for the seven
  covered adapters via PC-4 declared `NpgsqlDataSource` pools.
- **R-06** Kafka prefetch → `BOUNDED-OBSERVABLE` via PC-6 declared
  consumer envelope.
- **R-07** projection lag → `BOUNDED-OBSERVABLE` via PC-7 durable
  broker-timestamp histogram.

The runtime now has **explicit retryable-refusal seams** for:
- intake (HTTP 429 + `Retry-After`)
- policy unavailability (HTTP 503 + `Retry-After`)
- outbox saturation (HTTP 503 + `Retry-After`)

each backed by a typed exception (or limiter `OnRejected`) flowing
through the existing `IExceptionHandler` chain at the API edge.

**Key operational parameters now live in declared configuration
blocks:**
- `Intake.{GlobalConcurrency, QueueLimit, PerTenantConcurrency, RejectionResponse, RetryAfterSeconds}`
- `Opa.{Endpoint, RequestTimeoutMs, BreakerThreshold, BreakerWindowSeconds, OpenStateBehavior}`
- `Outbox.{MaxRetry, HighWaterMark, SamplingIntervalSeconds, SaturationResponse, RetryAfterSeconds}`
- `Postgres.Pools.{EventStore, Chain}.{MaxPoolSize, MinPoolSize, ConnectionTimeoutSeconds, CommandTimeoutSeconds}`
- `KafkaConsumer.{QueuedMaxMessagesKbytes, FetchMessageMaxBytes, MaxPollIntervalMs, SessionTimeoutMs}`

Six canonical meters export the §5.2.1 observability surface:
`Whyce.Intake`, `Whyce.Policy`, `Whyce.Outbox`, `Whyce.Postgres`,
`Whyce.Chain`, `Whyce.Projection.Consumer`.

## RESIDUAL ITEMS (NOT BLOCKERS)

The following items are explicitly **outside** the §5.2.1 PASS gate
because the opening-pack acceptance criteria require *bounded and
observable*, not *restructured*, for every in-scope surface:

- **Projections-pool refactor** — `PostgresProjectionWriter` (and
  `TodoController.Get`) still construct raw `NpgsqlConnection` per
  call. Refactoring touches the domain bootstrap surface. PC-4
  residual.
- **DLQ depth bounds** — R-08, S2. No `dlq.depth` gauge, no DLQ
  high-water-mark.
- **`LoadEventsAsync` streaming** — R-09, S2. Reads full event
  stream into in-memory list per command.
- **Native Npgsql / Confluent.Kafka client counter bridging** — out
  of scope; explicit `Whyce.*` meters cover §5.3.x needs.
- **`ChainAnchorService` structural restructuring** — lock is now
  observable; restructuring (move I/O outside the critical section,
  shard, etc.) is a §5.2.2 candidate.
- **Per-route admission policies** — limiter is global by design;
  per-route is §5.2.x.
