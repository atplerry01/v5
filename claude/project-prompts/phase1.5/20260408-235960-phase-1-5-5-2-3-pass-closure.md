## CONTEXT

§5.2.3 was opened on 2026-04-08 immediately after §5.2.2 closed
PASS, as the third workstream in the §5.2.x Runtime
Infrastructure-Grade Hardening cluster. Where §5.2.1 made the
runtime *refuse work safely* under load and §5.2.2 made the runtime
*process accepted work predictably* under concurrent pressure,
§5.2.3 answered *"under what declared timeout, cancellation, and
circuit-protection discipline does the runtime fail fast instead of
hanging?"*. Specifically, §5.2.3 closed the indefinite-wait,
untokened, and circuit-unprotected seams that PC-5 / KW-1 had made
*observable* but not *bounded* on the chain anchor path, and
extended that discipline across every hot-path adapter, the
projection worker, the workflow engine, and the host shutdown
boundary.

## CLOSURE STATEMENT

Phase 1.5 §5.2.3 made **runtime cancellation, timeout, and
circuit-protection discipline explicit instead of incidental**.
End-to-end `CancellationToken` now flows from
`HttpContext.RequestAborted` (linked to
`IHostApplicationLifetime.ApplicationStopping` via TC-9) through
`ISystemIntentDispatcher` → `RuntimeControlPlane` → middleware
pipeline → `ICommandDispatcher` → `ExecuteEngineAsync` /
`ExecuteWorkflowAsync` → `IEventStore` / `IOutbox` /
`IIdempotencyStore` / `IChainAnchor` / `IWorkflowEngine` →
`IWorkflowStep` → database round-trip. Every previously-indefinite
waiter is bounded by a declared timeout, every previously-untokened
adapter honors cancellation, and the chain-store path is protected
by a Closed/Open/HalfOpen breaker mirroring the §5.2.1 PC-2 OPA
pattern.

Specific outcomes:

- **Dispatcher CT desert resolution** — TC-1 co-evolved every
  contract on the request execution path in a single coordinated
  pass. The middleware pipeline closure shape is now
  `Func<CancellationToken, Task<CommandResult>>`; every link
  forwards `next(ct)`. Once TC-1 landed, every subsequent §5.2.3
  patch inherited a real CT at every seam it touched.
- **Chain-anchor refusal family closure** — TC-2 + TC-3 form the
  canonical chain-anchor refusal family: TC-2 bounds the
  `_lock.WaitAsync` with `WaitTimeoutMs` and the typed
  `ChainAnchorWaitTimeoutException`; TC-3 threads CT into the
  chain-store I/O and adds a declared consecutive-failure circuit
  breaker with the typed `ChainAnchorUnavailableException`. Both
  edge handlers are co-resident in `Program.cs` and mirror the
  prior canonical refusal handlers exactly. Caller-driven
  cancellation propagates as `OperationCanceledException` and does
  not advance the breaker.
- **Postgres hot-path adapter token threading** — TC-5
  mechanically extended the four canonical hot-path contracts
  (`IEventStore`, `IOutbox`, `IIdempotencyStore`, `ISequenceStore`)
  with `CancellationToken cancellationToken = default` and
  threaded CT into every `Postgres*Adapter` `BeginTransactionAsync`,
  `ExecuteReaderAsync`, `ReadAsync`, `ExecuteScalarAsync`,
  `ExecuteNonQueryAsync`, and `CommitAsync` call site. The
  wrappers (`EventStoreService`, `OutboxService`) and the live
  runtime call sites (`EventFabric.ProcessInternalAsync`,
  `RuntimeCommandDispatcher.ExecuteEngineAsync`,
  `IdempotencyMiddleware.ExecuteAsync`) forward CT end-to-end.
  The failure-branch `ReleaseAsync` calls deliberately use
  `CancellationToken.None` so rollback is not aborted by upstream
  cancellation.
- **Projection handler hung-step seam closure** — TC-6 widened
  both projection-handler contracts with CT and made the worker
  forward `stoppingToken` into `handler.HandleAsync`.
  `TodoProjectionHandler` actively threads CT into its
  `ExecuteReaderAsync(ct)` / `ReadAsync(ct)` / `ExecuteNonQueryAsync(ct)`
  calls; `WorkflowExecutionProjectionHandler` carries CT at the
  signature level (the `IWorkflowExecutionProjectionStore`
  contract is a separate workstream).
- **Workflow gate token-drop closure** — TC-8 was the smallest
  patch in the workstream: two call-site edits in
  `RuntimeCommandDispatcher` (`ExecuteWorkflowAsync` and
  `ResumeWorkflowAsync`) plus the helper signature extension to
  receive CT from `DispatchAsync`. Closes the single mechanical
  gap left over from TC-1's pipeline pass.
- **Workflow execution timeout discipline** — TC-7 introduced
  the canonical two-tier deadline shape: an execution-level
  linked CTS bounded by `MaxExecutionMs` and a per-step linked
  CTS bounded by `PerStepTimeoutMs`, both linked to the upstream
  request/host token. Cancellation discrimination inside the
  engine `catch` distinguishes caller-driven cancellation
  (propagated as `OperationCanceledException`) from execution
  timeout (`WorkflowTimeoutException("execution", null, …)`)
  from per-step timeout (`WorkflowTimeoutException("step",
  stepName, …)`). The typed exception is mapped to 503 +
  `Retry-After` by the new `WorkflowTimeoutExceptionHandler`.
- **Host-shutdown drain declaration** — TC-9 declared
  `Host:ShutdownTimeoutSeconds` (default 30) and applied it via
  `services.Configure<HostOptions>`. The narrow
  `HostShutdownLinkingMiddleware` replaces
  `HttpContext.RequestAborted` with a linked CTS combining the
  original client-disconnect token and
  `IHostApplicationLifetime.ApplicationStopping`. Because TC-1
  already plumbed the controller-bound CT end-to-end, the
  shutdown signal inherits all of that plumbing without touching
  a single downstream contract: the instant the host begins to
  drain, every in-flight request observes the same cancellation
  that TC-1 plumbed.

The runtime now has **seven canonical retryable-refusal seams**
(intake 429, OPA policy 503, outbox saturation 503, workflow
saturation 503, **chain-anchor wait timeout 503**, **chain-anchor
unavailable 503**, **workflow timeout 503**) backed by typed
exception → `IExceptionHandler` mappings, plus the pre-existing
concurrency-conflict 409 REJECT. Three new declared configuration
sub-blocks externalise every §5.2.3 timeout / breaker / shutdown
parameter: `Host.ShutdownTimeoutSeconds`,
`ChainAnchor.{WaitTimeoutMs, RetryAfterSeconds, BreakerThreshold,
BreakerWindowSeconds}`, `Workflow.{PerStepTimeoutMs, MaxExecutionMs}`.

## RESIDUAL ITEMS (NOT BLOCKERS)

The following items are explicitly **outside** the §5.2.3 PASS
gate because the opening-pack acceptance criteria require *bounded
and observable*, not *restructured*, for every in-scope surface:

- **Structural restructuring of `ChainAnchorService` held-section
  I/O** — declared waiver. Wait + holder are both bounded by
  TC-2 / TC-3; moving the chain-store persist outside the held
  section, sharding by correlation hash, or replacing the global
  semaphore with a per-aggregate primitive remains a future
  structural workstream. PC-5 / KW-1 / TC-2 / TC-3 together
  provide the declared-bounded envelope and the operator signal.
- **`IWorkflowExecutionProjectionStore` token threading** — TC-6
  widens the handler contract but not the store contract.
  Store-side threading is a separate workstream.
- **Workflow lifecycle event loss on declared timeout** — events
  accumulated in `context.AccumulatedEvents` up to the moment of
  timeout are not persisted (the throw bypasses the fabric persist
  → chain → outbox path). This is consistent with the rest of the
  canonical refusal family.
- **`Workflow:MaxExecutionMs` resume budget** — a resumed workflow
  gets a fresh execution-level budget; the prior failed run's
  wall-clock is not durably tracked.
- **`ISequenceResolver` / HSID prelude token threading** — TC-5
  widens the `ISequenceStore` contract but
  `PersistedSequenceResolver` and `RuntimeControlPlane.StampHsidAsync`
  still call `NextAsync(scope)` with default CT.
- **Per-call Postgres command timeout** — TC-3 / TC-5 thread CT
  but do not introduce a per-call DB timeout below the existing
  `CommandTimeoutSeconds` from the chain pool.
- **Engine-side cooperative cancellation in CPU-bound steps** —
  the two CPU-bound in-tree steps accept CT at the signature
  level only; new steps must remember to honor the token at any
  cancellable await.
- **Background-worker shutdown linking** — `BackgroundService`
  instances receive `stoppingToken` directly from the .NET host
  and do not transit `HostShutdownLinkingMiddleware`. TC-9
  linking is request-scoped only — correct by construction.
- **Kafka producer timeout work** — out of scope per opening
  pack §2.6.

None of these match an opening-pack §2.9 acceptance criterion
that would block §5.2.3 PASS.

## NEXT WORKSTREAM

§5.2.4 — Health, Readiness, and Degraded Modes. Natural successor:
the canonical retryable-refusal edges and the declared timeout /
breaker primitives introduced in §5.2.3 are the inputs to a
declared health / readiness model that distinguishes "live but
refusing" from "down" from "degraded". The `ChainAnchorUnavailable`
and `WorkflowTimeout` typed exceptions are early candidates for
readiness-probe state inputs; the `chain.store.failure` counter
and the `Whyce.Chain` wait/hold histograms are early candidates
for degraded-mode signals.
