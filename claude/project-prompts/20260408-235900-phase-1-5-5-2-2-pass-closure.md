## CLOSURE STATEMENT

Phase 1.5 §5.2.2 made **runtime concurrency and resource ceilings
explicit instead of incidental**. Every concurrency primitive on
the hot path is now either declared and bounded, declared and
observable, or declared and waived — there is no remaining
incidental hardcoded concurrency constant in `src/`.

Specific outcomes:

- **C14 capacity-model resolution** — KC-1 dropped intake ceilings
  to fit the declared event-store pool envelope; KC-2 further
  improved the envelope by coalescing the idempotency 2× connection
  amplification into a single round-trip.
- **DLQ growth observability** — KC-3 added `outbox.deadletter_depth`
  on the outbox side and `consumer.dlq_publish_failed` on the
  consumer side, plus the declared `OutboxOptions.DeadletterRetention`
  posture. DLQ growth is no longer silent on either side.
- **Projections-pool declaration** — KC-4 introduced the third
  declared `NpgsqlDataSource` pool (`projections`) and refactored
  `PostgresProjectionWriter` and `TodoProjectionHandler` to flow
  through it. **`TodoController.Get` recorded as residual** because
  `Whycespace.Api → Whycespace.Host` would create a project
  dependency cycle.
- **Advisory-lock wait observability** — KC-5 added the
  `Whyce.EventStore` meter with `event_store.append.advisory_lock_wait_ms`
  measuring `pg_advisory_xact_lock` round-trip distinct from
  PC-4 pool acquisitions.
- **Workflow in-flight ceiling** — KC-6 introduced `WorkflowAdmissionGate`
  composing two `PartitionedRateLimiter<string>` instances
  (per-workflow-name + per-tenant) and the fourth canonical
  RETRYABLE REFUSAL edge (`WorkflowSaturatedException` → 503).
- **ChainAnchor declared-options / waiver** — KW-1 promoted
  `ChainAnchorService._lock` from `DECLARED-OPAQUE` (incidental
  literal) to `DECLARED-BOUNDED` (`ChainAnchorOptions.PermitLimit=1`).
  Structural restructuring is the **declared waiver**, not silent
  residual.
- **`ChainLock` deletion** — KC-7 removed the dead-code
  `ChainLock` and `ChainLockHandle` types after Step B P-K1 confirmed
  zero callers. The "second hidden global semaphore on the chain
  path" hypothesis is resolved by deletion rather than reconciliation.
- **`LoadEventsAsync` observability + declared waiver** — KC-8
  added the `event_store.replay_rows` histogram with explicit
  structural waiver recorded at the declaration site. The
  `IEventStore` interface contract change required for true
  streaming is the named follow-on workstream concern.

The runtime now has **four canonical RETRYABLE REFUSAL edges**
(intake 429, OPA 503, outbox 503, **workflow 503**), **eight declared
configuration blocks** (`Intake.*`, `Opa.*`, `Outbox.*`,
`Postgres.Pools.*`, `KafkaConsumer.*`, **`Workflow.*`**, **`ChainAnchor.*`**,
plus the existing `Logging`), and **eight canonical `Whyce.*` meters**
(`Whyce.Intake`, `Whyce.Policy`, `Whyce.Outbox`, `Whyce.Postgres`,
`Whyce.Chain`, `Whyce.Projection.Consumer`, **`Whyce.EventStore`**,
**`Whyce.Workflow`**).

## RESIDUAL ITEMS (NOT BLOCKERS)

The following items are explicitly **outside** the §5.2.2 PASS
gate because the opening-pack acceptance criteria require *bounded
and observable*, not *restructured*, for every in-scope surface:

- **`TodoController.Get` projections-side raw `NpgsqlConnection`** —
  blocked by the api↔host project dependency cycle. Cleanest fix
  is a shared `IDbConnectionFactory` abstraction in
  `shared/contracts` using BCL `System.Data.Common.DbConnection`.
  Recorded as declared residual via KC-4.
- **Structural restructuring of `ChainAnchorService._lock`** —
  declared waiver via KW-1. Moving external `IChainAnchor.AnchorAsync`
  outside the held section, sharding by correlation hash, or
  replacing with a per-aggregate primitive is a §5.2.3 candidate
  that naturally co-evolves with timeout / cancellation work.
- **`LoadEventsAsync` streaming/paging refactor** — declared
  waiver via KC-8. Requires extending the `IEventStore` interface
  contract; future structural workstream owns the change.
- **Multi-instance `KafkaOutboxPublisher` drain** — single-instance
  is the current declaration; multi-instance is architecturally
  supported but not registered.
- **Multi-worker per-topic projection parallelism** — in-flight=1
  per worker is correct for ordering; §5.3.x throughput-tuning
  concern.
- **Native Npgsql / Confluent.Kafka client counter bridging** —
  out of scope; the explicit `Whyce.*` counter set covers §5.3.x
  load-work needs.
- **`PostgreSqlHealthCheck.cs` raw `NpgsqlConnection`** — health
  probe, not a hot-path projections site.

None of these match an opening-pack §2.9 acceptance criterion
that would block §5.2.2 PASS.
