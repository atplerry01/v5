Authoritative doctrine to audit against

## CLASSIFICATION RELATIONSHIP MODEL

`structural-system`
- parent binding truth

`content-system`

`economic-system`
- monetary/accounting/capital truth
- binds to structural-system

`operational-system`
- execution/use-case truth

Audit method
================================
Use the repository/domain tree as the source of truth.

For each of the five systems:
- inspect actual contexts
- inspect grouped topology
- inspect leaf-domain coverage
- determine which leaves are materially implemented vs scaffold-only
- assess whether the system is acting like the truth class it is supposed to be
- assess whether it properly binds to or references other systems in line with doctrine
- assess whether it appears over-expanded, under-expanded, or misclassified

Do not rely on assumptions.
Ground the audit in the actual modeled code and directory state.

Required audit dimensions
================================

For each of the five systems, review:

### A. Completeness
- total contexts
- total domain-groups
- total leaves
- implemented leaves vs scaffold-only leaves
- rough maturity level:
  - low
  - medium
  - high

### B. Doctrinal alignment
Assess whether the system matches its intended role:

- `structural-system`
  - is it really serving as parent binding truth?
  - does it own hierarchy/topology/reference/master truth strongly enough?
- `business-system`
  - is it really only business meaning truth?
  - has non-business truth been removed?
- `content-system`
  - is it really content artifact + lifecycle truth?
  - is it cleanly bound to structural truth?
- `economic-system`
  - is it really monetary/accounting/capital truth?
  - has it avoided swallowing business/operational/reference truth?
- `operational-system`
  - is it really a self-sufficient execution/use-case truth system?
  - is it broad enough for that role?
  - does it bind to structural and reference business/content/economic correctly?

### C. Implementation standard
Review:
- grouped topology discipline
- naming consistency
- leaf implementation depth
- aggregate/event/value-object/specification completeness
- cross-leaf reference discipline
- determinism discipline if visible
- whether the system looks production-grade or still transitional

### D. Relationship-model fit
For each system, explain:
- what it should bind to
- what it should reference
- what it should not own
- whether the current implementation actually reflects that
